# checkup

Run the three review skills back to back, autonomously, in one invocation.

## Purpose

`audit` has two preconditions: a clean `consistency` report (project-level schema and skill drift cleared) and a clean `lint` report (page-level structural drift cleared). Those preconditions force only that **audit runs last** — audit gates on the two reports independently, not on their relative order. Checkup runs consistency before lint because consistency's project-level fixes (stale wiki-page section names, index drift) feed what lint then checks — and because consistency can itself edit wiki pages, whose log entry would make a prior lint stale under audit's Step-1 recency gate, so consistency-first is correctness-preserving, not only churn-reducing. One invocation takes the knowledge base from unchecked toward fully reviewed and fixed — a clean full pass is the good case, not a guarantee: consistency may end `findings`/`blocked` or lint `blocking`, in which case audit is skipped (Step 3).

## Autonomous Operation

Checkup runs the check chain unattended and asks nothing mid-chain — it never stops to prompt between consistency, lint, audit, and the confirming lint, and it closes the loop so findings get fixed, not left in reports. What it genuinely cannot decide autonomously is not buried in prose either: at the end of the run those decisions are surfaced as `AskUserQuestion` prompts (Step 5). The split is the point — everything mechanical or determinate is applied; everything that needs the user's judgement or an outward-facing action is put to them as a question.

- `consistency` and `lint` apply every fix they own (mechanical and determinate) at every severity level, each looping its own layer until clean.
- `audit`, after fact-checking each page against its raw source, sets the page's status (`verified` / `needs-update`) and carries out its content fixes (splits, merges, rewrites), preserving the prior version.
- After audit edits pages, a confirming `lint` re-runs to verify audit introduced no structural drift (broken links, malformed pipes, hash mismatches, index drift). This is composition, not a new check — the same `lint` skill, invoked a second time at a second point. It does not re-gate audit (audit has already run); its result feeds the verdict and the end-of-run questions.
- The one thing checkup does not touch: `CLAUDE.md`, skill files, and scripts. That is the root the whole check chain answers to. `consistency` prepares any drift there as a concrete proposal; checkup surfaces those proposals as end-of-run questions for the user to approve. They are surfaced, never buried, and never applied autonomously.
- Checkup cannot intercept a sub-skill's blocking `AskUserQuestion` and proceed past it — mid-chain silence holds only because each orchestrated sub-skill declines to prompt under checkup: `consistency` defers its root proposals to checkup (its Step 10), `lint` is mechanical, and `audit` asks nothing for its actions. The one residual stop is audit's own precondition guard (its Step 1-2), a hard stop rather than an approval prompt. Checkup pre-empts it in Step 3 by checking audit's full precondition itself — the `result:` fields, recency, and `git status --porcelain` on `CLAUDE.md`/`README.md`/`.claude/skills` (audit halts on any uncommitted schema/skill edit) — and skipping audit with a partial-run report when any is unmet, rather than invoking audit into a mid-chain halt.

A checkup run ends with the wiki itself fixed (pages corrected, content restructured, status set) and the genuine decisions it cannot make — root-level proposals, a commit to clear post-audit items, suggested new pages, housekeeping — put to the user as questions. It never silently changes the schema, and it never takes an outward-facing action (a commit, a new page) on its own. Note that `consistency` and `lint` apply their fixes before audit's gate is checked, so a run that skips audit (an unmet precondition) has still applied every mechanical and determinate fix — the gate guards audit's content surgery, not all mutation.

## When To Invoke

- Before high-stakes work — synthesis, paper drafting, major reference work.
- After a batch of ingest operations (any mode), or a schema or skill change, when both the project layer and the page layer may have drifted.
- As a periodic health check of the whole knowledge base.
- When the user asks to run all the checks, do a full check, or run consistency, lint, and audit together.

## When Not To Invoke

- Only one layer needs checking. Run `consistency`, `lint`, or `audit` directly.
- The user wants a research answer. Use `query`.
- Mid-operation — checkup is a standalone pass, not a step inside another skill.

## Procedure

Checkup is an orchestrator. It runs each sub-skill inline in this same context — reading that skill's `SKILL.md` and performing its steps here, not spawning it as a separate run; this is what lets checkup suppress a sub-skill's own end-of-run prompt, read each report path directly, and keep the chain unattended. It does not re-implement the three skills and adds no checks or findings of its own — only orchestration policy (the fixed order, the `partial`-mode choice, the audit precondition gate, and proposal surfacing), running each skill in full, following that skill's own `SKILL.md`, in order. Each skill self-converges and fixes what it finds.

```text
Checkup Progress:
- [ ] Step 1: Load memory; run consistency
- [ ] Step 2: Run lint
- [ ] Step 3: Run audit
- [ ] Step 4: Run a confirming lint (when audit ran and may have mutated a page; run it whenever you cannot confirm every page was left untouched)
- [ ] Step 5: Validate each sub-skill reached a terminal result; collate the summary led by a verdict line; surface genuine decisions as AskUserQuestion prompts
```

1. **Load memory, then run consistency.** First read `.claude/skills/checkup/checkup-memory.md` and `.claude/skills/multi-skill/multi-skill-memory.md` to apply prior corrections about orchestration scope, when to stop mid-chain, and which sub-skill outputs the user has tuned how checkup handles. (The sub-skills load their own per-skill memory when they run; checkup only needs the orchestration-level corrections.) Then execute the `consistency` skill in full, following `.claude/skills/consistency/SKILL.md`. It loops to two consecutive clean passes and applies every mechanical fix it owns. Drift in `CLAUDE.md`, skill files, or scripts it does not fix — it prepares those as concrete proposals (consistency Step 10). Collect those proposals for checkup's closing summary. Consistency writes its own report to `2-outputs/consistency/` and prepends its own log entry.

2. **Run lint.** Execute the `lint` skill in full, following `.claude/skills/lint/SKILL.md`. Lint loops its deterministic layer to a clean pass and applies every determinate fix at every severity level (its loop bound and regression guard are lint's own — checkup does not re-specify them, so they cannot drift out of sync here). Its residual warnings and info (length caps, reciprocal contradictions, many pages still `status: draft`) are expected findings, not drift — `audit` carries out the authored fixes among them. Lint writes its own report to `2-outputs/lint/` and prepends its own log entry.

3. **Run audit.** First check audit's two preconditions against the `result:` fields of the consistency and lint reports this run just wrote (identify each by the path the sub-skill reported on completion, not merely the newest file on disk — a prior run's report must never be read as this run's gate): consistency passes when `result: clean` — including a clean report that carries pending root-level `proposals:` (proposals do not fail the gate) — and lint passes when `result: clean` (which already excludes the standing repo-state Criticals from blocking). The preconditions are unmet — skip audit and report a partial run (reported in Step 5; the confirming lint is skipped too) rather than running against a drifted base or letting audit's own stop-and-ask fire — when any of these holds: consistency ended `result: findings`/`blocked`; lint ended `result: blocking`; a stage wrote no this-run report, or a report with no `result:` field or an empty/absent finding array (a crash — a report is never `clean` merely because it carries no `blocking`, mirroring consistency's "an empty or absent JSON array is never `clean`"); or `git status --porcelain -- CLAUDE.md README.md .claude/skills` is non-empty — an uncommitted schema/skill edit that audit's own Step 2 stops-and-asks on, so checkup pre-empts the mid-chain halt by skipping audit and asking the user to commit or stash first. When both pass, execute the `audit` skill in full, following `.claude/skills/audit/SKILL.md`. Audit re-verifies the freshly written consistency and lint reports as its preconditions, reads the wiki and the raw sources, runs the semantic pass, and (for every page it fact-checks against its raw source) sets `status: verified` or `status: needs-update` and carries out its content fixes (splits, merges, rewrites) per audit's own Step 7. Checkup invokes audit in `partial` mode (the default; what `partial` covers is defined in `audit`'s SKILL.md and `CLAUDE.md` → Page Status — checkup's orchestration choice is the mode, not its definition), which keeps routine runs affordable. Audit's own Step 1-2 preconditions re-read the `result:` fields *plus* independent recency and `git status` guards; checkup's Step-3 gate above replicates all three kinds (result, recency, uncommitted-schema), so when checkup does run audit its guards agree and never fire mid-chain. (A committed schema change cannot postdate this run's consistency report — checkup never commits mid-chain — so only the uncommitted-schema disjunct can fire in-chain, and the gate catches it.) A `full` audit pass that includes verified pages is the user's call, invoked directly as `/audit full` outside checkup — a user asking to "do a full check" wants checkup's full chain (all three layers), not audit's `full` mode, which checkup does not run. Audit is a semantic judgement pass and runs once, not in a loop. Audit writes its own report to `2-outputs/audit/` and prepends its own log entry.

4. **Run a confirming lint.** Only when audit ran and mutated at least one page — a content fix (split, merge, rewrite), a status change, or a verification-neutral re-stamp; read this from audit's report (a non-empty Status-changes-applied section or recorded content fixes) and its log entry, and when you cannot confirm audit left every page untouched, run the confirming lint anyway, since it is cheap and idempotent. Audit is the last mutator in the chain, so its content fixes (link sweeps, rewrites, splits, index edits) can move structure after lint last looked. Execute the `lint` skill again in full, following `.claude/skills/lint/SKILL.md`, to confirm audit introduced no structural drift — broken wikilinks, malformed pipes, `verified_hash` mismatches, index drift. This is a second composed invocation of an existing skill, not a loop and not a new check; checkup adds no logic to it. Read its result with one discrimination: a genuine new structural finding (a broken link the sweep created, a hash mismatch) is a real problem — name it in the summary and, if it is determinate, lint has already fixed it. The `verified_anchor_unaudited` finding is its own case: lint emits it (a Critical it does not auto-fix) when a `status: verified` page carries a section or figure anchor added or changed since the last commit without being re-verified, demoted, or marked `*[unverified]*`. Audit's Step 7 obliges audit never to leave a verified page tripping it, so a survivor in the confirming lint is a real finding — surface it and name the affected page for a re-audit. Do not treat it as cleared by a commit: committing only moves HEAD so the diff-guard stops seeing it, hiding a still-unverified anchor rather than resolving it. The confirming lint writes its own report to `2-outputs/lint/` and prepends its own log entry. (If audit was skipped on an unmet precondition, skip this step too — nothing mutated the pages since the Step-2 lint.)

5. **Validate, collate the summary, then put genuine decisions to the user.** Before reporting, confirm each sub-skill reached a terminal state by validating the report it wrote this run (the timestamped file it just created, not merely the newest file in the folder, which could be a prior run's): consistency one carrying `result: clean|findings|blocked`, the Step-2 lint one carrying `result: clean|blocking`, audit (when it ran) one whose "Status changes applied" section is finalized and whose log entry was prepended — audit drafts its report before its fixes run, so a bare report file is not proof of completion — and the Step-4 confirming lint (when it ran) one carrying `result: clean|blocking`. A stage that left no this-run report, no `result:` field, or an audit report still in its pre-fix draft state did not complete — report that as an incomplete checkup naming the stage, never fold it into a clean run. If audit crashed mid-authored-fix it may have left a page half-rewritten or quarantined with status unset: name the affected page and tell the user a manual re-check is needed. (Checkup can detect non-completion and report it, but it cannot un-apply a half-finished audit edit — re-run idempotency of an authored fix is audit's/supersede's concern, not checkup's.)

   Lead the combined chat summary with one overall-state verdict line, chosen by first match in this order so co-occurring states resolve deterministically: `incomplete` (a stage did not reach a terminal result, including a sub-skill that crashed before writing its report), then `partial — audit skipped` (a precondition was unmet), then `drift — re-audit needed` (the confirming lint ended `result: blocking`, left a surviving `verified_anchor_unaudited`, or auto-demoted a page audit had just verified — audit introduced structural drift lint could not silently absorb; name the affected page for a re-audit), then `fixed, residuals remain` (clean but decisions pending or audit left pages `needs-update`), then `healthy` (all clean, no decisions pending, and no uncommitted sub-skill edits left in the working tree) — so a run that merely ran to completion is never read as a healthy wiki. Then a roll-up: consistency `result:` + proposal count, Step-2 lint `result:` + audit-blocking count, audit mode + (verified / needs-update / pages-checked) counts, confirming-lint `result:` + whether it found genuine new drift, each naming its report path. Then the per-phase detail: findings by severity, fixes applied (mechanical, determinate, content), loop iteration counts, and the status changes audit applied. If the precondition gate skipped audit, name which precondition was unmet (consistency `findings`/`blocked`, lint `blocking`, a crashed stage, or an uncommitted schema/skill tree) and what must clear first — for a dirty tree, commit or stash the `CLAUDE.md`/skill edits; a `blocked` from a crashed check may clear on a plain re-run, but a `findings`/bounded-out or a `blocking` is a real fix, not a retry. A partial checkup is then safe to re-run, since consistency and lint re-run idempotently and the gate re-reads their fresh reports (an audit that ran but crashed mid-fix is the exception above — re-check the named page first). Do not write a separate checkup report or a separate log entry; the sub-skill reports and their log entries are the record.

   Then surface the genuine decisions — the things checkup cannot do autonomously — as `AskUserQuestion` prompts, one focused question each, never as a prose list the user has to act on by hand. The standing candidates, included only when the run actually produced them: (a) consistency's root-level proposals (the exact `CLAUDE.md` / skill / script edits), one question per proposal — checkup never applies a root-level proposal even when approved (Limits: it never edits CLAUDE.md/skills/scripts), so hand the exact edit to the user, and note that approving one after audit ran changes the base audit just verified against, so a `full` audit is then due; (b) a commit, when any sub-skill edited working-tree files this run (consistency/lint mechanical fixes or audit's content fixes), not only when audit ran — committing is outward-facing, so it is always the user's call, never automatic; on approval, checkup performs the commit (approval lifts the "on its own" bar), naming what is staged and the message; (c) suggested new pages for source-supported danglers audit flagged worth an ingest; (d) housekeeping audit or lint surfaced (e.g. a `log.md` rotation); (e) a `cleanup` run, when consistency's memory-cap check fired (memory over its soft cap) or this checkup left superseded lint/consistency reports — checkup does not prune, but points at `cleanup` as the follow-up. A `healthy` run with no proposals, no page edits, and nothing flagged asks nothing — there is no decision to put. Recommend an option and order it first per the standard `AskUserQuestion` convention, unless the choice is a genuine no-lean one.

## Limits

- Checkup adds no checks and no findings of its own — every finding and action comes from `consistency`, `lint`, or `audit`, whose rules are authoritative. Its only logic is orchestration policy: the fixed order, the `partial`-mode choice, the audit precondition gate, the post-audit confirming-lint pass, and surfacing genuine decisions as questions.
- Checkup produces no `2-outputs/` artifact and no log entry of its own. The sub-skill invocations (consistency, lint, audit, and the confirming lint) each write their own report and log entry.
- The order is fixed: consistency, then lint, then audit, then a confirming lint. Running audit before its two preconditions are fresh produces findings against a drifted schema. Checkup never runs audit when a precondition is unmet (consistency `findings`/`blocked`, lint `blocking`, a stage that crashed without a terminal report, or an uncommitted `CLAUDE.md`/`README.md`/`.claude/skills` edit that would trip audit's Step-2 git guard); it skips audit and reports a partial run rather than acting on a drifted base or letting audit halt mid-chain. When audit is skipped, the confirming lint is skipped too — nothing mutated the pages.
- Each skill self-converges within its own procedure: `consistency` and `lint` loop their layer to clean; `audit` runs once. Checkup never wraps a sub-skill in a loop of its own — the confirming lint is a second composed invocation of `lint` at a distinct point in the chain (after audit), not a loop around it.
- Checkup fixes the wiki (pages, content, and status) through the sub-skills, but it never edits `CLAUDE.md`, skill files, or scripts, and it never takes an outward-facing action (a commit, a new page) on its own. `consistency`'s findings at the root, a commit, and suggested new pages are surfaced as end-of-run questions, not applied.
