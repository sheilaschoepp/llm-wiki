# audit — Apply The Fixes (Step 7)

The fix mechanics for `audit` Step 7. SKILL.md Step 7 is the overview; this file is the detail it directs you to read before editing any page. The scope model, the terminal-verification invariant, and the marker/hash mechanics these fixes rely on are defined once in SKILL.md → Verification Model — read that first; this file does not restate it.

Step 7 applies every content edit; the results are re-verified and stamped in Step 8 (`references/verify-and-set-status.md`). Verification is terminal, so every edit here lands before that stamp — no fix follows it. Act without asking the user — audit is the autonomy exception, and `CLAUDE.md` grants that autonomy only after a fact-check against the raw.

## Confirm The Defect Before Fixing It

A suspicion is not a finding. This guard governs every fix below, and it exists because audit's errors are asymmetric. A missed defect stays on the page, visible, for the next run to catch. An invented defect does the opposite: audit overwrites correct content, re-stamps the page `verified`, and every later `partial` run trusts it — the error is now certified and self-concealing. A false-positive fix is strictly worse than the defect it imagines. So the burden of proof runs one way: before changing existing content, establish the defect positively against the raw — never from a pattern, a plausible inference, or a prior report's say-so.

Four per-instance guards:

- **Per-instance, never per-pattern.** "This class of anchor is usually off by one" is a hypothesis, not evidence. Confirm the specific locator, claim, or bullet is wrong by opening the raw at that place. Sections and tables straddle two physical pages, so a locator naming either page can be correct; a table floats away from the subsection that discusses it; a section that "should" be 3.3 may genuinely also sit on 3.2's page. Each reads as an off-by-one to a pattern-matcher and is not one.
- **Never inherit a finding from a prior report.** Re-derive it, or drop it. A finding copied forward carries the earlier run's errors with laundered authority — a wrong "correction" recorded as a correction is the hardest kind to see, because it arrives pre-justified. Re-check any figure, count, or defect a previous audit claims to have fixed; that is where a `full` pass earns its cost.
- **A carve-out is confirmed against every instance before any bulk application.** A rule that holds for one source (an appendix with no printed pages) may be false for the next (one that restarts at 1, or continues the body's numbering). Applying it on faith to the whole corpus destroys the correct instances. Check each source; expect the corpus to split more ways than you assumed.
- **When a check's own tooling supplies the evidence, test the tooling first.** A regex or script that "finds" a defect is a hypothesis until you have shown it does not also match correct content. Run it against a known-good case before believing a known-bad one.

A defect you cannot confirm is not fixed: report it as a suspicion in the report's Warning tier, or mark the bullet `*[unverified]*` and leave the content alone. Silence is a safe failure; a confident wrong edit is not. When the fix is structural (a split, a merge, a dropped bullet, a rewritten claim), state the load-bearing inference as a yes/no audit assumption in the report (`references/verification-spec.md`) so it can be checked rather than rationalized.

**The mis-located-citation exception.** The guard above rests on "a missed defect stays visible." A citation-location defect does not: a true, well-formed claim pinned to the wrong page is stamped `verified` with the rest of the page, its `verified_hash` written, and every later `partial` run skips it — certified and self-concealing, the exact outcome the guard prevents, but reached by inaction. So for a cited claim you cannot confirm at its cited page, "leave it and let the next run catch it" is false comfort. The resolution is not to wave the citation through on the claim's plausibility (its plausibility is what certifies the error) — it is to require confirmation: open the cited page (never a document-wide search — a distinctive figure or name recurs elsewhere in the same document and would falsely confirm) and check the claim is there. Confirm it and the bullet is verifiable; cannot confirm it this run and you mark the bullet `*[unverified]*` (the page stays promotable on its verified content, the claim rides as the pending delta) rather than stamping it. This does not contradict "confirm the defect before fixing it": both demand opening the raw at the exact place — the guard stops you inventing a defect on a page you did not check, and this exception stops you certifying a citation on a page you did not open. Neither permits passing a cross-source citation on plausibility alone.

## Verification-Neutral Worklist Fixes On Out-Of-Scope Verified Pages

In `partial` mode, lint's authored worklist (Step 1) often names a verification-neutral fix on a `verified` page that carries no `*[unverified]*` markers and so is not in Step 5's fact-check scope. The four categories `CLAUDE.md` → Page Status assigns to audit:

- an unlinked genuine reference to an existing page (`unlinked_page_mention`);
- an open compound to de-hyphenate (`hyphenated_open_compound`, or the slug-derived bidirectional `hyphenated_open_compound_noun` — which opens a hyphenated bare noun and inversely re-hyphenates an open compound used as a modifier before a head noun; both heuristic, so confirm the noun-vs-modifier call against the page's own prose before applying — a context judgement, not a raw fact-check, so a genuinely ambiguous direction-2 re-hyphenation on a `verified` page is marked `*[unverified]*` rather than self-re-stamped, or recorded as a false positive in the check's verified-ignore list);
- a US spelling to normalize;
- an incidental reciprocity-callout link to unwrap (the `missing_reciprocal_contradiction` over-count; see the dispositions below).

(The four format fixes — `callout_block_id`, `wikilink_pipe_spacing`, `citation_bracket_style`, `embed_not_isolated` — are lint's to apply and re-stamp, not audit's.)

These edits are on the verification-neutral allowlist (`CLAUDE.md` → Page Status; the shared spec `.claude/skills/multi-skill/references/verification-neutral-fixes.md`): they provably change no claim. Before applying, confirm the page is verification-stable — run `python3 .claude/skills/multi-skill/scripts/body_hash.py {page}` and check its output equals the current `verified_hash:` (the same self-check Step 5 applies before skipping a verified page). Only a page that still matches may be neutral-re-stamped; if it does not match, the page drifted out of band — pull it into this run's Step 5 fact-check scope and earn the stamp there. When it matches, apply the fixes and re-stamp `verified_hash:` with `body_hash.py` in the same pass, keeping the page `verified` — no raw re-read. This is what lets a partial audit clear lint's link / hyphenation / spelling worklist across the wiki without demoting dozens of verified pages.

The guard is the allowlist's tightness: wrap an existing plain-text reference only where the rendered display stays byte-identical and the target page exists (never change which page a link targets, or a claim's wording); de-hyphenate only from a published open-compound mapping (the always-open mapping everywhere, the noun-only mapping in bare-noun position) and normalize spelling only in running prose — never inside a verbatim quote (`"…"`), an inline `` `code` `` span, a math span, a title, or a dataset/model identifier, where rewriting the token would change what the page asserts the source wrote. A hyphenated compound or US spelling inside one of those is not claim-neutral: leave it, and if it is the only worklist item, leave the page untouched.

Any worklist item that is not verification-neutral on a verified page — a content rewrite, a split or merge, a citation-target or locator change, a claim correction — is not allowlisted. It either requires fact-checking the page against the raw (the Step 5 front-load promotes the page into fact-check scope and verifies it after editing) or it demotes the page to `draft` for a later audit. These two — verify-in-scope, or fix-and-demote — are the only dispositions for a confirmed non-neutral defect on a `verified` page; deferring the fix to keep the stamp is not a third option. Demotion is the `verified` stamp working as designed: it certifies a specific body, so a real content change retires it. When in doubt whether a fix is claim-neutral, treat it as not — demote, don't re-stamp.

## Grow The Check Data Lists

Two lint checks are heuristic and carry curated lists audit maintains, in two agent-writable data files under `.claude/skills/lint/`. Both are data, not script (editing them is not the "audit does not edit scripts" case), and both are loaded as runtime data, so an entry changes a check's behaviour, not its code.

- `hyphenation-lists.md` drives the bidirectional `hyphenated_open_compound_noun` check with four lists (DISALLOWED open-noun compounds, ALLOWED keep-hyphenated look-alikes, direction-2 HEAD nouns, VERIFIED-IGNORE phrases). Audit maintains all four, learning from what it sees: a confirmed slug-derived open compound → DISALLOWED; a hyphenated proper name or established term a flag would wrongly open → ALLOWED; a genuine head noun → HEADS; a flagged-but-correct use → VERIFIED-IGNORE.
- `unlinked-mention-ignore.md` is the verified-ignore list for `unlinked_page_mention`. An occurrence audit confirms is generic wording — a homograph, a common noun that happens to be a page title, the term inside a larger established phrase — is appended as one `{page path} :: {target stem} :: {phrase}` line under `## verified-ignore`. Entries are page-scoped and phrase-anchored, so an entry never suppresses the same wording on another page, and rewording the sentence re-flags the mention.

**No addition is written on audit's lone judgement.** Before appending any entry, spawn independent sub-agents to verify it — each asked to confirm or refute the addition on its own: that a DISALLOWED compound is genuinely open as a noun and hyphenated as a modifier (not a proper name); that an ALLOWED term is a fixed hyphenation; that a HEAD noun is genuinely a noun; that a hyphenation VERIFIED-IGNORE phrase reads correctly as written; that an unlinked-mention entry's occurrence reads as generic wording rather than a reference to that page. Give each sub-agent the candidate plus read access to judge it in context — the flagged occurrence in its own page and the page it might reference — and let it open the relevant rule (`CLAUDE.md` → Wikilink Format's genuine-vs-generic test) itself rather than a curated excerpt that pre-frames the call: a refuter reads its own ground truth, the same principle the tiered refuter gate follows. Add the entry only on sub-agent consensus (default ≥ 2 of 3 confirm); on a split or refutation, do not add it — leave the case for the user, and for an unlinked mention wikilink the occurrence rather than suppressing it. Append in each file's documented format, and never touch `check_wiki.py` or its test suite — both are edit-only. If a DISALLOWED/ALLOWED/HEAD addition warrants a paired regression case in `.claude/skills/multi-skill/scripts/tests/test_check_wiki.py`, surface it as a proposal rather than writing it (a verified-ignore entry needs none). Both files are version-controlled, so a wrong addition is recoverable. In a fanned-out run these appends and their quorum are deferred to the central post-fan-out pass (Step 8, `references/verify-and-set-status.md`) and spawned from the top-level agent — a fan-out subagent may be unable to nest the quorum, and concurrent appends collide.

## Resolve `*[unverified]*` Markers

When audit fact-checks a claim marked `*[unverified]*`:

- If it passes, remove the marker (leave a co-occurring `*[tentative]*` marker — that is a separate epistemic judgement that persists) — but clear a marker only when the claim's `#page=N` anchor is already at git HEAD (a marker from a prior, committed run), where the relocation exemption covers the marker-removal line change. Do not clear a marker whose anchor is new-versus-HEAD (authored this same, uncommitted run): un-masking it re-fires `verified_anchor_unaudited` (no committed predecessor for the relocation exemption), and clearing-then-keeping-`verified` would be a same-session self-re-stamp — leave it marked this run and let the next post-commit audit clear it.
- If it fails, fix it inline when the raw settles it (a determinate fix, below), or set the page `needs-update` when the failure is beyond audit's reach.

The `verified_hash:` re-stamp that seals a cleared marker is a Step 8 write (`references/verify-and-set-status.md`); clearing the marker text is the Step 7 edit. A page returns to fully-clean `verified` — no pending delta — when no `*[unverified]*` markers remain and its checked content is faithful to the raw; a verified page that still carries unresolved markers keeps them and stays `verified` with that pending delta visible.

## Build The Fix List And Resolve The Authored Worklist

A page's fix list is the union of audit's own Step 4 findings for it and the lint authored-tier findings carried in from Step 1 — resolve every item, not only the ones audit's pass independently surfaced.

**Authored worklist items are resolved this run — deferral is not a disposition.** Every authored-tier finding lint carried in is resolved now, not logged as a "focused follow-up." Run length, token economy, and the fact that a prior run deferred the same backlog are not grounds to defer — a standing authored backlog is skipped work. Batch size is not a deferral ground either: a large run (100+ `unlinked_page_mention` flags) is resolved this run. The genuine-vs-generic triage is the careful judgement; once made, applying it is not rushing. Apply every wrap the triage marks genuine — wrapping an already-genuine reference is a verification-neutral allowlisted edit, so 74 safe wraps are 74 determinate edits — and run the sub-agent quorum for every generic ignore-list entry, batched so a large ignore set stays affordable: each of the (default three) refuters judges the whole batch and returns a per-candidate verdict, so the ≥ 2-of-3 quorum holds per entry in three calls rather than 3×N. Never split the batch across refuters, which would silently drop each entry to a single judgement.

Only two costs may stage across follow-up audits, and only when the worklist genuinely exceeds one run's budget: the Tier-3 re-verification of a page a content edit demoted this run, and the sub-agent quorum for `unlinked-mention-ignore.md` additions. Both stage under one hard invariant (SKILL.md → Verification Model): a page never keeps or earns `verified` while a content-changing worklist item on it is unapplied or its post-edit body unverified. A page whose edit landed but whose re-verification could not finish stays `draft`/`needs-update` with its residual work recorded, and the next `partial` audit re-verifies it. The edits themselves never stage: every determinate wrap, re-voicing, reciprocal mirror, and confirmed split or merge is applied this run.

"Resolve" differs by check, but every disposition is reached this run:

- `missing_reciprocal_contradiction` and `concept_source_asymmetry` clear this run, but which way depends on whether the named page is a genuine party. `concept_source_asymmetry` — and a `missing_reciprocal_contradiction` whose named page is a genuine party — is mutual by nature: author the reciprocal entry or sync the `sources:` link (on a `verified` partner, via the non-demoting `*[unverified]*`-marked mirror below). But a `missing_reciprocal_contradiction` whose named page is an incidental (non-party) mention wikilinked in the callout is the over-count the check cannot distinguish: the cure is the opposite — unwrap the incidental link (never author a false reciprocal), which on a `verified` page is a verification-neutral re-stamp, not a demotion. Confirm the non-party call through the same sub-agent verification the resulting `unlinked-mention-ignore.md` entry requires, and when the party-versus-incidental call is genuinely uncertain, author the reciprocal rather than unwrapping — erasing a genuine contradiction relationship is worse than completing one.
- `unlinked_page_mention` is resolved by a per-occurrence genuine-vs-generic judgement made now: wrap the occurrence when it is a genuine reference to that page; leave it when it is generic wording or a homograph (`chain` inside `AI Chains`, `chat chain`, or `chain-of-thought` is not the chain-topology concept; `the survey` is not a page). A genuine reference left unwrapped is non-compliant. Record a correctly-judged-generic occurrence in `unlinked-mention-ignore.md` (the grow-the-lists clause governs the write) so lint stops flagging it. Leaving the occurrence alone without recording the judgement is a false positive left to recur every run.
- `stale_mention_ignore` is determinate: a verified-ignore entry that suppresses nothing is dead, and lint has proven it. Delete the entry from `unlinked-mention-ignore.md` — no sub-agent verification, since removing an inert entry asserts nothing. The one judgement: if the occurrence still exists but was reworded, re-record it against the current phrase (re-confirming it reads as generic) instead of editing the old phrase to match.
- `intra_page_redundancy` resolves when the content settles which bullet merges or drops; a genuine coin-flip falls to the ambiguity clause. On a `verified` page the merge-or-drop is a real content change, so it fixes-and-demotes: confirm the dropped point survives (re-voiced) in the bullet that stays (the merge-content-conservation check), then drop or merge, strip `verified_hash:`, touch `updated:`, set `draft`; or, when the page is already in Step 5 scope, re-earn `verified` in the same pass. A genuine redundancy on a `verified` page is never deferred as a standing Warning to protect the stamp.

The only finding audit hands back unresolved is a genuinely ambiguous structural call the fact-check cannot settle — not a determinate fix dressed up as one.

## A Reciprocal Mirror Does Not Demote Its Verified Partner

This is the genuine-party branch — an incidental mention is unwrapped instead of mirrored (above), so reach this clause only once the named page is confirmed a real party. When the page lacking the reciprocal entry is `verified`, authoring the mirror onto it is not the generic claim-add that demotes: the disagreement is already established on the page that names this one, and audit confirms it against the raw(s) this run. So author the reciprocal bullet and confirm the mirrored disagreement against the raw — but mark the authored bullet `*[unverified]*` rather than re-stamping the page. That sourced mirror carries a new `#page=N` anchor, and an anchor addition on a `verified` page trips lint's `verified_anchor_unaudited` unless the line is marked (`check_wiki.py` masks only `*[unverified]*` lines, and `CLAUDE.md` → Page Status forbids self-re-stamping an anchor change). The page stays `verified` with that bullet as the pending delta — `body_hash.py` masks the marked line, and the next `partial` audit re-checks and clears the marker. (If the partner side is itself unverified, the page is in Step 5 scope and earns its stamp there.)

## Apply The Fixes

**Determinate fixes** — a finding the raw source or the schema settles outright: a restated number that does not match the raw, a misquoted phrase, a banned-vocabulary word or source-context phrase lint detected and left for audit, a non-obvious claim missing its inline citation or located locator where the fact-check identifies the supporting source and location (write it in one of the two canonical forms, `CLAUDE.md` → Source Support And Verification, with the structural anchor and printed page inside the `#page=N` deep-link and N computed from the printed→physical offset), a claim whose wording overstates or misinterprets the source, a genuine reference to an existing page left unlinked, the source's actual stated reason supplied for a bare causal/comparative/evaluative claim, or a fabricated mechanism removed from a claim that wrongly attributed it to the source. Apply inline. A non-obvious claim that no held source supports is not determinate and is never dropped on a single pass — route it through the bullet-removal guard below. Correct a distortion inline only when the faithful replacement is bounded: a single-bullet reword on one page the fact-check settles outright. A distortion whose faithful correction requires re-deriving an aggregate the pass cannot fully validate, spans more than one page, or produces wording audit cannot certify faithful this run routes to the distortion-disposition rule below, not to a forced inline rewrite.

**Repeated-literal sweep after any citation fix.** Correcting a mis-located or wrong citation on one page usually leaves the identical error on sibling pages — and in `partial` mode some are `verified` pages this run is otherwise skipping. After any citation fix, grep the distinctive literal across `1-wiki/` (`grep -rn "{literal}" 1-wiki/` for a figure or percentage; the distinctive phrase for a qualitative claim) and re-check every occurrence: fix a sibling whose cited page you confirm this run; for a sibling on a `verified` page not otherwise in scope, pull it into Step 5's fact-check scope and verify it, or mark the bullet `*[unverified]*` — a mis-located citation is a support failure, never a verification-neutral edit, so it does not silently ride a `verified` page. Record the literal searched and the sites checked and fixed in the report.

**Authored fixes** — splitting an over-long or multi-idea page, merging near-duplicates, rewriting a source-shaped page, dropping a bullet the raw does not support (only through the bullet-removal guard), authoring a missing contradiction bullet, adding a missing cross-paradigm `Connections`/`Not This` link the relationship sweep surfaces (reciprocal on both pages): carry them out grounded in the fact-check, following ingest's existing-source-mode procedure for source-page content and the `supersede` procedure's preservation and bookkeeping mechanics.

- Each split-off or rewritten page is a new page: author it with the complete frontmatter template for its page type (`CLAUDE.md` → the page-type frontmatter block — a concept/entity page carries `type`, `aliases`, `sources`, `source_count`, `tags`, `created`/`updated`, `status: draft`; a synthesis additionally carries `origin` and, when applicable, `single_source_stub`) and all required callouts with block IDs, and let it earn `verified` only from its own fact-check, never by inheriting a parent's stamp.
- Copy the prior version to `2-outputs/supersede/preserve/`, naming the preserved copy per `.claude/skills/multi-skill/references/quarantine-path-convention.md` (a `-N` clash suffix before the final extension, then `git check-ignore -q` to confirm the copy did not land on a gitignored name — audit routinely does a split and a merge in one run, so a same-minute clash can otherwise silently lose the only preserved copy). Preserve the prior view there; update index/hot/log.
- When a fix removes or renames a page (a merge, a redirect, or a split that retires a slug), find and rewrite the body-prose inbound wikilinks that point at it, following `.claude/skills/multi-skill/references/inbound-reference-discovery.md` — `check_wiki.py` does not see prose wikilinks, so an inbound link left unrewritten dangles silently until a later lint.
- Inherit those mechanics but not supersede's approval gates: audit is the autonomy exception and asks the user nothing.

**Bullet removal requires three independent confirmations** (the three-pass non-support re-read — a separate counter from the three-round rewrite re-check loop in Step 8). Never drop a bullet on a single fact-check: one pass produces false negatives (it misses supporting text present in a wrong section, a paraphrase mismatch, an unchecked appendix), and a deleted-but-correct bullet is silent data loss the quarantine copy records without preventing. A claim one pass finds unsupported is a removal candidate, not a removal: re-confirm non-support with two further independent re-reads of the raw (three in total), each reopening it with a different access strategy (a different section, different search terms, the appendix and any offloaded tables or prompts). Treat their agreement as false-negative reduction, not proof of non-support. A pass that cannot find support because the relevant region is unreadable — file missing or moved, image-only scan, OCR-garbled, truncated extraction, encrypted, or paywalled (the Step 5 coverage-gate conditions) — is void, not a confirming pass: unreadability is not absence, and three passes over one blind spot are one pass. When non-support traces to an unreadable region, never drop the bullet — mark it `*[tentative]*` and set the page `needs-update` with a `needs_update_reason:` naming the unreadable region. Drop the bullet only when all three passes independently agree the raw legibly does not support it (then quarantine it via the authored-fix mechanics). If any pass finds support, do not drop it — apply the determinate fix that support enables. If the passes disagree, or support stays uncertain after three, mark the claim `*[tentative]*` and set the page `needs-update` naming the unresolved support question, leaving the bullet visible. This gate governs every bullet-dropping fix — the determinate "no held source supports it" case, the authored "drops a bullet the raw does not support" case, and resolving an `*[unverified]*` claim by deletion alike: preserve under doubt; deletion is the last resort, never the default.

**A confirmed distortion audit cannot safely correct in-pass — the distortion-disposition rule — is set `needs-update`, not force-fixed and not deferred to the user.** A distortion whose faithful correction is not bounded to a single-bullet reword — it requires re-deriving an aggregate the pass cannot fully validate (recomputing a whole table or evidence base — `references/verification-spec.md`, the summary-claim discipline), the same distorted framing is inherited across more than one page (a source page and the concept/synthesis pages that lifted its wording), or audit cannot produce replacement wording it can certify faithful this run — is not force-fixed: a wrong rewrite re-certifies a fresh distortion, the asymmetric error the confirm-before-fixing guard prevents. Nor is it left as a passive finding for the user. Set every page the distortion touches `needs-update`, with a `needs_update_reason:` carrying the full derivation — the recomputed aggregate, the specific cells, and what the faithful claim must instead say — so the re-ingest or the user receives a precise hand-off, not a re-check. Find inherited siblings the way the repeated-literal sweep finds inherited citations — grep the distinctive phrase of the distorted framing across `1-wiki/` — and set each `needs-update` too. This is the general rule behind every status disposition: a confirmed defect always ends the run acted on — fixed when the correction is a safe in-pass determinate edit, set `needs-update` with a derivation when it is not. The one confirmed-defect case handed back as a bare Warning is the genuinely ambiguous structural call below.

**When a structural call is genuinely ambiguous** — is this one idea stated twice or two distinct ideas? does the merge target page survive or the merged-away one? — and the fact-check does not settle it, do not perform the surgery. Downgrade the finding to a Warning that names the options and leave the page for the user. Act autonomously only when the fact-check makes the structural call determinate; preserving the prior version is necessary but not sufficient licence to act on a coin-flip.

**Carry the markup on anything you author or move.** Each authored or moved callout carries its block ID as the last line inside the `>` block (`> ^idea`, `> ^evidence`, …; `CLAUDE.md` → Callout Block IDs). Each page locator you write or move carries its `#page=N` raw-file deep-link, in the form `CLAUDE.md` → Source Support And Verification defines for the page type:

- **Source page — one form** (`CLAUDE.md` → Source page rules): every source-page locator — `Evidence`/`Method` pointers and in-body non-obvious claim citations alike — carries the structural anchor and page together inside the `#page=N` deep-link display (`[[0-raw/papers/<stem>.pdf#page=N|sec. 4, p. M]]`, `[[0-raw/papers/<stem>.pdf#page=N|fig. 1, p. M]]`), used as a sentence subject, a leading `Evidence` label, or a trailing parenthetical. The source key is dropped (it would be a self-link). Do not split the anchor outside the link (`sec. 1, [[…#page=N|p. N]]`) or flatten to a page-only `|p. N` display — both drop the section the schema requires (lint `source_locator_incomplete`).
- **Concept/entity/synthesis page — the two canonical forms** (Form 1 attributive / Form 2 parenthetical, defined in `references/semantic-checks.md` and `CLAUDE.md` → Source Support And Verification): the source-page wikilink names which source, the parenthesized raw deep-link carries the anchor + page in its display. Form 2 wraps the whole citation in round brackets — `… claim ([[…|key]]; [[…|sec. X, p. M]])`.
- **The unpaginated-supplement exemption — on every page type** (`CLAUDE.md` → Source Support And Verification). Where the cited region genuinely carries no printed page number (common in published appendices), `p. M` cannot be stated honestly, so the display cites its structural anchor alone — any anchor, not only `app.`, keyed on the pagination map where the cited physical page prints no number: `[[0-raw/papers/<stem>.pdf#page=16|app. D.1, tab. 8]]`. Never resolve a missing `p. M` by writing the physical page as though it were printed — that fabricates a number no reader can find, and on a body-paginated paper it lands beside a genuine five-digit printed page from the same source. When audit reopens a raw and finds the cited supplement unpaginated, the fix is to drop `p. M`, not to invent one. Migrating an existing `app. …, p. N` locator to the anchor-alone form is a locator change, so it follows the rule above: mark the bullet `*[unverified]*` or demote — audit never self-re-stamps a locator change. Confirm the absence of printed pagination in the raw before applying the exemption, and never use it to dodge a printed page that does exist.
- Keep every locator inside a `[[…]]` display or a backticked span (a non-PDF uses a backticked per-type locator in place of the `#page=N` deep-link), with N the physical PDF page — a bare locator left as plain text after the link trips lint's `page_locator_unlinked`.

**Touch `updated:`** on every page whose body audit edits — inline and authored fixes alike. (The only exception is the full-mode no-correction case in SKILL.md → Verification Model, where a verified page is left unchanged and keeps its `updated:` and `verified_hash:`.)

**Strip a stale `verified_hash:`** on any page audit demotes to `draft` or `needs-update` — a non-`verified` page carries no `verified_hash:`; it is re-added only when audit next sets the page `verified` (Step 8).

Step 7 ends with every in-scope page edited and every `verified` page a content edit targeted demoted into scope — nothing stamped. Step 8 (`references/verify-and-set-status.md`) re-verifies the rewritten bodies and sets each page's status.
