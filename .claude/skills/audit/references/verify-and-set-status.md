## Re-Fact-Check Anything You Rewrote

A split, merge, source-shaped rewrite, or dropped bullet (all applied in Step 7) produces a new body Step 5 never checked — audit's own edit is a fresh compression step that can drop a subtle distinction the raw made. Before stamping `verified`, re-verify the rewritten body against the raw source. This re-verification is same-agent, so it does not bound a raw-misreading that produced the rewrite — the three-round loop below adds iterations, not independence. A rewritten non-obvious claim is certified only through the tiered independent-refuter gate like any other (three agreeing refuters for the broad default; `references/verification-spec.md`, `CLAUDE.md` → Page Status) — the same mechanism, not a weaker one, because a rewrite is exactly where a same-agent re-read is most circular. Run the rewritten bullet through its tier's refuters before stamping; on any refutation or split, leave it `*[unverified]*` for the next independent pass rather than self-stamping.

Three checks beyond raw-faithfulness apply to any rewritten body before stamping:

- **Semantic re-check.** Step 4's catalogue ran before this authoring existed, so an audit-authored split-off, merge survivor, or rewrite has never had its authored prose seen by any semantic pass — re-run the Step 4 catalogue (`references/semantic-checks.md` — one-idea atomicity, plain language, citation form, semantic AI-writing tells, reworded intra-page redundancy) over each authored or rewritten body before stamping. The confirming lint catches only the mechanical tells, not these.
- **Structural re-validation.** Audit's clean-lint precondition (Step 1) held before these edits, but audit's own surgery can re-introduce structural drift, and the re-fact-check only checks faithfulness. Run `python3 .claude/skills/multi-skill/scripts/check_wiki.py "1-wiki"` over the touched pages and confirm the edit introduced no new Critical/Warning — a split that drops a required callout or block ID, a merge that leaves a dangling inbound link, a malformed frontmatter delimiter. A faithfulness-clean but structurally-broken page must not be stamped: fix the structural finding, or set `needs-update`.
- **Merge content conservation.** Before stamping a merge survivor, confirm every load-bearing bullet from each merged-away page survives (re-voiced) in the survivor, or was deliberately dropped as redundant. The raw-faithfulness check passes a survivor that is internally accurate yet silently lost a parent's claim the raw under-emphasizes.

Bound this loop: re-verify, fix, and re-check at most three rounds per rewritten page (the three-round rewrite re-check loop — a separate counter from the three-pass non-support re-read in Step 7; mirroring ingest's fix-and-rerun cap and lint's regression guard). If the page is still not faithful after three rounds, stop — do not loop and do not stamp `verified`; set the page `needs-update` with a `needs_update_reason:` naming the residual failure. Reaching the cap without convergence is the one point audit may hand back to the user for more rounds: when the rounds are making genuine progress (each clears findings and re-introduces none) but have not converged, audit may surface the residual findings via `AskUserQuestion` and, on approval, run a further bounded batch (three more rounds per approval, re-offering at each cap). Absent an interactive user (an unattended run such as `/checkup`) or absent approval, take the safe default: stop and set `needs-update`. Oscillation is different and is never extended: a fix that re-introduces a finding a prior round cleared is cycling, and more rounds cannot fix what undoes itself — stop immediately, set `needs-update`, and record the oscillation.

## Central Locator-Format Pass

After a fanned-out audit, run one central locator-format pass before any `verified_hash` re-stamp. When fact-checking is parallelized (one subagent per raw source), subagents apply the citation format divergently and mislabel sections — a known failure mode this pass guards against. Centrally re-verify every `#page=N` and section anchor against the raw (never trust a subagent's section label or page number), confirm each locator sits inside its `[[…]]` display or a backticked span, and normalize to `sec.` (not a bare `§`). Run this pass after the re-fact-check and the structural re-validation, immediately before the stamp, and apply all shared-file writes — `index.md`, `log.md`, co-cited pages, and every append to the two agent-writable shared data files (with the sub-agent-verification quorum from the grow-the-lists clause in `references/apply-fixes.md`, deferred here from Step 7) — centrally from the top-level agent rather than from individual fan-out subagents, so parallel writes never collide and the quorum is never nested. Only then stamp.

Lint emits the Critical `verified_anchor_unaudited` only when a page that was `verified` at git HEAD carries a locator whose anchor was added or relabelled versus HEAD; a pure relocation of an already-present anchor for the same `#page=N` is exempt, and the check does not fire on a page audit is promoting draft/needs-update→`verified` this run — the head-status gate, since the promotion is itself the fact-check. So when audit's locator work adds an anchor or relabels which section or page one names on an already-`verified` page, mark the bullet `*[unverified]*` or demote — never self-re-stamp. Marking satisfies "never leave a verified page tripping this Critical": `check_wiki.py` masks a marked line. Leave that marker in place this run — the anchor is still new-versus-HEAD, so clearing un-masks the line and the Critical re-fires; the next `partial` audit, after this run commits so the anchor sits at HEAD, re-checks and clears it.

## Execution Order For A Rewritten Page

Before any stamp: (1) apply the fixes (Step 7, `references/apply-fixes.md`); (2) re-fact-check the rewritten body against the raw; (3) the three pre-stamp checks — semantic re-check, structural re-validation, merge content conservation — within the three-round cap; (4) the central locator-format pass; (5) set status / stamp.

## Set The Status

**Set `verified`** only when, after the fixes and the re-check, the page is fully faithful to its raw source with confirmed full-text coverage (Step 5 coverage gate) and every non-obvious claim on it has passed the tiered independent-refuter gate (Tier 3 — three agreeing refuters — for the broad default; `references/verification-spec.md`, `CLAUDE.md` → Page Status). A claim that did not clear its tier's refuters is not certified: it stays `*[unverified]*` and the page is promoted only on its cleared content, or set `needs-update` if the failure is load-bearing. The same re-stamp applies to an already-`verified` page whose `*[unverified]*` markers audit cleared this run (Step 7): stamp `verified_hash:` with the body's hash, computed after the fixes with `python3 .claude/skills/multi-skill/scripts/body_hash.py {page}` — `body_hash.py` masks any `*[unverified]*` claims that remain, so the stamp covers only checked content.

`body_hash.py` needs well-formed frontmatter delimited by `---` lines before hashing: it silently hashes the whole file when there is no opening `---` (lint then reads a mismatch and demotes the page), and aborts with a non-zero exit when an opening `---` has no closing `---`. Before hashing, confirm the page begins with a `---` line and has a matching closing `---`; if the delimiters are missing or malformed, repair them or abort the promotion rather than stamping a hash of the whole file (a valid 64-hex return is not proof the body was isolated). If `body_hash.py` is missing, exits non-zero, or does not return a 64-character hex, abort the promotion for that page, leave it non-`verified`, and report it as a Critical — never write an empty or unverified `verified_hash:`.

**Set `needs-update`** when the page carries an unresolved problem: a residual issue beyond audit's reach (the source page needs a raw re-extraction — a job for ingest's existing-source mode — or the page needs a source the wiki does not hold), a coverage failure (Step 5), a confirmed faithfulness or support defect whose faithful correction audit cannot safely produce in-pass (the distortion-disposition rule in `references/apply-fixes.md` — re-derivation beyond the pass, a multi-page inherited framing, or wording audit cannot certify), OR a genuine unresolved cross-source contradiction the fact-check did not resolve, including one audit surfaced or authored this run. Making a contradiction visible is not resolving it (`CLAUDE.md` → Concepts and Entities). A `needs-update` change must carry an explanation — either a real `Contradictions`/`Tensions` entry on the page, or a one-line `needs_update_reason:` naming exactly what the re-ingest or the user must resolve. A precise hand-off, never a bare label, and never both where one suffices. If the page already carries a `needs_update_reason:`, append the new cause rather than overwriting it (the shared rule in `.claude/skills/multi-skill/references/dependent-cascade.md`).

Never resolve a contradiction by deleting one side — preserve genuine disagreement. Every fix and every status change is recorded in the audit report.
