# audit

Audit is the judgement pass. It asks whether the notes are good thinking objects.

## Purpose

Structural lint can say whether a page has the right sections. Audit says whether a concept/entity page is one idea, understandable, supported, and useful.

`verified` here means fact-checked against the raw source and passed. It is not the Ahrens-style human re-voicing gate that the foundation design (`a-archive/reference/smart-notes-llm-wiki-integration.md`) assigns to promotion. This wiki deliberately relocates that safeguard: pages stay LLM-drafted, and the defence against AI-voice prose lives in the ingest note-quality packet plus audit's own semantic-tell check (Step 4), not in a human rewrite at promotion. Audit's autonomy is therefore a conscious departure from that document's human-commit-point model, authorized by `CLAUDE.md`. Do not weaken the Step 4 tell-check or the ingest packet without re-introducing a human re-voicing step in their place.

Four load-bearing terms recur below (full rules in `CLAUDE.md` → Page Status, Bullet Markers):

- `verified_hash:` — the SHA-256 of a page body with `*[unverified]*` claim lines excluded. It guarantees the page's unmarked content is byte-identical to what was last fact-checked: immutability since the check, not correctness of the check. Re-confirm correctness with `full` mode.
- `*[unverified]*` — a claim-level marker on a non-obvious claim added since the last fact-check (or one whose backing image was re-extracted without a text change), pending a raw re-check. A *changed* claim demotes the page rather than riding it, so a marker on a `verified` page is an addition, not an edit. It is the unit `partial` mode re-checks, and the hash excludes it so it can ride a `verified` page as the pending delta.
- `*[tentative]*` — a claim-level marker for weak or thin support. Unlike `*[unverified]*` it is an epistemic judgement, not a process state: a fact-check does not clear it (it persists on a checked but thinly-supported claim), and the two can co-occur on one claim. It is the safe-failure marker Step 7 falls back to whenever the raw cannot positively settle a claim's support — mark `*[tentative]*` and set `needs-update` rather than delete or self-stamp.
- authored-tier worklist — the Warning-tier findings lint detects but leaves for audit to carry out. Audit's authored worklist *is* lint's Warning output, so a Warning check lint adds later automatically becomes audit's job with no edit here.

## Arguments

`audit` accepts one optional positional argument selecting the verification scope:

- `partial` (default; equivalent to no argument) — fact-check every non-verified page (every `draft` and every `needs-update`) **and** every `*[unverified]*`-marked claim on an otherwise-`verified` page (the pending delta from edits made since the last check — CLAUDE.md → Page Status, Bullet Markers). It no longer fully skips verified pages: it skips a verified page's *unmarked* (already-checked) claims — the schema invariant guarantees those bodies have not changed since the last fact-check (`verified_hash:` excludes marked claims) — but it still processes that page's `*[unverified]*` claims. A verified page with no markers is skipped entirely for fact-checking (Step 7 may still apply an allowlisted verification-neutral worklist fix to it — a link, a de-hyphenation — and re-stamp, with no raw re-read). This claim-level scope is what keeps audit affordable as the wiki grows.
- `full` — fact-check **every claim on every page**, verified or not, including the unmarked claims a `partial` run trusts. Use after a schema rule change that retroactively affects already-verified content (e.g., a new style rule, a new rule about what belongs in a callout), when you have reason to suspect a prior audit was rushed, or as a periodic deep-confirmation pass. Verified pages whose fact-check still holds keep their existing `verified_hash:` unchanged; only pages that need corrections (or carry markers to clear) get fixed and re-stamped.

Invoke as `audit` (partial, the default) or `audit full`. Any other argument is unrecognized. If the token looks like a page path or name (contains `/`, ends in `.md`, or matches an existing page title), the caller likely wants to scope audit to one page — but audit has no single-page or target mode and would otherwise run an autonomous whole-wiki pass, so confirm scope via `AskUserQuestion` before proceeding rather than silently running the full pass and disclosing the ignored token only in the after-the-fact report. Any other unrecognized token: default to `partial` and note the ignored token in the report — audit has no single-page or target argument. `/checkup` invokes audit in `partial` mode to preserve the claim-level verified-hash optimization across routine runs.

## When To Invoke

Use before high-stakes synthesis, paper drafting, major reference work, or after large ingests.

## When Not To Invoke

- No recent lint report exists. Run `lint` first.
- Lint has unresolved critical structural findings other than expected `needs-update` pages.
- No recent consistency report exists, or the project has changed `CLAUDE.md`, `README.md`, or `.claude/skills/` since the last clean consistency run. Run `consistency` first — auditing pages against a drifted schema produces findings against rules the project no longer agrees on.
- The user wants a research answer. Use `query`.
- The wiki is too small to audit meaningfully.

## Procedure

```text
Audit Progress:
- [ ] Step 1: Verify lint; collect its authored-tier worklist
- [ ] Step 2: Verify consistency is clean and recent
- [ ] Step 3: Load memory; read wiki pages and raw sources
- [ ] Step 4: Run semantic checks
- [ ] Step 5: Fact-check non-verified pages and `*[unverified]*` claims against raw
- [ ] Step 6: Draft audit report
- [ ] Step 7: Fix, then set status
- [ ] Step 8: Finalize report; update index, hot, and log
```

1. **Verify lint first.** Read the newest `2-outputs/lint/lint-*.md` and gate on its frontmatter `result:` field. If `result: blocking` (equivalently `audit_blocking:` > 0), stop and ask the user to clear the findings listed under the report's `## Audit-Blocking` section. `result: clean` means no audit-blocking finding remains — lint excludes the standing repo-state items audit cannot act on (`raw_without_source_page`, `uningested_raw_source`, `status_needs_update`) from the blocking count, so a clean lint can still carry those Criticals — the same three the manual-classification fallback below names, matching lint's `STANDING_NONBLOCKING` set. If the report predates this field (no `result:` in its frontmatter), classify manually: a Critical is audit-blocking unless its `check_id` is `raw_without_source_page`, `uningested_raw_source`, or `status_needs_update`; Warnings and Info never block; when a critical's class is unclear, treat it as blocking. **Recent means the lint is at least as new as the most recent log entry that touched a wiki page** (read `1-wiki/log.md` to determine this), with no uncommitted working-tree wiki edit unaccounted for — also run `git status --porcelain -- 1-wiki`, mirroring Step 2's git-status hardening for the schema, since a hand edit in the editor need not have written a log entry and would otherwise ride through this gate against a drifted wiki. A non-empty result is not itself the stop signal: no check commits mid-run, so the gating lint's own auto-fixes and its `log.md` / `hot.md` / `index.md` bookkeeping — and any page a prior audit edited before that lint in an `audit → lint → audit` cycle — are expected to sit uncommitted and are exactly what the newest lint report accounts for. Stop only on an uncommitted `1-wiki` change the pipeline does not account for: a wiki page whose content changed after the gating lint, named in neither that lint report's Auto-Fixed list nor a wiki-touching `log.md` entry at or before its timestamp (a genuine post-lint hand edit lint never saw). In that case, stop and ask the user to re-run lint before audit; a tree dirtied only by the pipeline's own uncommitted output does not block, so this gate does not fire on the ordinary post-lint state or mid-chain inside `/checkup`. (Audit's Step 5 body-hash self-check and per-page fact-check independently backstop a drifted page that slips past this early gate.) A six-month-old "newest" lint against a wiki that's drifted since then is not a basis for audit.

   Then collect lint's open authored-tier findings — **every Warning-tier finding in lint's report** (the ones lint detects but, by design, leaves for audit to carry out): e.g. missing reciprocal contradictions, length-cap review (a `length_cap_exceeded` page is split only when it genuinely holds more than one idea — length alone never forces a split, and a page long only from inline-citation density is not over the cap; `CLAUDE.md` → Length), source-context-phrase / vague-source-referent / unlinked-existing-page-mention (`unlinked_page_mention`) / mechanical AI-writing-tell rewrites, and bare page locators needing a `#page=N` deep-link (`page_locator_unlinked` — audit reopens the raw anyway, so it can compute the printed→physical page offset: `N = printed − (first_printed − 1)`, where `first_printed` is the printed number on the raw's first physical page; never guess offset zero on a source not paginated from 1, e.g. ACL/NAACL proceedings). Take lint's Warning tier as the worklist, not a fixed sub-list, so a Warning lint adds later is still carried in. **Info-tier findings are not audit's to action** — in particular the user-owned rename/re-path checks `filename_not_kebab`, `source_stem_mismatch`, and `bare_basename_link`, whose only safe fix is a file rename or link re-path the user performs (renaming is a safety operation; audit does not rename files). These Warnings join audit's worklist for Step 7. A finding lint deterministically caught must be resolved even when audit's own semantic pass (Step 4) does not independently re-notice it — here the lint report is a worklist, not just a gate.

2. **Verify consistency next.** Read the newest `2-outputs/consistency/consistency-*.md` and gate on its frontmatter `result:` field, the same way Step 1 gates on lint's. If `result: findings` or `result: blocked`, stop and ask the user to clear consistency. `result: clean` means no auto-fixable drift remains — a clean consistency report can still carry `proposals:` > 0 (root-level proposals awaiting the user); those do not block audit. If the report predates this field (no `result:` in its frontmatter), fall back to the prose check: if any unresolved findings remain in the body, stop. **Recent means the consistency report is at least as new as the most recent change to `CLAUDE.md`, `README.md`, or `.claude/skills/`** — check via `git log -1 --format=%ad -- CLAUDE.md README.md .claude/skills ':(exclude).claude/skills/lint/hyphenation-lists.md' ':(exclude).claude/skills/lint/unlinked-mention-ignore.md'`, and also `git status --porcelain` over that same pathspec for uncommitted edits the git-log date misses — the two excluded paths are the agent-writable lint *data* files audit itself grows in Step 7 (runtime data, not schema `consistency` re-checks, and already exempt from consistency's own content scan), so audit's own sanctioned writes to them are never grounds to re-run consistency; every other `.claude/skills` change (and `pagination-map.md`) still trips the gate (a git-log date does not move on an uncommitted working-tree change, so a stale report would otherwise pass this gate against a changed-but-uncommitted schema — the live pre-release-cleanup case). If those committed changes postdate the last consistency run, or any uncommitted change to them exists, stop and ask the user to re-run consistency before audit. Auditing pages against a drifted schema produces findings against rules the project no longer agrees on.

3. **Load memory, then read the wiki and the raw sources.** First read `.claude/skills/audit/audit-memory.md` and `.claude/skills/multi-skill/multi-skill-memory.md` to apply prior corrections about scope, fact-check aggressiveness, and which judgement calls audit has been told to make differently in the past. Then read source pages, concept/entity pages, syntheses, `hot.md`, `index.md`, and `log.md`. For the raw-source fact-check, reopen the raw sources behind the pages with pending work (Step 5) — every non-verified page, plus any otherwise-`verified` page carrying `*[unverified]*` claims. Read each raw source once and check every `draft` or `needs-update` page that traces to it, plus the marked claims on any verified page that traces to it. A verified page with no `*[unverified]*` markers needs no raw re-read in `partial` mode.

4. **Run semantic checks** across all pages, verified or not — these are cross-page and prose-quality checks that need no raw re-read, so they apply even to pages Step 5 will skip.

   The full Critical / Warning / Info check catalogue is in `references/semantic-checks.md` — run every check in it. Critical findings are support/faithfulness failures; Warnings are the authored-worklist items Step 7 carries out; Info are candidates and suggestions.

5. **Verification pass — fact-check pending claims against the raw source.** The pending work is every non-verified page (`draft` and `needs-update`) plus every `*[unverified]*`-marked claim on an otherwise-`verified` page (CLAUDE.md → Page Status, Bullet Markers). Scope depends on the run mode (see Arguments):
   - `partial` (default): fact-check every non-verified page in full, plus the `*[unverified]*` claims on verified pages. A `verified` page's *unmarked* content was fact-checked on an earlier audit, and the schema resets a page to `draft` when its unmarked (checked) content changes (CLAUDE.md → Page Status, the `verified_hash:` hash excludes marked claims), so unmarked claims on a page still marked `verified` have not changed and need no re-check. But an `*[unverified]*` claim is the pending delta the editor added without checking — it rides on a `verified` page precisely because its hash is excluded — so `partial` re-checks just those marked claims. A verified page with no `*[unverified]*` markers has no pending claims and is skipped entirely for the fact-check (Step 7 may still apply an allowlisted verification-neutral worklist fix and re-stamp, no raw re-read).
   - `full`: fact-check every claim on every page, verified or not — including the unmarked claims `partial` trusts. Use when a schema rule change retroactively affects previously-verified content, after suspected rushed prior audits, or as a periodic deep-confirmation pass. For verified pages, only re-stamp `verified_hash:` if the fact-check surfaces a correction or clears a marker; otherwise leave the existing hash and `updated:` unchanged.

   **Treat a page marked `verified` that has no `verified_hash:`, or whose hash does not match its body, as non-verified for this run** and fact-check it regardless of mode. An unstamped or stale-stamped `verified` is not trustworthy; lint normally demotes it first, but do not rely on that having happened: in `partial` mode, before skipping any `verified` page as unchanged, run `python3 .claude/skills/lint/scripts/body_hash.py {page}` yourself and compare to its `verified_hash:` — audit owns this check rather than assuming lint ran it this cycle, so a crashed or stale prior lint cannot let a drifted `verified` page ride through skipped. If `body_hash.py` is missing, exits non-zero, or returns a non-64-hex value here (e.g. on malformed or unclosed-`---` frontmatter), do not skip the page and do not let the error abort the run: treat it as non-verified, fact-check it this run, repair the delimiters or set it `needs-update` with a `needs_update_reason:` naming the malformed frontmatter, and surface a Critical — the same handling Step 7's promotion path applies.

   **If no non-verified pages and no `*[unverified]*` claims exist** (a fully-verified wiki with no pending markers, in `partial` mode), there is nothing to fact-check. Still run the Step 4 semantic checks across all pages, write the report with empty status-change sections, and note that the wiki was fully verified. Do not invent work to fill the pass. A `verified` page that carries `*[unverified]*` claims is *not* in this clause — those marked claims are pending work in `partial` mode and must be fact-checked.

   For each non-verified page, reopen the raw source — not just the wiki source page — and verify the page in full. For an otherwise-`verified` page that carries `*[unverified]*` claims, audit need only reopen the raw for those marked claims and check them; the page's unmarked content was already verified and is not re-checked in `partial` mode.

   The verification spec — the coverage gate (a positive full-text probe plus a `#page=N` spot-check before any `verified` stamp), the tiered independent-refuter gate that earns `verified` (three agreeing refuters on every non-obvious claim — the broad default — spawned read-only from this top-level agent; `CLAUDE.md` → Page Status), the per-page-type callout checklist, the claim-by-claim support / faithfulness / reasoned axes, the page-scoped citation check (verify each cited claim at the page it cites — never a document-wide search — with every cross-source-bullet locator opened, not sampled), and the yes/no audit-assumption framing for every Critical (and any acting Warning) — is in `references/verification-spec.md`. Run it for each page in scope.

6. **Draft the audit report** at `2-outputs/audit/audit-YYYY-MM-DD-HHMM.md` — or `2-outputs/audit/audit-YYYY-MM-DD-HHMM-full.md` for a `full`-mode run, so a deep full pass is distinguishable from routine partials in the folder. The `-full` is the registry's `-extra` suffix (it sits *after* the timestamp, so `file_naming_consistency` accepts it with no script change); never put the mode marker before the date. Obtain the timestamp at save time by running `TZ='UTC' date '+%Y-%m-%d-%H%M'` — the session context provides the date but not the current minute. This is a draft: write the findings now, but the **Status changes applied** section records what Step 7 actually did, so finalize it in Step 8 after the fixes run — never let it claim a change that a partial Step 7 failure left undone.

   Record the run **Mode** (`partial` or `full`) in the report so a later run can tell when the wiki was last fully re-checked. Then check whether a `full` pass is overdue: run `git log -1 --format=%ad -- CLAUDE.md a-archive/style` (plus `git status --porcelain -- CLAUDE.md a-archive/style` for uncommitted edits the git-log date misses, the same hardening Steps 1–2 carry) and compare against the most recent `full` audit in `2-outputs/audit/` history. If schema or style rules changed since the last `full` audit, note under **Recommendations** that a `full` pass is due — a `partial` run trusts the unmarked (already-checked) content on verified pages, which the rule change may have invalidated.

7. **Fix, then set status — autonomously.** For every page audit fact-checked in Step 5 — every non-verified page, and every otherwise-`verified` page whose `*[unverified]*` claims audit re-checked — act without asking the user. Audit also applies allowlisted verification-neutral worklist fixes to verified pages not otherwise in scope, re-stamping rather than demoting (the verification-neutral clause below).

   **Confirm the defect before fixing it. A suspicion is not a finding.** This guard governs every fix below, and it exists because audit's errors are asymmetric. A *missed* defect stays on the page, visible, for the next run to catch. An *invented* defect does the opposite: audit overwrites correct content, re-stamps the page `verified`, and every later `partial` run trusts it — the error is now certified and self-concealing. **A false-positive fix is strictly worse than the defect it imagines.** So the burden of proof runs one way: before changing existing content, establish the defect **positively against the raw** — never from a pattern, a plausible-sounding inference, or a prior report's say-so.

   **Exception — a mis-located citation breaks the asymmetry, so silence is not a safe failure for it.** The guard above rests on "a missed defect stays visible for the next run to catch." A citation-location defect does not: a true, well-formed claim pinned to the wrong page is stamped `verified` along with the rest of the page, its `verified_hash` is written, and every later `partial` run skips it — the defect is certified and self-concealing, exactly the outcome the guard is built to prevent, but reached by inaction rather than a bad edit. So for a cited claim you cannot confirm at its cited page, "leave it alone and let the next run catch it" is false comfort. The resolution is not to lower the confirmation bar and wave the citation through on the claim's plausibility (its plausibility is what certifies the error) — it is to *require* the confirmation: open the cited page (never a document-wide search — a distinctive figure or name recurs elsewhere in the same document and would falsely confirm) and check the claim is there. Confirm it and the page is verifiable on that bullet; cannot confirm it this run and you mark the bullet `*[unverified]*` (the page stays promotable on its verified content, the claim rides as the pending delta) rather than stamping it. This does not contradict "confirm the defect before fixing it": both demand opening the raw at the exact place — the guard stops you inventing a defect on a page you did not check, and this exception stops you certifying a citation on a page you did not open. What neither permits is passing a cross-source citation on plausibility alone.

   - **Per-instance, never per-pattern.** "This class of anchor is usually off by one" is a hypothesis, not evidence. Confirm the *specific* locator, claim, or bullet is wrong by opening the raw at that place. Sections and tables routinely straddle two physical pages, so a locator naming *either* page can be correct; a table floats away from the subsection that discusses it; a section that "should" be 3.3 may genuinely also be on 3.2's page. Each of those reads as an off-by-one to a pattern-matcher and is not one.
   - **Never inherit a finding from a prior report.** Re-derive it, or drop it. A finding copied forward carries the earlier run's errors with laundered authority — and a wrong "correction" recorded as a correction is the hardest kind to see, because it arrives pre-justified. Re-check any figure, count, or defect a previous audit claims to have fixed; that is exactly where a `full` pass earns its cost.
   - **A carve-out is confirmed against every instance before any bulk application.** A rule that genuinely holds for one source (an appendix with no printed pages) may be false for the next (one that restarts at 1, or continues the body's numbering). Applying it on faith to the whole corpus destroys the correct instances. Check each source; expect the corpus to split more ways than you assumed.
   - **When a check's own tooling supplies the evidence, test the tooling first.** A regex or script that "finds" a defect is a hypothesis until you have shown it does not also match correct content. Run it against a known-good case before believing a known-bad one.

   A defect you cannot confirm is **not** fixed: report it as a suspicion in the report's Warning tier, or mark the bullet `*[unverified]*` and leave the content alone. Silence is a safe failure; a confident wrong edit is not. When the fix is structural (a split, a merge, a dropped bullet, a rewritten claim), state the load-bearing inference as a yes/no audit assumption in the report (`references/verification-spec.md`) so it can be checked rather than rationalized.

   **Verification-neutral worklist fixes on verified pages not otherwise in scope.** In `partial` mode, lint's authored worklist (Step 1) often names a verification-neutral fix — an unlinked genuine reference to an existing page (`unlinked_page_mention`), an open compound to de-hyphenate (`hyphenated_open_compound`, or a slug-derived open compound via the bidirectional `hyphenated_open_compound_noun` — which opens a hyphenated bare noun and, inversely, re-hyphenates an open compound used as a modifier before a head noun; both are heuristic, so audit confirms the noun-vs-modifier call against the use before applying — a context judgement against the page's own prose, not a raw fact-check, so a genuinely ambiguous direction-2 re-hyphenation on a `verified` page is marked `*[unverified]*` (the page stays `verified` with that claim pending) rather than self-re-stamped — or records a false positive in the check's verified-ignore list), or a US spelling to normalize — the three verification-neutral fix categories CLAUDE.md → Page Status assigns to audit (the four format fixes — `callout_block_id`, `wikilink_pipe_spacing`, `citation_bracket_style`, `embed_not_isolated` — are lint's to apply and re-stamp, not audit's) — on a `verified` page that carries no `*[unverified]*` markers and so is *not* in Step 5's fact-check scope. These edits are on the verification-neutral allowlist (CLAUDE.md → Page Status): they provably change no claim. Before applying, confirm the page is verification-stable — run `body_hash.py` and check its output equals the current `verified_hash:` (the same self-check Step 5 applies before skipping a verified page); only a page that still matches may be neutral-re-stamped. If it does not match, the page drifted out of band — pull it into this run's Step 5 fact-check scope and earn the stamp there rather than re-stamping over an unchecked body. When it matches, apply them and **re-stamp** `verified_hash:` with `body_hash.py` in the same pass, keeping the page `verified` — no raw re-read. This is what lets a partial audit clear lint's link / hyphenation / spelling worklist across the wiki without demoting dozens of verified pages or paying a full re-verification. The guard is the allowlist's tightness: wrap an existing plain-text reference only where the rendered display stays byte-identical and the target page exists (never change which page a link targets, or a claim's wording); de-hyphenate only from a published open-compound mapping (the always-open mapping everywhere, or the noun-only mapping in bare-noun position), and normalize spelling, only in running prose — never inside a verbatim quote (`"…"`), an inline `` `code` `` span, a math span, a title, or a dataset/model identifier, where rewriting the token would change what the page asserts the source wrote (CLAUDE.md → Page Status, the text-content exclusion). A hyphenated compound or US spelling inside one of those is not claim-neutral: leave it exactly as written, and if it is the only worklist item, leave the page untouched. Any worklist item that is *not* verification-neutral on a verified page — a content rewrite, a split or merge, a citation-target or locator change, a claim correction — is not allowlisted: it either requires fact-checking the page against the raw (promote the page into this run's Step 5 scope and verify it before editing) or it demotes the page to `draft` for a later audit. When in doubt whether a fix is claim-neutral, treat it as not — demote, don't re-stamp. The allowlist membership, the lint/audit partition, the text-content exclusion, and the relocation-vs-addition rule are the shared operational spec `.claude/skills/multi-skill/references/verification-neutral-fixes.md`.

   **Grow the check data lists — autonomously, but each addition sub-agent-verified.** Two lint checks are heuristic by construction and carry curated lists audit maintains, in two agent-writable data files under `.claude/skills/lint/`. Both are data, not script — editing them is *not* the "audit does not edit scripts" case (Limits) — and both are loaded as runtime data, so an entry changes a check's behaviour, not its code.

   - `hyphenation-lists.md` drives the bidirectional `hyphenated_open_compound_noun` check with four lists (DISALLOWED open-noun compounds, ALLOWED keep-hyphenated look-alikes, direction-2 HEAD nouns, and the VERIFIED-IGNORE phrases). Audit maintains all four, learning from what it sees: a new slug-derived open compound it confirms → DISALLOWED; a hyphenated proper name or established term a flag would wrongly open → ALLOWED; a genuine head noun → HEADS; a flagged-but-correct use → VERIFIED-IGNORE.
   - `unlinked-mention-ignore.md` is the verified-ignore list for `unlinked_page_mention`, the escape valve for the genuine-vs-generic judgement above. An occurrence audit confirms is generic wording — a homograph, a common noun that happens to be a page title, the term inside a larger established phrase — is appended as one `{page path} :: {target stem} :: {phrase}` line under `## verified-ignore`. Entries are page-scoped and phrase-anchored, so an entry never suppresses the same wording on another page, and rewording the sentence re-flags the mention rather than letting a stale judgement ride.

   **No addition to either file is written on audit's lone judgement.** Before appending any entry, spawn independent sub-agents to verify it — each asked to confirm (or refute) the addition on its own: that a DISALLOWED compound is genuinely open as a noun and hyphenated as a modifier (not a proper name); that an ALLOWED term is genuinely a fixed hyphenation; that a HEAD noun is genuinely a noun; that a hyphenation VERIFIED-IGNORE phrase genuinely reads correctly as written; that an unlinked-mention entry's occurrence genuinely reads as generic wording rather than a reference to that page. Give each sub-agent the candidate plus read access to judge it in context — the flagged occurrence in its own page and the page it might reference — and let it open the relevant rule (`CLAUDE.md` → Wikilink Format's genuine-vs-generic test) itself rather than being handed a curated excerpt that pre-frames the call, or the whole schema file as context: a refuter reads its own ground truth, the same principle the tiered refuter gate (given the claim and read access to the raw, not the page's framing) and `skill-llm-council`'s refuter ("a curated excerpt can pre-frame the check, so read the source") both follow. Add the entry only on sub-agent consensus (default ≥ 2 of 3 confirm); on a split or refutation, do not add it — leave the case for the user, and for an unlinked mention wikilink the occurrence rather than suppressing it. Append in each data file's documented format, and never touch `check_wiki.py` or its test suite — both are edit-only (`CLAUDE.md` → Stay In Your Lane carves out only these two data files as agent-writable). If a DISALLOWED/ALLOWED/HEAD addition genuinely warrants a paired regression case in `.claude/skills/lint/scripts/tests/test_check_wiki.py`, surface it as a proposal for the user rather than writing it (a verified-ignore entry in either file needs none). Both data files are version-controlled, so a wrong addition is recoverable. In a fanned-out run these appends and their sub-agent quorum are deferred to the central post-fan-out pass and spawned from the top-level agent — a fan-out subagent may be unable to nest the quorum, and concurrent appends to one file collide.

   **Resolve `*[unverified]*` markers on verified pages.** When audit fact-checks a claim marked `*[unverified]*` (CLAUDE.md → Bullet Markers): if the claim passes, remove the `*[unverified]*` marker (leave a co-occurring `*[tentative]*` marker in place — that is a separate epistemic judgement that persists) — but clear a marker only when the claim's `#page=N` anchor is already at git HEAD (a marker from a prior, committed run), where the relocation exemption covers the marker-removal line change and the clearing pass is independent of the pass that authored the anchor. Do *not* clear a marker whose anchor is new-versus-HEAD (authored this same, uncommitted run): un-masking it re-fires `verified_anchor_unaudited` (no committed predecessor for the relocation exemption), and clearing-then-keeping-`verified` would be a same-session self-re-stamp — leave it marked this run and let the next post-commit audit clear it (the locator clause under Apply the fixes). If it fails, fix it inline when the raw settles it (a determinate fix, below), or set the page `needs-update` when the failure is beyond audit's reach (per the status rules below). After clearing markers and applying any fixes on a verified page audit fact-checked this run, re-stamp `verified_hash:` with `body_hash.py` (which masks any markers that remain) so the hash again covers the now-larger checked body. A page returns to fully-clean `verified` — no pending delta — when no `*[unverified]*` markers remain and its checked content is faithful to the raw; a verified page that still carries unresolved markers after this run keeps them and stays `verified` with that pending delta visible.

   **Build the fix list.** A page's fix list is the union of audit's own Step 4 findings for it and the lint authored-tier findings carried in from Step 1 — resolve every item, not only the ones audit's pass independently surfaced.

   **Authored worklist items are resolved in this run — deferral is not a disposition for them.** Every authored-tier finding lint carried in is *resolved* now, not logged as a "focused follow-up." Run length, token economy, and the fact that a prior run (or its `hot.md` / `log.md` note) deferred the same backlog are not grounds to defer — a standing authored backlog is skipped work, not an acceptable steady state. **Batch size is not a deferral ground either.** A large run — 100+ `unlinked_page_mention` flags — is resolved this run, not deferred as "substantial" or "risky to rush at the tail of a checkup": the genuine-vs-generic triage is the careful judgement, and once made, applying it is not rushing. Apply every wrap the triage marks genuine — wrapping an already-genuine reference is a verification-neutral allowlisted edit (`CLAUDE.md` → Page Status), so 74 safe wraps are 74 determinate edits, not a risk to hold back — and run the sub-agent quorum for every generic ignore-list entry, batched so a large ignore set stays affordable: each of the (default three) refuters judges the *whole* batch and returns a per-candidate verdict, so the ≥ 2-of-3 quorum still holds per entry in three calls rather than 3×N — never split the batch across refuters, which would silently drop each entry to a single judgement and defeat the quorum. The quorum is the anti-corruption safety, a reason to verify, never a reason to defer; and the Step-7 structural re-validation plus, inside `/checkup`, the confirming lint re-check every link the pass touched, so a large link pass is not unsafe at a checkup's tail. The lone bounded exception: when the quorum volume genuinely exceeds one run's budget, the *ignore-list additions alone* may be staged across follow-up audits (recorded in the report; the un-actioned flags harmlessly re-surface next lint) — but every determinate wrap is applied this run, and the batch is never deferred as a unit or handed back as an `AskUserQuestion`. "Resolve" differs by check, but every disposition is reached this run:
       - `missing_reciprocal_contradiction` and `concept_source_asymmetry` are cleanly determinate (the disagreement / source-link is mutual by nature): author the reciprocal entry or sync the `sources:` link. There is no generic-exemption escape — these must clear, and one still present in the next lint means the audit was non-compliant.
       - `unlinked_page_mention` is resolved by a per-occurrence genuine-vs-generic *judgement*, made now: wrap the occurrence when it is a genuine reference to that page; leave it when it is generic wording or a homograph the check cannot distinguish (`chain` inside `AI Chains`, `chat chain`, or `chain-of-thought` is not the chain-topology concept; `the survey` is not a page). A genuine reference left unwrapped is non-compliant. Record a correctly-judged-generic occurrence in the check's verified-ignore list — the data file `.claude/skills/lint/unlinked-mention-ignore.md`, one `{page path} :: {target stem} :: {phrase}` line per occurrence context under `## verified-ignore` (the grow-the-lists clause below governs the write) — so lint stops flagging it and a later run does not re-litigate it. That is what closes the item: leaving the occurrence alone without recording the judgement is not a disposition, it is a false positive left to recur every run.
       - `stale_mention_ignore` is determinate: a verified-ignore entry that suppresses nothing is dead, and lint has already proven it (the check reports only entries no occurrence fell inside). Delete the entry from `.claude/skills/lint/unlinked-mention-ignore.md` — no sub-agent verification, since removing an inert entry asserts nothing, and the file is version-controlled. The one judgement: if the occurrence still exists but was reworded, re-record it against the *current* phrase (re-confirming it still reads as generic wording) instead of editing the old phrase to match — editing the phrase would carry a judgement onto text nobody re-checked.
       - `intra_page_redundancy` resolves when the content settles which bullet merges or drops; when it is a genuine coin-flip, it falls to the ambiguity clause below.
       The *only* finding audit hands back unresolved is a genuinely ambiguous structural call (the clause below) the fact-check cannot settle — not a determinate fix dressed up as one.

   **A `missing_reciprocal_contradiction` mirror does not demote its `verified` partner.** When the page lacking the reciprocal entry is `verified`, authoring the mirror onto it is not the generic claim-add that demotes (the verification-neutral clause above): the disagreement is already established on the page that names this one, and audit confirms it against the raw(s) this run. So author the reciprocal bullet and confirm the mirrored disagreement against the raw — but mark the authored bullet `*[unverified]*` rather than re-stamping the page. That sourced mirror carries a new `#page=N` anchor, and an anchor *addition* on a `verified` page trips lint's `verified_anchor_unaudited` unless the line is marked (the locator clause below; `check_wiki.py` masks only `*[unverified]*` lines, and CLAUDE.md → Page Status forbids self-re-stamping an anchor change). The page stays `verified` with that bullet as the pending delta — `body_hash.py` masks the marked line, and the next `partial` audit re-checks and clears the marker. The disagreement is already settled on the partner, so this is not a self-stamp of an unchecked guess; the marker is simply the code-sanctioned way to keep the verified partner while the new anchored bullet rides as pending. (If the partner side is itself unverified, the page is in Step 5 fact-check scope and earns its stamp there.)

   **Apply the fixes.**
   - **Determinate fixes** — a finding the raw source or the schema settles outright (a restated number that does not match the raw, a misquoted phrase, a banned-vocabulary word or source-context phrase `lint` detected and left for audit, a non-obvious claim missing its inline citation or its located locator where the fact-check identifies the supporting source and location (write it in one of the two canonical forms, CLAUDE.md → Source Support And Verification, with the structural anchor and printed page inside the `#page=N` deep-link and N computed from the printed→physical offset), a claim whose wording overstates or misinterprets the source, a genuine reference to an existing page left unlinked, the source's actual stated reason supplied for a bare causal/comparative/evaluative claim, or a fabricated mechanism removed from a claim that wrongly attributed it to the source): apply inline. A non-obvious claim that no held source supports is not determinate, and is never dropped on a single pass — route it through the bullet-removal guard below, which re-confirms non-support before any deletion and otherwise marks `*[tentative]*` and sets `needs-update`, per the status rules. Likewise, correct a distortion inline only when the faithful replacement is **bounded** — a single-bullet reword on one page the fact-check settles outright; a distortion whose faithful correction requires re-deriving an aggregate the pass cannot fully validate, spans more than one page, or produces wording audit cannot itself certify faithful this run routes to the distortion-disposition rule below, not to a forced inline rewrite.
   - **Repeated-literal sweep after any citation fix.** Correcting a mis-located or wrong citation on one page usually leaves the identical error on sibling pages — and in `partial` mode some of those are `verified` pages this run is otherwise skipping. After any citation fix, grep the distinctive literal across `1-wiki/` (`grep -rn "{literal}" 1-wiki/` for a figure or percentage; the distinctive phrase for a qualitative claim) and re-check every occurrence: fix a sibling whose cited page you confirm this run; for a sibling on a `verified` page not otherwise in scope, pull it into Step 5's fact-check scope and verify it, or mark the bullet `*[unverified]*` — a mis-located citation is a support failure, never a verification-neutral edit, so it does not silently ride a `verified` page. Record the literal searched and the sites checked and fixed in the report. A single-site correction that leaves an identical error a few bullets or a few pages away is the half-repair this sweep closes.
   - **Authored fixes** — splitting an over-long or multi-idea page, merging near-duplicates, rewriting a source-shaped page, dropping a bullet the raw source does not support (only through the bullet-removal guard below), authoring a missing contradiction bullet, adding a missing cross-paradigm `Connections`/`Not This` link the relationship sweep surfaces (reciprocal on both pages): carry them out grounded in the fact-check, following ingest's existing-source-mode procedure for source-page content (each split-off or rewritten page is a new page: author it with the complete frontmatter template for its page type (`CLAUDE.md` → the page-type frontmatter block — e.g. a concept/entity page carries `type`, `aliases`, `sources`, `source_count`, `tags`, `created`/`updated`, `status: draft`; a synthesis additionally carries `origin` and, when applicable, `single_source_stub`) and all required callouts with block IDs, and let it earn `verified` only from its own fact-check below, never by inheriting a parent's stamp), and the `supersede` procedure's preservation and bookkeeping mechanics — copy the prior version to `2-outputs/supersede/preserve/` (naming the preserved copy per `.claude/skills/multi-skill/references/quarantine-path-convention.md`: a `-N` clash suffix *before* the final extension, then `git check-ignore -q` to confirm the copy did not land on a gitignored name — audit routinely does a split and a merge in one run, so a same-minute clash can otherwise silently lose the only preserved copy), preserve the prior view there, update index/hot/log — for splits, merges, and re-extractions. When a fix removes or renames a page (a merge, a redirect, or a split that retires a slug), also find and rewrite the body-prose inbound wikilinks that point at it, following `.claude/skills/multi-skill/references/inbound-reference-discovery.md` — `check_wiki.py` does not see prose wikilinks, so an inbound link left unrewritten dangles silently until a later lint. Inherit those mechanics but not supersede's approval gates: audit is the autonomy exception and asks the user nothing.
   - **Bullet removal requires three independent confirmations (the three-pass non-support re-read — a separate counter from the three-round rewrite re-check loop below).** Never drop a bullet on a single fact-check. A single pass produces false negatives — it misses supporting text that is present (wrong section, paraphrase mismatch, unchecked appendix) — and a deleted-but-correct bullet is silent data loss the quarantine copy records without preventing. A claim one pass finds unsupported is a removal *candidate*, not a removal: re-confirm non-support with two further independent re-reads of the raw (three in total), each reopening it with a different access strategy — a different section, different search terms, the appendix and any offloaded tables or prompts. Three sequential reads in one context are not fully independent: vary the access strategy genuinely, and treat their agreement as false-negative reduction, not proof of non-support — the quarantine copy makes a wrong deletion recoverable, not prevented, so preserve under any doubt. A pass that cannot find support because the relevant raw region is unreadable — file missing or moved, image-only scan, OCR-garbled, truncated extraction, encrypted or password-protected, or paywalled (the Step 5 coverage-gate conditions) — is void, not a confirming pass: unreadability is not absence, and three passes over one blind spot are one pass, not three. When non-support traces to an unreadable region, never drop the bullet — mark it `*[tentative]*` and set the page `needs-update` with a `needs_update_reason:` naming the unreadable region (the Step 5 coverage failure). Drop the bullet only when all three passes independently agree the raw legibly does not support it (then quarantine it via the authored-fix mechanics above). If any pass finds the support, do not drop it — apply the determinate fix that support enables (add the missing citation and locator) or keep the corrected bullet. If the passes disagree, or support stays uncertain after three, do not drop it — mark the claim `*[tentative]*` and set the page `needs-update` with a `needs_update_reason:` naming the unresolved support question, leaving the bullet visible for the user. This gate governs every bullet-dropping fix — the determinate "no held source supports it" case, the authored "drops a bullet the raw does not support" case, and resolving an `*[unverified]*` claim by deletion alike: preserve a bullet under doubt; deletion is the last resort, never the default.
   - **A confirmed distortion audit cannot safely correct in-pass — the distortion-disposition rule — is set `needs-update`, not force-fixed and not deferred to the user.** A distortion whose faithful correction is *not* bounded to a single-bullet in-pass reword — it requires re-deriving an aggregate the pass cannot fully validate (recomputing a whole table or evidence base — `references/verification-spec.md`, the summary-claim discipline), the same distorted framing is inherited across more than one page (a source page and the concept/synthesis pages that lifted its wording), or audit cannot produce replacement wording it can certify faithful this run — is not force-fixed: a wrong rewrite re-certifies a fresh distortion, the asymmetric error the confirm-before-fixing guard exists to prevent. Nor is it left as a passive finding for the user to authorize. Set every page the distortion touches `needs-update`, with a `needs_update_reason:` carrying the full derivation — the recomputed aggregate, the specific cells, and what the faithful claim must instead say — so the re-ingest or the user receives a precise hand-off, not a re-check. Find the inherited siblings the way the repeated-literal sweep finds inherited citations — grep the distinctive phrase of the distorted framing across `1-wiki/` — and set each `needs-update` too; a distortion corrected on its source page but left standing on the concept page that inherited it is the half-repair this closes. This is the general rule behind every status disposition: a *confirmed* defect always ends the run acted on — fixed when the correction is a safe in-pass determinate edit, set `needs-update` with a derivation when it is not. The one confirmed-defect case handed back as a bare Warning is the genuinely ambiguous *structural* call below, where audit cannot tell whether a defect exists at all — never a distortion audit has confirmed but cannot safely reword.
   - **When a structural call is genuinely ambiguous** — is this one idea stated twice or two distinct ideas? does the merge target page survive or the merged-away one? — and the fact-check does not settle it, do NOT perform the surgery. Downgrade the finding to a Warning that names the options and leave the page for the user. Act autonomously only when the fact-check makes the structural call determinate; preserving the prior version is necessary but not sufficient licence to act on a coin-flip.
   - **Carry the markup on anything you author or move.** Each authored or moved callout carries its block ID as the last line inside the `>` block (`> ^idea`, `> ^evidence`, …; see `CLAUDE.md` → Callout Block IDs). Each page locator you write or move carries its `#page=N` raw-file deep-link, in the form `CLAUDE.md` → Source Support And Verification defines for the page type:
     - **Source page — one form** (`CLAUDE.md` → Source page rules): every source-page locator — `Evidence`/`Method` pointers and in-body *non-obvious claim* citations (TL;DR, Key Claims, any callout) alike — carries the structural anchor and page **together inside** the `#page=N` deep-link display (`[[0-raw/papers/<stem>.pdf#page=N|sec. 4, p. M]]`, `[[0-raw/papers/<stem>.pdf#page=N|fig. 1, p. M]]`), used as a sentence subject (`… indicates …`), a leading `Evidence` label (`[[0-raw/papers/<stem>.pdf#page=3|sec. 3.3, p. 3]]: …`), or a trailing parenthetical. The source key is dropped (it would be a self-link). Do not split the anchor outside the link (`sec. 1, [[…#page=N|p. N]]`) or flatten to a page-only `|p. N` display — both drop the section the schema requires (lint `source_locator_incomplete`).
     - **Concept/entity/synthesis page — the two canonical forms** (Form 1 attributive / Form 2 parenthetical, defined in `references/semantic-checks.md` and `CLAUDE.md` → Source Support And Verification): the source-page wikilink names which source, the parenthesized raw deep-link carries the anchor + page in its display. Form 2 wraps the whole citation in round brackets `( )` — `… claim ([[…|key]]; [[…|sec. X, p. M]])`.
     - **The unpaginated-supplement exemption — on every page type** (`CLAUDE.md` → Source Support And Verification). Where the cited region genuinely carries no printed page number (common in published appendices), `p. M` cannot be stated honestly, so the display cites its structural anchor alone — any anchor, not only `app.`, keyed on the pagination map where the cited physical page prints no number (`CLAUDE.md` → Source Support And Verification): `[[0-raw/papers/<stem>.pdf#page=16|app. D.1, tab. 8]]`. **Never resolve a missing `p. M` by writing the physical page as though it were printed** — that fabricates a number no reader can find on the page, and on a body-paginated paper it lands beside a genuine five-digit printed page drawn from the same source. When audit reopens a raw and finds the cited supplement unpaginated, the fix is to *drop* `p. M`, not to invent one. Migrating an existing `app. …, p. N` locator to the anchor-alone form is a locator change, so it follows the rule above: mark the bullet `*[unverified]*` or demote — audit never self-re-stamps a locator change. Confirm the absence of printed pagination in the raw before applying the exemption, and never use it to dodge a printed page that does exist. Lint exempts it in both `citation_locator_incomplete` and `source_locator_incomplete`.
     - Keep every locator inside a `[[…]]` display or a backticked span (a non-PDF uses a backticked per-type locator in place of the `#page=N` deep-link), with N the physical PDF page — a bare locator left as plain text after the link trips lint's `page_locator_unlinked`.
   - **After a fanned-out audit, run one central locator-format pass before any `verified_hash` re-stamp.** When fact-checking is parallelized (one subagent per raw source), subagents apply the citation format divergently and mislabel sections — a known failure mode this central pass guards against. Centrally re-verify every `#page=N` and section anchor against the raw (never trust a subagent's section label or page number), confirm each locator sits inside its `[[…]]` display or a backticked span, and normalize to `sec.` (not a bare `§`). Run this central pass after the re-fact-check and the structural re-validation (both specified later in this step), immediately before the stamp, and apply all shared-file writes — `index.md`, `log.md`, co-cited pages, and every append to the two agent-writable lint data files (`hyphenation-lists.md`, `unlinked-mention-ignore.md`), with their sub-agent-verification quorum spawned from this top-level agent — centrally here rather than from individual fan-out subagents, so parallel writes never collide and the quorum is never nested inside a fan-out subagent. Only then stamp. (Lint emits the Critical `verified_anchor_unaudited` only when a page that was `verified` *at git HEAD* carries a locator whose anchor was added or relabelled versus HEAD — an anchor *addition* or section *relabel* trips it; a pure *relocation* of an already-present anchor for the same `#page=N` is exempt, and the check does **not** fire on a page audit is promoting draft/needs-update→`verified` this run — the head-status gate, since the promotion is itself the fact-check, so an anchor audit writes onto a page it is promoting needs no marker. So when audit's locator work adds an anchor or relabels which section or page one names on an already-`verified` page, mark the bullet `*[unverified]*` (or demote) as the verification-neutral rule above already requires — never self-re-stamp. Marking is what satisfies "never leave a verified page tripping this Critical": `check_wiki.py` masks a `*[unverified]*` line, so the marked anchor does not fire the check. Leave that marker in place this run — do not clear it here, because the anchor is still new-versus-HEAD (no committed predecessor for the relocation exemption), so clearing un-masks the line and the Critical re-fires, and clearing-then-keeping-`verified` would be the same-session self-re-stamp the schema forbids. The next `partial` audit — after this run's change is committed, so the anchor sits at HEAD — independently re-checks and clears the marker, where the relocation exemption then covers the clean removal.)
   - **Touch `updated:`** on every page whose body audit edits — inline and authored fixes alike. (The only exception is the full-mode no-correction case in Arguments, where a verified page is left unchanged and keeps its `updated:` and `verified_hash:`.)
   - **Strip a stale `verified_hash:`** on any page audit demotes to `draft` or `needs-update` (e.g. a verified page that fails the fact-check in `full` mode) — a non-`verified` page carries no `verified_hash:`; it is re-added only when audit next sets the page `verified`.

   **Re-fact-check anything you rewrote.** A split, merge, source-shaped rewrite, or dropped bullet produces a new body that Step 5 never checked — audit's own edit is a fresh compression step that can drop a subtle distinction the raw made. Before stamping `verified`, re-verify the rewritten body against the raw source. This re-verification is same-agent, so it does not bound a raw-misreading that produced the rewrite in the first place — the three-round loop below adds iterations, not independence. A rewritten non-obvious claim is certified only through the tiered independent-refuter gate like any other (three agreeing refuters for the broad default; `references/verification-spec.md` → the tiered-gate paragraph, `CLAUDE.md` → Page Status) — the same mechanism, not a weaker one, because a rewrite is exactly where a same-agent re-read is most circular. Run the rewritten bullet through its tier's refuters before stamping; on any refutation or split, leave it `*[unverified]*` for the next independent pass rather than self-stamping. `CLAUDE.md` grants audit's autonomy only after a fact-check against the raw, so an unchecked post-rewrite body is outside that licence.

   Three checks beyond raw-faithfulness apply to any rewritten body before stamping. **Semantic re-check:** Step 4's semantic catalogue ran before this authoring existed, so an audit-authored split-off, merge survivor, or rewrite has never had its authored prose seen by any semantic pass — re-run the Step 4 catalogue (`references/semantic-checks.md` — one-idea atomicity, plain language, citation form, the semantic AI-writing tells, reworded intra-page redundancy) over each authored or rewritten body before stamping `verified`; the confirming lint catches only the mechanical tells, not these. **Structural re-validation:** audit's clean-lint precondition (Step 1) held *before* these edits, but audit's own surgery can re-introduce structural drift, and the re-fact-check above only checks faithfulness, not structure. Run `python3 .claude/skills/lint/scripts/check_wiki.py "1-wiki"` over the touched pages (audit already invokes `body_hash.py` from the same scripts dir) and confirm the edit introduced no new Critical/Warning — a split that drops a required callout or block ID, a merge that leaves a dangling inbound link, a malformed frontmatter delimiter. A faithfulness-clean but structurally-broken page must not be stamped `verified`: fix the structural finding, or set `needs-update`. **Merge content conservation:** before stamping a merge survivor, confirm every load-bearing bullet from each merged-away page survives (re-voiced) in the survivor, or was deliberately dropped as redundant. The raw-faithfulness check passes a survivor that is internally accurate yet silently lost a parent's claim the raw under-emphasizes, and the quarantine copy documents that loss without preventing it. Bound this loop: re-verify, fix, and re-check at most three rounds per rewritten page (the three-round rewrite re-check loop — a separate counter from the three-pass non-support re-read above; mirroring ingest's fix-and-rerun cap and lint's regression guard). If the page is still not faithful after three rounds, stop — do not loop and do not stamp `verified`; set the page `needs-update` with a `needs_update_reason:` naming the residual fact-check failure, and record it. **Reaching the three-round cap without convergence is the one point audit may hand back to the user for more rounds:** when the rounds are making genuine progress (each clears findings and re-introduces none) but have not yet converged, audit may surface the residual findings via `AskUserQuestion` and, on approval, run a further bounded batch (three more rounds per approval, re-offering at each cap) rather than settling for `needs-update` — the user opting into further hardening. Absent an interactive user to approve — an unattended run such as `/checkup` — or absent approval, take the safe default: stop and set `needs-update`. Oscillation is different and is never extended: a fix that re-introduces a finding a prior round cleared is cycling, and more rounds cannot fix what undoes itself — stop immediately, set `needs-update`, and record the oscillation.

   **Execution order for a rewritten page, before any stamp:** (1) apply the fixes; (2) re-fact-check the rewritten body against the raw; (3) the three pre-stamp checks — semantic re-check, structural re-validation, merge content conservation — within the three-round cap; (4) the central locator-format pass; (5) set status / stamp.

   **Then set the status.**
   - Set `verified` only when, after the fixes and the re-check, the page is fully faithful to its raw source with confirmed full-text coverage (Step 5 coverage gate) **and every non-obvious claim on it has passed the tiered independent-refuter gate** (Tier 3 — three agreeing refuters — for the broad default; `references/verification-spec.md`, `CLAUDE.md` → Page Status). A claim that did not clear its tier's refuters is not certified: it stays `*[unverified]*` and the page is promoted only on its cleared content, or set `needs-update` if the failure is load-bearing. The same re-stamp applies to an already-`verified` page whose `*[unverified]*` markers audit cleared this run (above): stamp `verified_hash:` in the frontmatter with the body's hash, computed after the fixes with `python3 .claude/skills/lint/scripts/body_hash.py {page}` — `body_hash.py` masks any `*[unverified]*` claims that remain, so the stamp covers only the checked content. The page must have well-formed frontmatter delimited by `---` lines before hashing: `body_hash.py` silently hashes the whole file only when there is no opening `---` (lint then reads a mismatch and demotes the page); an opening `---` with no closing `---` makes it abort with a non-zero exit instead (caught below). Before hashing, confirm the page begins with a `---` line and has a matching closing `---` (the delimiter `body_hash.py` keys on); if the delimiters are missing or malformed, repair them or abort the promotion rather than stamping a hash of the whole file (a valid 64-hex return is not proof the body was isolated). If `body_hash.py` is missing, exits non-zero, or does not return a 64-character hex, abort the promotion for that page, leave it non-`verified`, and report it as a Critical — never write an empty or unverified `verified_hash:`.
   - Set `needs-update` when the page carries an unresolved problem: a residual issue beyond audit's reach (the source page needs a raw re-extraction — a job for ingest's existing-source mode — or the page needs a source the wiki does not hold), a coverage failure (Step 5), a confirmed faithfulness or support defect whose faithful correction audit cannot safely produce in-pass (the distortion-disposition rule above — re-derivation beyond the pass, a multi-page inherited framing, or wording audit cannot certify faithful this run), OR a genuine unresolved cross-source contradiction the fact-check did not resolve — including one audit surfaced or authored this run. Making a contradiction visible is not resolving it (`CLAUDE.md` → Concepts and Entities: an unresolved disagreement sits at `needs-update`). A `needs-update` change must carry an explanation — either a real `Contradictions`/`Tensions` entry on the page, or a one-line `needs_update_reason:` frontmatter field naming exactly what the re-ingest or the user must resolve. A precise hand-off, never a bare label, and never both where one suffices. If the page already carries a `needs_update_reason:`, append the new cause rather than overwriting it (the shared rule in `.claude/skills/multi-skill/references/dependent-cascade.md`), so a prior hand-off survives.

   Never resolve a contradiction by deleting one side — preserve genuine disagreement. Every fix and every status change is recorded in the audit report.

8. **Finalize the report; update index, hot, and log.** First update the draft report's **Status changes applied** section to reflect what Step 7 actually did, reconciling it per page — for each page audit set out to promote or demote, record the actual outcome (promoted / aborted-promotion / downgraded-to-Warning / needs-update) so a partial Step 7 failure cannot leave the report claiming a change that never landed. Reconcile the **Verification proof** section the same way: it is drafted in Step 6 before promotions are decided, so drop any page that ended non-`verified` — a page that failed the post-rewrite re-check and became `needs-update` must not be left certified as promoted. Then, when Step 7 created or removed a page — a split adds a page, a merge removes one, a support-link removal can orphan one — update `1-wiki/index.md` to match: add new pages under the right heading, drop removed ones. When a merge or support-link removal changed any page's `sources:` list, apply the shared source-support bookkeeping in `.claude/skills/multi-skill/references/dependent-cascade.md`: sync `sources:`, recompute `source_count:` as the resulting list length (never a blind ±1), and keep the `Sources` callout in step with `sources:` (audit edits the frontmatter list but must not leave the on-page `Sources` callout stale). Do **not** autonomously set `single_source_stub: true` on a synthesis a removal leaves at one source — that flag records a deliberate user decision to keep a stub (`CLAUDE.md` → Synthesis Pages), which a fact-check cannot establish; set the page `needs-update` with a `needs_update_reason:` naming the lost support (it now needs a second source, deletion, or an explicit user stub decision) instead. Finally update `hot.md` (a Recent-activity entry in the form `- [YYYY-MM-DD HH:MM] verb | …`, newest-first, per `CLAUDE.md` → Hot, Index, And Log) and prepend the log entry. Note that audit's own Step 7 edits leave touched pages in a state the Step 1 clean lint no longer covers — the per-page structural re-validation in Step 7 confirms each touched page, but a fresh full lint is still owed before the next audit run.

```markdown
## [YYYY-MM-DD HH:MM] audit | {N} findings ({C} critical, {W} warning, {I} info)
- Saved: [[2-outputs/audit/audit-YYYY-MM-DD-HHMM.md|audit-YYYY-MM-DD-HHMM]]
- Promoted to verified: [[1-wiki/concepts/self-attention.md|self-attention]] (or "none")
- Set needs-update: [[1-wiki/concepts/positional-encoding.md|positional encoding]] (or "none")
```

## Worked Example

User invokes audit on a wiki of 14 concept/entity pages, 5 source pages, 1 synthesis. The newest lint report is clean except for two pages already marked `needs-update`.

The audit pass produces:

- **Critical (1):** [[1-wiki/concepts/positional-encoding.md|positional encoding]] lists [[1-wiki/sources/Vaswani2017AttentionIA.md|Vaswani2017AttentionIA]] in `sources:` but the source page does not actually support one of its bullets — the claim drifted in from a different source during a prior reingest. Proposed fix: drop the unsupported bullet and the bad sources entry.
- **Warning (2):** [[1-wiki/concepts/scaled-dot-product-attention.md|Scaled Dot-Product Attention]] covers two distinct ideas; recommend splitting into [[1-wiki/concepts/scaled-dot-product-attention.md|Scaled Dot-Product Attention]] and [[1-wiki/concepts/multi-head-attention.md|Multi-Head Attention]]. [[1-wiki/concepts/layer-norm.md|layer norm]] and [[1-wiki/concepts/layer-normalization.md|layer normalization]] are near-duplicate pages for the same idea; recommend merging via `/supersede`.
- **Info (1):** *verified candidate* — [[1-wiki/concepts/residual-connection.md|residual connection]] is supported by three source pages, has no open contradictions, reads in plain language.

Audit acts on the findings autonomously, because it fact-checked each page against its raw source. It re-confirms across three independent re-reads of the raw that the claim is genuinely unsupported — the raw does not back it on any pass — then drops the bullet from [[1-wiki/concepts/positional-encoding.md|positional encoding]], removes the bad `sources:` entry, re-checks the trimmed page, and sets it `verified`. Had any re-read found the support, it would have added the missing citation instead of dropping; had the passes disagreed, it would have marked the bullet `*[tentative]*` and set `needs-update` rather than deleting. It promotes [[1-wiki/concepts/residual-connection.md|residual connection]] to `verified` — its raw-source fact-check passed. It carries out the split of [[1-wiki/concepts/scaled-dot-product-attention.md|Scaled Dot-Product Attention]] and the merge of the two near-duplicate pages [[1-wiki/concepts/layer-norm.md|layer norm]] and [[1-wiki/concepts/layer-normalization.md|layer normalization]], following the `supersede` procedure and quarantining the prior versions. (A page slated for a merge is not also independently promoted — structural retirement governs; positional encoding above is promoted because it is only corrected, not merged away.) The audit report records every finding and every change applied.

A second run, in `partial` mode, exercises the marker machinery the first example omits. [[1-wiki/concepts/self-attention.md|self-attention]] is `verified` but carries one `*[unverified]*` claim whose `#page=N` anchor is already at git HEAD (left by a prior committed run); audit reopens the raw, the claim checks out, so it clears that marker and re-stamps `verified_hash:` — the page returns to fully-clean `verified`. On the same run a `missing_reciprocal_contradiction` finding needs a mirror bullet authored onto [[1-wiki/concepts/layer-normalization.md|layer normalization]], which is also `verified`; audit writes the reciprocal bullet with a new `#page=N` anchor and confirms the disagreement against the raw, but the anchor is new-versus-HEAD, so audit leaves that bullet `*[unverified]*` rather than self-re-stamping — the page stays `verified` with that one pending delta, and the next `partial` audit (after this run commits) re-checks and clears it.

## Edge Cases

- **A note is accurate but too broad.** When the fact-check makes the split boundary clear, audit splits it following the `supersede` procedure rather than leaving one sprawling page in place. When the boundary is genuinely ambiguous, audit downgrades to a Warning and leaves the page for the user instead of guessing (Step 7).
- **A source page is wrong.** Audit sets the source page and the affected concept/entity pages `needs-update`; a full source-page rebuild from the raw is a job for ingest's existing-source mode, so audit flags that rather than re-extracting the raw itself.
- **A claim distorts the source by generalizing from the cell it cites.** A summary or comparison claim ("the numeric half fares worse", "most tasks improve") can be true of the single cell it cites yet false of the whole table — Step 5 recomputes the full set rather than confirming the cited cell (`references/verification-spec.md`, the summary-claim discipline), which is what catches a defect a cell-level check nods through even on a full pass. When confirmed, the faithful correction usually requires re-deriving the aggregate and often spans the source page and the concept/synthesis pages that inherited the framing, so it is rarely a safe in-pass inline fix: set every affected page `needs-update` with a `needs_update_reason:` carrying the recomputed derivation (Step 7's distortion-disposition rule), rather than force-fixing or leaving it for the user to authorize.
- **A page has no resolvable source.** A concept/entity page whose `sources:` is empty (or a synthesis with none) cannot be fact-checked — there is no raw to reopen — so it can never be set `verified`. Set it `needs-update` with a `needs_update_reason:` naming the missing support, or surface removal as a Warning for the user (`forget` territory); audit does not auto-remove a page.
- **A contradiction is productive.** Preserve it; do not treat all contradictions as errors, and never resolve a contradiction by deleting one side.

## Limits

- Audit acts autonomously after a raw-source fact-check — it carries out content fixes (splits, merges, rewrites, support-link changes) following the `supersede` and ingest existing-source-mode procedures and preserving the prior version, then sets status. This autonomy is a deliberate departure from the foundation design's human-commit-point model — see Purpose. In `partial` mode (default) it processes every non-verified page — every `draft` and `needs-update` — plus every `*[unverified]*`-marked claim on an otherwise-`verified` page (the pending delta from edits since the last check — CLAUDE.md → Page Status, Bullet Markers). Only a fully-clean `verified` page — one carrying no `*[unverified]*` markers — is skipped for the *fact-check* in `partial`: its unmarked content was fact-checked already and has not changed since (`verified_hash:` excludes marked claims), so it needs no re-check (Step 7 may still apply an allowlisted verification-neutral worklist fix — a link, a de-hyphenation — to such a page and re-stamp, with no raw re-read; that is not a fact-check). In `full` mode the user has explicitly asked for a re-pass that re-checks every claim, including the unmarked claims on verified pages — see Arguments.
- Do not audit from memory. Audit findings drive autonomous status and content changes, so they must be grounded in the current page content and in the raw source — not in recollection and not in the wiki's own pages alone.
- Never drop a bullet on a single fact-check. Bullet removal requires three independent confirming re-reads of the raw; on any found support or disagreement, preserve the bullet — add the citation the support enables, or mark `*[tentative]*` and set `needs-update` — rather than deleting. Deletion is the last resort (Step 7).
- A confirmed defect always ends the run acted on — fixed when the correction is a safe in-pass determinate edit, set `needs-update` with a derivation when it is not. A distortion audit confirms but cannot safely correct in-pass (a re-derivation the pass cannot fully validate, a framing inherited across pages, or wording audit cannot certify) is never force-fixed with a risky rewrite and never left as a passive finding for the user to authorize — set every affected page `needs-update` with a `needs_update_reason:` carrying the full derivation (the distortion-disposition rule, Step 7). The only confirmed defect handed back as a bare Warning is a genuinely ambiguous structural call audit cannot settle.
- Audit does not edit `CLAUDE.md`, skill files, or scripts. Schema or skill drift is `consistency`'s domain — audit leaves it to that skill's proposals. The exceptions are the two agent-writable data files `.claude/skills/lint/hyphenation-lists.md` and `.claude/skills/lint/unlinked-mention-ignore.md`, which audit grows autonomously under the sub-agent-verification gate in Step 7 — those are data, not script logic (CLAUDE.md → Stay In Your Lane), so they are not skill/script edits. The folder holds a third data file, `.claude/skills/lint/pagination-map.md` (what each physical page of each raw prints), which audit **reads** — it is the fact a `p. M` locator is checked against — but does **not** maintain: an entry is registered when a raw is ingested, after a human eyeballs a rendered footer, not as a side effect of an audit pass. An audit that meets an unregistered raw reports it (`pagination_map_unregistered`) rather than guessing its pagination.
- A `verified` stamp certifies the page body against the raw *as read at verification time*. `verified_hash:` binds the page body, not the raw, so a raw that is silently re-OCR'd, re-curated, or replaced under the same stem after verification is not detected — the page stays `verified` against a source that may no longer say what it said. This rests on the `0-raw/` immutability invariant. When a fact-check does notice the raw diverging from what a page records (offsets shifted, sections renumbered, different text at a cited `#page=N`), set the page `needs-update` naming the suspected raw change rather than re-verifying against it.
- Never modify `0-raw/`.
