# ingest — Planning Questions (Step 3)

The context-post catalogue and the per-decision question types for `ingest` Step 3. Post the context message first, then ask each decision through its own `AskUserQuestion` call — one decision per call, never batched, recommended option first and marked `(Recommended)`. Step 3 is complete only when every decision type with at least one candidate has fired its question(s).

**Context post (chat text, no questions yet):**
- Frame/purpose echo: state the frames or deep purpose back in one line. In existing mode, state whether the reingest is unscoped, reusing the saved frames, appending a new lens, or clearing to unscoped (quote the frame texts).
- Takeaways, scoped to the frames/purpose. New + normal: 3–6 takeaways, each anchored to a section/figure/page locator (a whole-paper digest flattens and invents; a section-anchored one cannot, and the Evidence pointers fall out for free), plus the contribution in one sentence. Deep: as many as the scope earns (typically 5–12 for a paper; defer to the purpose, not a number) plus an evidence map of the figures/tables/equations/sections the purpose will revisit. Existing: a summary of intended source-page changes and of affected dependents, plus existing attachments with proposed disposition. Anything outside the frames/purpose is not listed.
- **Supporting-source attachment sweep** — a positive sweep of Step 1's `index.md` inventory for existing concept/entity pages this source *genuinely supports*: a page whose idea this source provides evidence for, a claim it backs, or a non-obvious point it contributes. This is the cross-source accumulation that is the point of the wiki — a concept page earns its value by gathering support across sources — so it is its own first-class pass, distinct from the relationship-edge sweep below (which draws graph links, not support) and from creating new pages. For each genuinely-supported existing page, propose attaching this source (add it to `sources:` + the `Sources` callout) and writing the 1–3 claim bullets it contributes. Attaching genuine support is not "bloating" — bloat is source-summary detail; support is registering that this source backs the page. Each proposed attachment becomes a gated question 1 below. List the existing pages this source supports but does *not* warrant attaching to (purely topical overlap, no real support) as "no attach" so the decision is visible, not silently skipped.
- **Sibling back-references** (when at least one prior sibling source is in the vault — a sibling source is an already-ingested source that shares a concept/entity page with this one): before drafting, sweep the prior source's concept/entity pages and propose targeted back-reference edits the new source motivates — add a new parent concept to A-side `Connections`, add a sibling mechanism, sharpen A-side `Not This` to disambiguate, add a reciprocal `Contradictions` bullet wherever the new page names an A-side page (reciprocity is schema-required — a one-sided `Contradictions` bullet is a known lint/audit finding). Keep edits small (1–3 bullets); they are relationship-graph plumbing, not source-shape changes. The back references are the half that gets missed at ingest and only surfaces later in audit, so sweep for them deliberately.
- **Contradiction cross-check** (new and existing mode): test this source's key claims against existing source and concept/entity pages on the same topic (use hot/index from Step 1). Where this source genuinely disputes a claim another page makes, surface it — it becomes question 6 below, never a silent divergence. This is distinct from the sibling sweep above (relationship plumbing); here the source's substantive claim disagrees with another page's.
- **Relationship sweep** — for every new or updated concept/entity, run the systematic role-by-role sweep in `.claude/skills/multi-skill/references/relationship-sweep.md` against Step 1's `index.md` inventory. That reference carries the method, the relationship taxonomy, the cross-paradigm emphasis, and the reciprocity and don't-satisfice rules — follow it, do not restate it here. Produce a visible relationship map in this context post — roles checked, relationships found by taxonomy type, and clusters with nothing related — so misses are surfaceable before drafting; each genuine relationship becomes gated question 8, with the reciprocal edit on the other page part of the approved action. ingest runs this source-local sweep over the pages it creates or touches; the exhaustive whole-graph completeness pass is `audit`'s job (its connection-completeness check). This supersedes the old "propose a few weak-tie links" approach and extends the same-cluster sibling sweep above to the whole inventory.
- **Retrieval tags** — a slip-box thinking task surfaced for the human, not silent bookkeeping: only when the page plausibly feeds a future project, reading thread, or talk, propose one, applying CLAUDE.md's retrieval-anchor test ("in what future situation will I want to stumble back onto this?"); default none, never a tag that just echoes the title. Becomes a gated question below (often zero on a given page).
- Possible synthesis candidate only if this source strengthens an existing multi-source topic cluster.
- **Disconfirming evidence (Darwin's golden rule).** List the items the critical read (Step 2) found that cut against the source's own claims, or against a concept as the page would state it — each named in a phrase, tagged with its destination (`Assumptions`, `Appraisal`, or a concept page's `Disconfirming Evidence`). This is surfaced so it is visible before it lands and is not silently softened; the user can veto or sharpen in chat. Where an item is load-bearing enough to flip an `Appraisal` verdict or push a page to `needs-update`, raise it as gated question 10 below instead of writing it silently.
- Partial coverage, if Step 2 flagged it: state what is and is not covered. This becomes gated question 9 below.
- Ambiguities, missing metadata, weak inferences, or contradictions.

**Then the questions (one `AskUserQuestion` call each):**
- **Emphasis (asked before the membership questions 1–10; new + normal):** one `AskUserQuestion` confirming the takeaway ordering — does what the `TL;DR` and `Key Claims` will foreground match what the user needs from this source, or should something else lead? The answer steers Step 5's emphasis. The membership questions below decide *what goes on the page*; this one decides *what leads* — a faithful page can still foreground the wrong thing, and nothing downstream catches mis-prioritization.
1. One per **concept/entity page the source touches** — covering both directions, one `AskUserQuestion` call per candidate:
   - **Create-new:** a reusable idea this source introduces that has no page yet. Name it and give a one-sentence reason it is one reusable idea. Options: include / drop / other. Recommend `Drop` only for a genuinely marginal idea; recommend `Include` when the idea is clearly reusable.
   - **Attach-to-existing** (from the supporting-source attachment sweep above): an existing page this source genuinely supports. Name the page and the support it provides (the claim it backs or the bullet it contributes). Options: attach / no attach / other, with `Attach (Recommended)` first whenever the support is genuine — registering a genuinely-supporting source is the cross-source point of the wiki, not bloat, so the default leans toward attaching, and `no attach` is the exception reserved for purely topical overlap. On attach, add the source to the page's `sources:` and `Sources` callout and write the 1–3 claim bullets it contributes (Step 6), keeping the page atomic — attach the support, do not turn the page into a source summary.
2. One per **image candidate**: state the source locator (e.g. `fig. 2, p. 6`, `tab. 3, p. 9`, `eq. 4, p. 5`), the proposed target page (source page by default, or a specific concept/entity page), the proposed kebab-case-lowercase filename under `1-wiki/attachments/{stem}/` (attachment files follow the wiki kebab-case rule — `CLAUDE.md` → Page Filenames), and the one-sentence reason. Pre-filter to **at most one candidate per concept/entity page** (the single most defining image); never ask the user to choose between two images for the same concept page — route the rest to source-page candidates. Deep mode may propose more figures than normal, but only the figures the depth purpose justifies.
3. **Existing mode adds:** one per source-page change (e.g. "Tighten TL;DR from 3 bullets to 2?"); one per affected dependent (mark `needs-update` / additively touch / split / merge / leave untouched); one per attachment disposition (`keep` / `replace` / `remove` / `add` — `add` against a page that already embeds an image must be a `replace`); and, if Step 1 found a legacy kebab-case source page that should be renamed to match the raw stem, one asking before the rename (it cascades to every inbound wikilink — rewrite them per `.claude/skills/multi-skill/references/inbound-reference-discovery.md`).
4. **(If framed or scope is narrow)** one per **missed-within-frame/scope item** — items in the paper that fit the frames/purpose (their union) but didn't appear in the user's framing. Default drop unless the user says include; phrase as "Include this missed item?" not "Confirm drop."
5. **(If framed or scope is narrow)** one per **frame/scope-adjacent item** — items outside every frame/the purpose that might matter. Default skip (chat-only flag, not page content). Options: skip / append a lens and include / other. Note in the option that appending re-opens the plan under the wider scope (expect more questions, the body widens, and a `verified` page resets to `draft` — see the Step 1 frames rule). If the user appends a lens, re-run Step 3 with the added frame, re-asking every decision whose inputs the wider scope changes — including prior out-of-frame drops the new lens now admits, not only brand-new candidates (the raw is already fully read, so the in-scope content is available to write). Keep all prior approvals; re-ask only the decisions the wider scope newly opens and the prior out-of-frame drops it now admits. A further append during the re-run recurses the same way, with approvals persisting across loops. Termination: each append strictly enlarges the approved set (a fully-read raw has finitely many candidates), so stop when an append admits no new in-scope candidate or prior out-of-frame drop.
6. One per **cross-source contradiction** found in the cross-check above (new or existing mode): name the disputed claim and the existing page that holds the other side. On approval, write reciprocal `Contradictions` bullets on both pages (preserve both sides — never resolve by deleting one). Set the existing page `status: needs-update` with a `needs_update_reason:` only when the dispute is load-bearing for it — it built a claim on the disputed point; a passing mention gets the reciprocal `Contradictions` bullet but no status change, so pages are not ratcheted into `needs-update` faster than `audit` clears them. Default surface, not silently keep both.
7. **(Only when a retrieval anchor plausibly exists) one tag-decision question:** name the proposed retrieval-anchor tag (a hierarchical `project/…`, `thread/…`, or `talk/…` form) and the future situation it answers. Tagging is a human judgement, not bookkeeping — surface it rather than auto-filling or silently leaving `tags: []`. Default no tag; do not ask "what tags?" on every page or propose a content-derived tag that echoes the title. Options: add tag / no tag / other.
8. One per **relationship-sweep candidate** (new/updated concept/entity pages): name the related existing page, the relationship type (cross-paradigm parallel / specialization / part-of / contrast / enables-depends / contradicts / same-as), and the destination. Options: add to `Connections` / add to `Not This` / merge / drop / other. The context-post relationship map lists every candidate across roles and paradigms; this gates each, and the approved action includes the reciprocal edit on the other page. A same-as verdict means update the existing page instead of creating a new one (and add the alias).
9. **(Only if Step 2 flagged partial coverage)** one question: proceed with what is genuinely covered / skip the source / wait for a better copy. Never summarize partial content as if it were the whole source.
10. **(Only if the critical read found a load-bearing disconfirming item)** one `AskUserQuestion` call per load-bearing disconfirming item — include / soften / drop, one decision per call per Conventions, not batched. Fires only when an item would flip an `Appraisal` verdict or push a page to `needs-update`; routine disconfirming notes are surfaced in the context post and written without a gate.
