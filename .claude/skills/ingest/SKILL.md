# ingest

Process one raw source into a source page, then create or update the concept/entity pages it supports. One skill, three composable modes:

- **New vs. existing** — auto-detected by whether `1-wiki/sources/{stem}.md` already exists.
- **Normal vs. deep** — depth of the source page (deep = reference-grade detail for load-bearing sources).
- **Frames (optional)** — one or more short angles scoping the source page.

The modes compose on one shared spine (read → write/update the source page → create/update concept-entity pages → verify). What each mode adds:

| Mode | What it adds to the spine |
|---|---|
| New (default) | Create the source page first-time; propose new concept/entity pages; add index entries. |
| Existing (page exists) | Confirm a reason; needs-update + schema-migration checks; reuse the saved frames, append a new lens, or clear to unscoped; attachment keep/replace/remove/add; cascade `needs-update` to dependents; preserve `created:`. |
| Normal depth (default) | High-level orientation only. |
| Deep (explicit opt-in) | Reference-grade source page — equations, hyperparameters, tables, more figures, deeper Method/Assumptions/Limitations/Appraisal — within a stated purpose. Concept/entity pages stay atomic. |
| Frames (optional) | Scope every source-page section to the frames' union (an upper bound on scope — sections no frame admits stay empty); save each to `frames:`. |

## Purpose

The source page preserves what this source contributes and where the exact evidence lives. Concept/entity pages capture reusable ideas in simple language — do not turn them into source summaries.

**New vs. existing source (auto-detect).** Check whether `1-wiki/sources/{stem}.md` exists.
- **New** → create the source page first-time and propose the concept/entity pages it genuinely supports.
- **Existing** → reingest in place for a stated reason; update the evidence layer without bloating concept/entity pages into source summaries — "bloat" is source-summary detail, not source support, so attaching a genuinely-supporting source and the specific claims it contributes (including to a page the re-read newly reveals it supports) is maintaining the cross-source record, not bloat; preserve the prior view (via git history of the committed page — see the Step 5 clean-working-tree precondition). This mode prevents duplicate source pages. Terminology: "update" = any in-place modification; "refresh" = the specific `schema refresh` reason.

**Depth.** Default normal (high-level orientation); deep is an explicit opt-in that Step 1 asks about, never inferred from the source or the request's tone. Deep mode produces a detailed, reference-grade source page to support writing, high-stakes work, or foundational reference; concept/entity pages stay one-idea and simple regardless. Deep ingest is the longest-lasting page in the wiki and the easiest to bloat, so it **requires a stated depth purpose** — what the depth is *for* (a frame, a paper section the page will support, a specific question, a paper being written, or another concrete future use). "I want a thorough page" is not a purpose; it produces an indiscriminate source dump. Deep expands the *depth* of in-scope sections, never the breadth of out-of-scope ones, and never the concept/entity pages. Throughout, "deep mode" and "deep ingest" name the mode as a noun phrase; `ingest-deep` is the verb token used in log and hot entries.

**Frames are the ceiling, not a starting point.** A frame (also called a lens or angle) is one positive scope on the source page — a short angle like "only the attention-efficiency results" or "the optimizer-convergence proof" that narrows what the page covers; `frames:` may hold several, and the page is scoped to their union. When one or more frames are provided, every section of the source page is scoped to that union. Sections no frame admits stay at the empty placeholder (`> - None noted`). Do not fill from outside the frames even when the source supports it. The Step 8 faithfulness pass judges the page against the frames' union (or deep purpose), not against the whole source — a thin page under a narrow frame is the correct outcome. The raw is still fully read regardless (Step 2). In deep mode, a frame, when given, *is* the depth purpose. A frame names what is *in* scope (a positive lens), not a hard exclusion — union is defined over positive scopes, so appending a lens that admits material an earlier frame set aside simply widens the page rather than "voiding" a rule. An "out of scope" note inside a frame text is reader context for that one lens only. A reingest never edits the saved frame string, and one frame's exclusion never overrides another frame's inclusion. If a later lens admits material an earlier frame set aside, that is the additive model working; note any nuance in the reingest report, not by rewriting the frame.

**A frame may be a pre-applied analysis layer.** When the frame is a categorization the user has already attached to the source (tags, a typology, a labelling scheme), treat those labels as the user's analysis layer, not as facts about the source. Assess each label against what the source actually does — never copy the labels verbatim onto the page as characterizations, and never blanket-reject them. The page stays grounded in the source's own language and mechanisms, not the frame's vocabulary. (Project-specific frame conventions, if any, live in `ingest-memory.md`.)

**Sections are not quotas.** Required callout sections must appear in every page, but they do not all need content. If a section has nothing genuine and source-supported to say, leave the empty placeholder (`> - None noted` for source pages, `> - None yet` for concept/entity pages). A page with several placeholders is correct when the source genuinely earns nothing there. Do not invent, paraphrase, or pad to avoid an empty section. Deep ingest expands the depth of filled sections, not their count.

**Pattern: analyze → plan → validate → execute → verify.** Step 2 reads the source (analyze). Step 3 surfaces the plan via per-decision `AskUserQuestion` calls; user approval on each is the validation gate (plan + validate). Steps 4–7 apply only approved edits (execute). Step 8 runs two independent verification packets before reporting done (verify); verification checks factual accuracy and clarity.

**A deliberate divergence.** The smart-note method this wiki builds on assigns a human to write the note in their own words at a commit point; this wiki instead keeps pages LLM-drafted at `status: draft`, with the Step 8 verification packets (factual accuracy, clarity, AI-voice scan) plus a later `audit` standing in for that human re-voicing. The packets are therefore load-bearing, not optional.

## Conventions (Recap)

- **Source-page filename matches the raw source stem exactly** (drop the extension, preserve case and internal punctuation): `0-raw/papers/<bibkey>.pdf` → `1-wiki/sources/<bibkey>.md`; `0-raw/articles/<slug>.md` → `1-wiki/sources/<slug>.md`. The attachments folder uses the same stem: `1-wiki/attachments/{stem}/`.
- **Every wikilink is path-qualified from the repo root, includes `.md`, and uses a pipe-rendered display name** that reads naturally: `[[1-wiki/concepts/scaled-dot-product-attention.md|Scaled Dot-Product Attention]]`, authors `[[1-wiki/entities/ashish-vaswani.md|Ashish Vaswani]]`. The pipe display is required (a bare path renders unreadably) and carries no whitespace around the pipe (`[[path|display]]`, never `[[path | display]]`); attachment embeds `![[...]]` take the path but no pipe.
- **No bold and no italic in wiki content** (source, concept/entity, synthesis, hot/index/log). The only structural exceptions are LaTeX math and the bullet markers `*\[unverified\]*` / `*\[tentative\]*` / `*\[disputed — see Contradictions\]*`. Inline backticks for paths, code, and quoted prompt addenda are fine.
- **Use `AskUserQuestion` for every user-facing decision** in Step 3. One decision per question; do not batch. Mark the option you recommend `(Recommended)` and place it first — base the lean on your read of the source (e.g. `Include (Recommended)` for a clearly-warranted concept page, `Drop (Recommended)` for a marginal candidate). Omit a recommendation only on a genuine no-lean call, and say so in the question. Phrase every question and option label for a reader who knows nothing about the wiki's internal vocabulary — gloss any callout, status, or frames term inside the option (present `add to Not This` as "record as a contrast (Not This)", `mark needs-update` as "flag for re-checking (needs-update)", `clear to unscoped` as "widen back to the whole source"), the same way the Step 1 depth question already describes both modes plainly.
- **Structural authority lives in `CLAUDE.md`.** The required-section lists, frontmatter blocks, block-ID placement, and locator rules are the shared wiki schema; this skill cites `CLAUDE.md` for them rather than copying them (copying would drift). The Step 8 verification spec lives in the shared `.claude/skills/multi-skill/references/verification.md` — a single canonical copy in the cross-skill location because `query` runs the same packets when it promotes a page; there is no `CLAUDE.md` mirror to keep aligned. The extraction and citation *procedures* the agent runs (page-offset formula, crop loop, image rename) are self-contained in the skill's own steps and `references/`.

## When To Invoke

Use when the user names a specific raw file under `0-raw/` and asks to ingest, process, capture, digest, import, deep-ingest, reingest, or refresh it. Implicit phrasings count too — "add this paper to the wiki" (or knowledge base / vault), "make a source page for X", "write up this PDF", "summarize this PDF into the wiki", "take notes on this paper". (Absent any destination signal, "take notes on this paper" defaults to ingest — the wiki. Route elsewhere only when the user names an external structured database as the destination and the project actually defines a capture skill for it; see When Not To Invoke.) The user may optionally pass one or more frame texts to scope the summary; without one, default behaviour applies. Mode is auto-detected (Step 1), so the user need not say which.

## When Not To Invoke

- The user asks a research question. Use `query`.
- The user wants to remove existing wiki content. Use `forget`.
- The user wants to replace content while preserving the old view (a swap/merge/split decided in advance). Use `supersede`.
- The user wants to re-verify an existing page against its raw source without rewriting it. Use `audit`. (In ingest, "refresh" means re-reading the raw and rewriting the page in place — a reingest — not re-verification.)
- The user wants a source captured into an external structured database (a project-specific field set) rather than the wiki. If the project defines a capture skill for that destination, use it. `ingest` instead builds **wiki** pages — a full source page plus the atomic, reusable concept/entity pages it supports — all source-grounded and interlinked. (The difference is destination and form, not depth.)

## Procedure

```text
Ingest Progress:
- [ ] Step 1: Load orientation; detect mode (new/existing, depth); confirm reason/purpose and frame
- [ ] Step 2: Read the source (confirm full-text coverage)
- [ ] Step 3: Surface the plan via per-decision AskUserQuestion calls
- [ ] Step 4: Extract or prepare approved attachments
- [ ] Step 5: Write or update the source page
- [ ] Step 6: Draft or update concept/entity (and synthesis) page changes
- [ ] Step 7: Apply approved edits; cascade needs-update (existing mode)
- [ ] Step 8: Run ingest verification (see .claude/skills/multi-skill/references/verification.md)
- [ ] Step 9: Compile the Recommended next ingests report section
- [ ] Step 10: Update hot.md
- [ ] Step 11: Prepend log.md
```

1. **Load orientation; detect mode; confirm reason/purpose and frame.** Read `1-wiki/hot.md`, `1-wiki/index.md`, `.claude/skills/ingest/ingest-memory.md`, and `.claude/skills/multi-skill/multi-skill-memory.md` if not already loaded this session. Use hot/index to avoid duplicate concept/entity pages and to identify related notes and near-duplicates. Apply prior memory corrections about scope, framing, and source-supportedness before drafting.

   Then settle the mode:
   - **Resolve the stem first.** If the user did not give the exact raw filename, resolve `{stem}` from `0-raw/` and confirm it before the existence check (see Edge Cases) — the confirmed stem drives new/existing detection.
   - **New vs. existing:** does `1-wiki/sources/{stem}.md` exist? If yes → existing-source (reingest) mode. If no → new-source mode. Before concluding "new," also rule out a legacy page that maps to this raw under a different name: grep `1-wiki/sources/` for any page whose `file:` frontmatter points at this raw file, and scan the directory for case- or kebab-variants of the stem — anchor the variant match on the stem's extension or wikilink-closing delimiter (`\.md`/`\.pdf` followed by `]`, `|`, or `#`, mirroring `supersede`; that delimiter anchors the `file:`-frontmatter grep, whose target is wikilink-wrapped, while the bare-filename directory scan carries no `]`/`|`/`#` and is anchored instead on the extension boundary or end of name, `\.md$`) so a prefix-of-a-longer-stem neighbour does not mis-match (a search for `Devlin2019BERTPO`, say, must not also catch a different, longer stem that merely begins with it), and eyeball each hit to confirm it maps to this exact raw. A match is existing mode and a rename candidate for Step 3, not new — skipping this check silently creates a duplicate source page and abandons the prior view, the hardest ingest error to reverse. Retain the grep run and its hits (or "no match") to record in the Step 8 report; new-source mode is not entered until this check is run and its result retained — an unretained check is treated as not-run (mirroring Step 8's proof-of-re-read gate). When `1-wiki/sources/{stem}.md` does exist, before concluding reingest, confirm its `file:` frontmatter resolves to the exact raw the user named: the stem drops both extension and subdirectory (Conventions), so `0-raw/papers/{stem}.pdf` and `0-raw/articles/{stem}.md` both map to `1-wiki/sources/{stem}.md`, and if the existing page's `file:` points at a different raw this is a stem collision, not a reingest — surface it via `AskUserQuestion` (same stem, different sources) rather than silently re-reading the other raw's `file:` in Step 2.
   - **Depth:** normal unless the user opts into deep. **Do not infer deep from the source or the request's tone — ask.** If the user's request did not already specify the depth, ask once with `AskUserQuestion`, describing both modes for a reader who knows nothing about them. Recommended wording: `"Normal or deep ingest? Normal gives a high-level orientation — the key claims and where the evidence lives — and stays compact. Deep produces a reference-grade source page: equations, hyperparameters, full result tables, and careful figure/page references, so it can support writing or serve as a long-term reference. Deep takes longer and the page is much larger, and you must say what the depth is for. Concept/entity pages stay short and atomic either way."` Options: `Normal` / `Deep`. Honour an explicit request ("deep ingest", "reference-grade") without re-asking.
     - **Deep requires a stated depth purpose** — what the depth is for (a frame, a paper section the page will support, a specific question, a paper being written). A frame is the purpose only when it is a genuine scope — a section, a question, or a concrete future use; echo it back. A bare topic word (e.g. "transformers") is not a scope — treat it like "thorough" and ask for the non-frame purpose. Without a frame, ask for the non-frame purpose before reading. Do not accept "thorough" or "long-term reference." If the user can't give a purpose after one focused ask, fall back to normal and proceed. Capture it: a frame goes in the `frames:` list at write time; a non-frame purpose goes in the report's `purpose:` field (not on the page).
     - **Existing-source mode never auto-instigates deep.** A reingest is often a re-framing, not a depth change — settle the reason and frame first, and switch to deep only if the user explicitly opts in via the same ask. The `fuller write-up` reason means more thorough writing within the existing scope, not the reference-grade `ingest-deep` treatment.
   - **Existing-source mode only — confirm reason, then run the always-on checks.** "Dependents" below means every page that lists this source in its `sources:` frontmatter — the concept/entity and synthesis pages that lean on it.
     - **Reason.** Confirm one of: `schema refresh` (to the current source-page template), `missed evidence`, `fuller write-up` (more thorough writing within the existing scope), `contradiction` surfaced by a newer source, `corrected interpretation`, or a concept/entity page that got too source-shaped and needs cleanup. The reason sets what changes on the page — schema-refresh and missed-evidence are high-level; fuller-write-up writes more; contradiction and corrected-interpretation are targeted and scope-limited. It does not change how much is read: every reingest fully re-reads the raw (Step 2). Nor does it by itself turn on the reference-grade deep mode (a separate explicit opt-in; see Depth). **No reason given → ask; reingest without a reason creates drift.**
     - **needs-update check.** If the source page is `status: needs-update`, read its `needs_update_reason:` and any `Contradictions`/`Tensions` entries — that is the precise statement of what an earlier `audit` flagged. Fold resolving it into scope; clear the status back to `draft` once addressed (a later `audit` re-verifies). Same for any `needs-update` dependent the reingest repairs.
     - **Schema-migration check.** Compare the existing source page's section list against the current required callouts in `CLAUDE.md`; record any missing ones (Step 5 adds them). Run the same check against the concept/entity and synthesis pages this source supports.
     - **Frames check.** Read the existing page's `frames:` frontmatter. State the chosen outcome and its consequence back to the user:
       - **reuse the saved frames** (default): scope unchanged — existing bullets stay, evidence is refreshed, breadth does not move.
       - **append a new lens**: existing bullets stay and the new lens's content is written this pass (the full re-read makes it available).
       - **clear to unscoped** (`frames: []`): existing bullets stay and the page is rewritten to whole-source breadth.
       **Scope-widening test:** a change widens scope only when the new `frames:` union admits source material the old union did not (a strict superset). A reword that admits the same material, or appending a lens the body already covers, is not widening — treat it as reuse (no forced body growth, no status change). A genuine widening (an append or clear that admits new material) is substantive. It grows the body to match this pass and resets the page to `status: draft`; a `verified` page resets too, and a later `audit` re-verifies it. A genuine widening also overrides a purely targeted reason: you cannot widen `frames:` while writing only a few bullets. A `frames:` value (including `frames: []`) that claims more scope than the body covers fails the Step 8 faithfulness packet. Reframing only ever adds, then widens the body to match; narrowing (dropping a lens while keeping its content) is not a reingest operation — use `forget` or `supersede`.
     - **Depth-purpose check.** A frame is recovered from frontmatter above; a non-frame depth purpose from a prior deep ingest is not on the page — it is in the `purpose:` field of the latest `2-outputs/ingest/ingest-*-{stem}.md` report. Only when the Depth ask above returned Deep, recover the prior purpose from there and offer to keep, replace, or drop it for this reingest. If no report is found or its `purpose:` is empty — the record was pruned, or a later normal reingest overwrote the latest report with an empty purpose — do not proceed on an unknown purpose: ask for a fresh deep purpose per the Depth rule above.
     - Locate every dependent (defined above) so Step 7's cascade can reach it.

2. **Read the source in full (confirm full-text coverage).** Read the raw file from `0-raw/` (use the existing page's `file:` frontmatter to find it in existing mode). **Every ingest — new, existing, normal, or deep — fully reads the raw, cover to cover. There is no targeted partial re-read; the reason, frames, and depth control what gets written, never how much gets read.** Deep mode reads the whole raw and additionally goes deeper on the equations/figures/tables the depth purpose needs — out-of-scope sections are still read, just written only as the scope admits. For long books or surveys, propose chapter-level source pages instead of one oversized note (the unit read is then the chapter, still read in full). If frames are in effect, read through them as well, keeping both the framed and a general read in mind. **Read critically as well as comprehendingly (Darwin's golden rule):** on sight, capture what cuts against the source's own claims, the assumptions it leaves unstated, and the strongest critique a sceptical reviewer would write — disconfirming evidence escapes attention unless caught as you read. When a frame is active, also ask what in the source cuts against the frame's thesis; a positive lens must not become confirmation-seeking. These feed `Assumptions`, `Appraisal`, and concept-page `Disconfirming Evidence` at write time, and `Contradictions` or the missed-within-frame bucket where they bear on scope — not the empty placeholder by default. **Confirm full-text coverage before drafting.** Verify the raw was fully readable by a positive probe, not a vibe: the final page or section is present and intelligible (catches truncation), and body text — not only figure captions — is extractable (catches an image-only scan or abstract-only paywall). Treat coverage as partial if either probe fails (truncated, image-only, OCR-garbled, paywalled to an abstract, or only partially extracted). For a genuinely image- or media-based raw (a slide deck, an infographic, a video still), body-text extraction is not the probe — confirm instead that the visual content the page draws on (every slide, frame, or figure) is present and legible; absent body text is expected there, not a coverage failure (mirrors `verification.md`, so Step 2 and Step 8 agree). If coverage is partial, surface it in the planning step and draft only what is genuinely covered; never summarize partial content as if it were the whole source. If the source's relevance is unclear after reading, see Edge Cases before drafting.

   **Register the raw's pagination before any locator is written (PDF raws).** A locator states two things about a page — where it sits in the file (`#page=N`, the physical page) and what the page prints (`p. M`). The second is a fact about the raw, not derivable by rule: an appendix may restart its numbering, a proceedings build may start at a high page number, and a whole body may print no number at all. Getting it wrong yields a confidently-wrong citation that renders plausibly and passes every structural check. So it is recorded once per raw, before it is needed, in `.claude/skills/lint/pagination-map.md`:

   ```bash
   python3 .claude/skills/lint/scripts/pagination_map.py 0-raw/papers/{stem}.pdf                    # propose a map from the footers
   python3 .claude/skills/lint/scripts/pagination_map.py --verify 0-raw/papers/{stem}.pdf {outdir}  # render footer crops to eyeball
   ```

   **The generator proposes; you confirm.** Eyeball every proposed line against a rendered footer crop before writing the section — a wrong `none` would license stripping a correct printed page from a citation and certifying the damage. Then add the `## 0-raw/papers/{stem}.pdf` section to the data file. An unregistered raw still ingests (the locator-completeness checks fall back to the older `app.`-anchor heuristic) but raises `pagination_map_unregistered`, and `locator_page_mismatch` — the check that catches a `p. M` contradicting what the page prints — cannot run on it. In existing-source mode, check the raw is already registered and register it if not.

3. **Surface the plan via per-decision `AskUserQuestion` calls.** Post a single chat context message first (so the user has the picture), then ask each decision through its own `AskUserQuestion` call — one decision per call, never batched, options framed as include / drop / other (or yes / no / other), with the option you recommend marked `(Recommended)` and placed first, so the user can override your lean. The numbered items below are decision TYPES, not a fixed question count: types 1, 2, 6, and 8 fire once per candidate, so a dense paper expands to 10–20 separate `AskUserQuestion` calls. The context post lists the full slate up front so the sequence is not a surprise, and the user can pre-empt it in chat ("just the obvious concepts, skip the images") before the questions begin. Do not collapse the per-candidate questions into one summary question to save calls — one decision per call is binding; the only sanctioned shortcut is the user pre-approving or pre-skipping whole classes in chat before the questions begin. Step 3 is complete only when every decision type that has at least one candidate has fired its question(s) — including the contradiction (type 6) and relationship-sweep (type 8) questions, which are easy to skip once the candidate-concept questions feel done; do not advance to Step 4 with mapped-but-unasked relationships or contradictions outstanding.

   The context-post catalogue (frame echo, takeaways, sibling back-references, contradiction cross-check, relationship sweep, retrieval tags, disconfirming evidence) and the per-decision question types (the emphasis question, then membership questions 1–10) are in `references/planning-questions.md` — work the plan through it, one `AskUserQuestion` call per decision.
   Only approved items become wiki content. Do not act on partial answers. If the user pauses mid-sequence, wait; on resume, keep the answers already given and continue from the next unanswered decision (do not re-ask answered ones or discard them).

4. **Extract or prepare approved attachments.** Write each approved image to `1-wiki/attachments/{stem}/{name}.png` (create the folder if needed). Tools:
   - PDFs: `pdftoppm -png -r 300 -singlefile -f {page} -l {page} "{raw}" {outprefix}` renders one page to exactly `{outprefix}.png` (300 DPI per `CLAUDE.md` → Attachments; `-singlefile` avoids the zero-padded page suffix plain `pdftoppm` adds on a 100+-page PDF, which would misname the file). `pdfimages -png -f {page} -l {page} "{raw}" {outprefix}` pulls an embedded image directly, but writes indexed names (`{outprefix}-000.png`) and may emit `.ppm`/`.pbm` for non-RGB images — rename (and convert with `magick in.ppm out.png` if needed) to the approved `{name}.png` before embedding, or the Step 5 embed dangles. To crop a region tightly to the figure plus its full caption — or to clip it directly by PDF coordinate with PyMuPDF (the most reliable path, no DPI-pixel arithmetic) — see `references/attachments.md` (the point-space clip, the render→inspect→crop loop, and a worked `magick -crop` example).
   - Non-PDF media already in `0-raw/media/`: copy or symlink into `1-wiki/attachments/{stem}/`. If the copy fails (missing or unreadable file), report it and ask rather than leaving a dangling embed.
   - **Existing mode:** these mutations run only after the Step 5 clean-working-tree gate and any approved legacy rename, since a `replace`/`remove` overwrites or moves a committed attachment and an ungated mutation loses an uncommitted prior view. `add`/`replace` write as above; when a `replace` changes the image content (a corrected figure or table, not a tighter crop or higher DPI), treat every page embedding that path as a dependent — grep the embed path `1-wiki/attachments/{stem}/{name}` across all of `1-wiki/` (not only this source's `sources:` dependents, since a path-qualified embed can be reused by a page under a different stem) and, on a `verified` embedding page whose non-obvious claim the image backs, mark that claim `*[unverified]*` because the body text is byte-identical, so `verified_hash` does not move and lint cannot catch a missing mark (mandatory, not discretionary; mirrors `supersede`, `dependent-cascade.md`). `remove` copies the file to `2-outputs/quarantined/attachments/{stem}/{name}.png`, then — per `.claude/skills/multi-skill/references/quarantine-path-convention.md` — verifies the copy byte-identical (`cmp -s`) and not git-ignored (`git check-ignore -q` returns non-zero) before unlinking the original (a binary copy has no diff to eyeball, so a truncated or git-ignored move silently loses the only preserved copy; `CLAUDE.md` Safety), never delete outright — the prior view is preserved. On a name clash with an already-quarantined file in that folder, apply the shared name-clash suffix rule (`-N` before the extension, never after; the `.gitignore` trap) in `.claude/skills/multi-skill/references/quarantine-path-convention.md`, so an already-quarantined file is never silently overwritten.
   - Never write into `0-raw/`. If extraction fails (encrypted PDF, vector-only figure), report it instead of silently skipping, and put the fallback to the user as a fresh `AskUserQuestion` (the Step 3 approval assumed a clean extraction that did not materialize): drop the embed, accept a locator-only reference, try a different render, or place a manual screenshot. A dropped embed must also be removed from the Step 5 write plan and never written to `attachments:` or embedded in the body, or Step 8 catches it as a dangling embed.

5. **Write or update the source page** at `1-wiki/sources/{stem}.md`. Use the source-page frontmatter and required callouts from `CLAUDE.md`. Every callout carries a block ID — the kebab-case of its display title — as the last line inside the `>` block (`> ^tldr`, `> ^evidence`, …; no blank `>` line before it) — see `CLAUDE.md` → Callout Block IDs; include it on every callout, including empty placeholders and any callouts inserted by the schema-migration check. For PDF raws, every page locator carries a `#page=N` deep-link to the raw file, in `Evidence`/`Method` and in any locator quoted in `Key Claims`/`TL;DR`. The structural anchor and the page go **together inside** the link display: `[[0-raw/papers/{stem}.pdf#page=5|sec. 3, p. 5]], …` (never split the anchor outside the link or leave a page-only display — lint `source_locator_incomplete`). `abstract` is a valid front-matter anchor — cite abstract-drawn content `abstract, p. 1`, never relabelled `sec. 1` (on most papers sec. 1 is not on the abstract's page). Non-PDF raws have no physical-page anchor — use the per-type locator formats from `CLAUDE.md` → Source Pages → Evidence pointer formats (articles: `sec. Heading`, `para. N`; media: `slide N`, a timestamp like `00:03:12`, or `fig. N`; other: a stable row, section, label, or path) and link the raw file without a `#page=N` anchor (`[[0-raw/articles/{stem}.md|{stem}]]`). N is the physical PDF page (the Nth page of the file), which differs from the printed page number whenever pagination does not start at 1 (proceedings, article pagination). Compute the offset by opening the PDF: read the printed number on its first physical page, then `N = printed − (first_printed − 1)`. Do not infer the offset from the cited range; it need not start at the paper's first page. **Read `p. M` off the pagination map registered in Step 2, not by re-deriving an offset** — a single offset does not describe a raw whose appendix restarts its numbering, and where the map says a cited page prints nothing, drop `p. M` and cite the structural anchor alone (`[[0-raw/papers/{stem}.pdf#page=16|app. D.1, tab. 8]]`). `lint`'s `locator_page_mismatch` checks every `p. M` against the map. Worked examples and the figure-crop loop are in `references/attachments.md`.
   - **New:** draft the full page with `status: draft` (a later `audit` promotes it). `file:` points at the raw path-qualified into `0-raw/` (`file: "[[0-raw/papers/<bibkey>.pdf]]"`); authors use pipe-rendered display, auto-linking any author already in `1-wiki/entities/` and leaving a one-off collaborator with no page as a dangling path-qualified `authors:` wikilink (a new author page is proposed only via a Step 3 create-new question, and only for a prominent author — `CLAUDE.md` → Author page creation policy).
   - **Existing:** before the first overwrite, confirm a clean working tree across every page this run will write — the source page, every dependent the Step 7 cascade will touch (located in Step 1), and, on a legacy rename, the legacy page and its attachments folder: `git status --short 1-wiki/sources/{stem}.md` plus the legacy path and each dependent path. If any is dirty, surface via `AskUserQuestion` (proceed / commit-or-stash first / abort) — the prior view is preserved by git history of the committed page only (ingest does not snapshot to `2-outputs/`), so an uncommitted page's prior edits would be lost silently, including on a cascade-written dependent (`CLAUDE.md` Safety: never overwrite silently). If Step 3 approved a legacy-page rename, then — working tree confirmed clean — `git mv` the legacy page `1-wiki/sources/{legacy}.md` → `1-wiki/sources/{stem}.md` and its attachments folder `1-wiki/attachments/{legacy}/` → `1-wiki/attachments/{stem}/`, and rewrite every inbound reference per `.claude/skills/multi-skill/references/inbound-reference-discovery.md`, before any write — otherwise the write lands a fresh `{stem}.md` beside the untouched legacy page, re-creating the duplicate the Step 1 guard exists to prevent. Then preserve `type:`, `file:`, and `created:`; touch `updated:`. Apply the Step 1 schema-migration check — insert each missing required callout in its correct position from `CLAUDE.md`, filling from the raw where the scope makes that natural, otherwise the empty placeholder. Sync `attachments:` with the current files (drop entries for quarantined files). Keep `Concepts and Entities` synchronized. Fold any stale Figures callout's embeds into `Evidence`/`Method` and drop it.
   - **Frames frontmatter:** record each frame as an item in the `frames:` list; in existing mode apply the Step 1 frames decision (reuse / append / clear, with its scope-widening consequence — a genuine widening grows the body and resets `status: draft`). `frames:` is always present — write `frames: []` when unscoped, never omit it — and appears in frontmatter only, never in the page body.
   - **Sections are not quotas.** Fill only with genuine, source-grounded content within the frames/purpose; leave `> - None noted` otherwise. Per-callout writing guidance (what genuinely belongs in each of `TL;DR`, `Contribution`, `Key Claims`, `Evidence`, `Method`, `Assumptions`, `Limitations`, `Appraisal`, `Concepts and Entities`, `Contradictions`, `Open Questions`, `Connections`), the inline-embed format, and the citation / `attachments:` / no-bold rules are in `references/source-page-writing.md`.

6. **Draft or update concept/entity (and synthesis) page changes.**
   - For each approved concept/entity page, apply the one-idea test: one idea or named thing? explained simply? reusable for later writing? reads standalone with its source page closed? supported by at least one source page? connects to existing notes?
   - Existing page: a small edit (usually 1–3 bullets or a tighter `Idea`). New page: the full concept/entity template from `CLAUDE.md`, written `status: draft` with `aliases:` populated when the source gives the concept a genuine acronym, alternate name, or older term (e.g. `aliases: [Adam]` on an Adaptive Moment Estimation page — `CLAUDE.md` → Concepts and Entities) and on a same-as merge, otherwise `aliases: []`, each callout ending in its block ID — the kebab-case of its display title, so `[!why]` Why It Matters takes `> ^why-it-matters` (`> ^idea`, `> ^why-it-matters`, …; see `CLAUDE.md` → Callout Block IDs). Only fill sections with genuine content (sections are not quotas) — a new page often earns only `Idea`, `Why It Matters`, `Sources`; the rest stay `> - None yet`.
   - At most one defining image per page, embedded in `Idea`, illustrating the idea (not summarizing the source); never in `Examples`. Deep mode: do not push equations, metric tables, or long method detail into concept/entity pages unless the page is explicitly about that idea.
   - Add the source to `sources:` (`- "[[1-wiki/sources/{stem}.md|{stem}]]"`) and the `Sources` callout. Cite each non-obvious factual claim inline to the specific source it draws from (`[[1-wiki/sources/{stem}.md|{stem}]]`, appended where it reads naturally); leave obvious/definitional bullets and own-voice judgement (`Disconfirming Evidence`, `Open Questions`) uncited — not citation wallpaper. Each citation pairs the source-page wikilink (which source) with a located raw-file deep-link (where in it), in one of two canonical forms chosen by sentence grammar — `CLAUDE.md` → Source Support And Verification is the authority for both. Form 1, attributive (source is the subject): `[[1-wiki/sources/{stem}.md|{stem}]] ([[0-raw/papers/{stem}.pdf#page=N|sec. 3.2, p. 5]]) shows …`. Form 2, parenthetical (state the claim, then cite): `… claim ([[1-wiki/sources/{stem}.md|{stem}]]; [[0-raw/papers/{stem}.pdf#page=N|sec. 3.2, p. 5]]).` (the whole citation is wrapped in round brackets `(` `)`, and a semicolon separates key from locator). For a PDF the deep-link display always carries both a structural anchor (`sec.`/`app.`/`ch.`/`fig.`/`tab.`/`eq.`/`abstract`) and the printed page `p. M`, with `N` the physical PDF page; for a non-PDF, swap the deep-link for a backticked per-type locator (`(\`sec. Heading\`)` in Form 1, `({stem}; \`sec. Heading\`)` in Form 2). A bare source link with no locator is an incomplete citation on a non-obvious claim. Never fabricate the locator to satisfy this: if you cannot confirm the exact page or section in the raw, do not invent one — mark the claim `*[unverified]*` or `*[tentative]*`, or quote-and-locate only what you can confirm; a plausible but wrong `#page=N` or section is a silent error worse than an honest gap (`CLAUDE.md` → Source Support And Verification, Citation principles). So the claim is verifiable at the exact spot (mirrors Step 5; see `CLAUDE.md` → Source Support And Verification). When a page draws on several sources, the inline citation is what tells the reader which source backs which claim. On a bullet naming two or more sources — the cross-source callouts especially (`Contradictions`, `Not This`, `Disconfirming Evidence`, and synthesis `Tensions`) — cite each clause to the source that states it (the claim's origin), not to the source the bullet is framed about; a contrast sentence has two origins and carries two locators, and pinning a true claim from source A to a locator in source B reads correctly and passes every structural check, so only opening the cited page catches it (`CLAUDE.md` → Source Support And Verification, the multi-source-bullet rule). Word each cited claim to faithfully represent its source — no overstatement or misinterpretation (Step 8 re-checks this claim by claim). A causal, comparative, or evaluative claim carries its reason or mechanism (from the source, faithfully), not just the conclusion — never fabricate a mechanism; recast as own-voice reasoning or flag `*[tentative]*` if the source gives none. Also wikilink every genuine reference to an existing wiki page (concept/entity/synthesis/source), on every occurrence, not only first use; a page never links to itself. See `CLAUDE.md` → Source Support And Verification, Plain-Language Style, and Wikilink Format.
   - Synthesis pages (existing mode): keep the cross-source answer, evidence map, tensions, and source list; update embeds the same way. ingest only updates an existing synthesis whose source list this source already belongs to; it never creates a synthesis page — that is `synthesis`'s job. A source that strengthens a new multi-source cluster is surfaced as a synthesis candidate in Step 3, not authored here.

7. **Apply approved edits; cascade (existing mode).** Mechanical fields need no separate approval: `created:` (new pages), `updated:` (touched pages), `source_count:` to match `sources:`, `attachments:` to match files, `Concepts and Entities` callout updates, and `1-wiki/index.md` entries for new/renamed pages.
   - **Existing mode — cascade `needs-update`.** Walk every dependent (located in Step 1) and apply the shared dependent-cascade mechanics in `.claude/skills/multi-skill/references/dependent-cascade.md` — what a dependent is, the `needs_update_reason:` hand-off and append-don't-overwrite rule, `source_count`/`Sources` sync, the `verified_hash` strip on demotion of a `verified` dependent, and the zero/single-source guards. In ingest, **material** means a deeper read corrects a claim the dependent built on, adds a contradiction it doesn't track, or revises an interpretation it inherited — set the dependent `status: needs-update` with the hand-off reason — e.g. `needs_update_reason: inherited the pre-correction claim from {stem}; reingest revised it — re-check the Key Claims figure against {stem} Evidence and update or drop`; **additive-only** (extra evidence, fixed typos, still-correct fuller write-up, attachment adds) does not cascade — touch `updated:` only on dependents whose `Sources` callout changed mechanically. When the cascade adds or changes a non-obvious claim on a `verified` dependent without re-fact-checking that claim against the raw in this pass, and the change is not material (no `needs-update`), mark just that claim `*[unverified]*` (`CLAUDE.md` → Bullet Markers, Page Status) so the dependent keeps `status: verified` with the marked claim as its pending delta, rather than demoting the whole page to `draft`; the cascade reference's blanket `verified → draft` strip applies only to an unmarked change to verified content.
   - **Existing mode — schema-migration to dependents.** For each dependent flagged in Step 1 as missing callouts, insert them in correct position as empty placeholders (`> - None yet`); do not fill from the source page unless the scope naturally produces the content. Remove obsolete line-by-line citation style when encountered. Update `sources:` and `source_count`. Preserve contradictions.

8. **Run ingest verification.** Run the two independent verification packets in `.claude/skills/multi-skill/references/verification.md` — independent meaning each packet re-derives its checks from the raw source, not from the drafted page — over the raw source and every page created or touched; each packet returns its own pass/fail. The packets' independence is the tiered independent-refuter model (`verification.md` → Tiered Independent-Refuter Verification), run here at the draft tier an untrusted draft warrants — the full three-refuter Tier-3 quorum that earns `verified` runs later at `audit`, not in this pass. Record in the report, for each packet, the one late-section raw detail (final section, last figure, or appendix) re-located this pass and the one `#page=N` link spot-checked (or `n/a` for a non-PDF raw); a packet that records neither has not re-read the raw and the ingest is not complete. The located-deep-links check opens every locator on a cross-source bullet (one naming two or more sources — no sampling, since a mis-located citation hides behind a good sibling), and any mid-run citation fix triggers the repeated-literal sweep across `1-wiki/` before finalizing — both specified in `verification.md`; record the sweep in the report. If either packet reports a fix, apply it and re-run both packets; the operation is complete only after at least two independent packets are clean in the same round — both pass with no fix applied. If fixes cannot bring both packets clean within three fix-and-rerun rounds, or a fix for one packet re-introduces a finding the other just cleared (an oscillation), stop, leave the page as last written, and report the unresolved findings rather than looping. The reference routes the report — both modes write `2-outputs/ingest/ingest-YYYY-MM-DD-HHMM-{stem}.md`, using the new-source report shape in new mode and the reingest shape in existing mode — and records the frames/purpose. The report's `Recommended next ingests` section is compiled in Step 9. Do not set any touched page to `status: verified` — verification is a safety pass; pages stay `status: draft` until a later `audit`. Status on a `verified` page the reingest edits is claim-level (`CLAUDE.md` → Page Status): a new or changed non-obvious claim the reingest does not fact-check against the raw in this pass is marked `*[unverified]*` (`CLAUDE.md` → Bullet Markers) and the page keeps `status: verified` with that claim as its pending delta; a changed claim is marked `*[unverified]*` even when this Step 8 pass re-located and re-checked it — the pass is a safety check, not the audit-grade fact-check that earns unmarked-`verified`, and ingest does not re-stamp `verified_hash:` (`CLAUDE.md` → Page Status: only `audit` fact-checking the raw earns or holds `verified`; the skill that changed a claim never self-re-stamps). Only a claim re-read and confirmed unchanged (no body edit) needs no marker. The whole page resets to `status: draft` (stripping the now-stale `verified_hash:` — the body-hash stamp `audit` writes when it verifies a page) only on an unmarked change to verified content — including a `frames:` scope-widening, which grows the unmarked body and so always resets (Step 1 frames rule). Mechanical frontmatter touches (`updated:`, `source_count:`) are hash-safe and do not reset it; a non-`verified` page carries no hash and needs no marker. Because lint is structural, it will not catch a `frames:` widening that fails to grow the body — this packet is that guard.

9. **Compile the `Recommended next ingests` report section.** Close the report written in Step 8 with its `Recommended next ingests` section: the papers that would fill a gap this ingest surfaced — a single-source page this ingest created or left wanting its primary, a watch item it added, a dangling concept that now warrants its own source, or uningested prior work this source cites and leans on. One entry per paper: author and year, the title, and one line on the gap it fills. This is ingest-derived, not a generic literature dump — list only what this source's gaps actually point to. List only papers you are confident genuinely exist — never fabricate a title, author, or venue (mark an uncertain one `(verify exists)`); write `none` when the ingest surfaced no specific next-source (it is not a quota — do not pad it). The report shapes and the honesty guard are in `.claude/skills/multi-skill/references/verification.md`.

10. **Update `1-wiki/hot.md`.** Reuse the single 24-hour UTC stamp from `TZ='UTC' date '+%Y-%m-%d-%H%M'` across the Step 8 report filename, this bullet, and the Step 11 log heading (one stamp, not re-fetched per step, never a bare `date` — a re-fetch drifts them across a minute boundary).
   - Prepend Recent activity: `- [YYYY-MM-DD HH:MM] {ingest | ingest-deep | reingest} | {stem}` (verb by mode).
   - Add new low-coverage, `needs-update`, and contradiction-bearing concept/entity or synthesis pages to Open threads (pipe-rendered wikilinks).
   - Never edit Active focus unless explicitly asked.

## Edge Cases

- **Ambiguous or missing filename.** If the user names a source without its exact raw filename (e.g. "ingest the attention paper"), resolve the `{stem}` from `0-raw/` and confirm it before reading. The resolved stem also determines new vs. existing mode (Step 1).
- **No durable idea (new).** Create only the source page; do not invent concept/entity pages.
- **Near-duplicate concept/entity page.** Update the existing note instead of creating a new one.
- **Contradiction.** Preserve both sides in `Contradictions`; set affected concept/entity or synthesis pages to `needs-update`.
- **Source relevance is unclear.** Use `AskUserQuestion` to ask whether to ingest, skip, or file only a minimal source page.
- **No frame provided.** Default to unscoped: generate your own takeaways and contribution; no missed/adjacent buckets surface.
- **Frame provided but the paper doesn't fit it.** Say so in Step 3 before any questions, via one `AskUserQuestion` with three options: (a) widen the frame, (b) drop the frame and ingest unscoped, (c) skip the source.
- **Deep ingest without a purpose.** Ask before reading; fall back to normal mode if the user can't articulate one.
- **Existing mode — reingest reveals a new reusable idea.** Propose it; do not create it without approval.
- **Existing mode — a reingest landed wrong and needs reverting.** The prior committed page view is in git (ingest does not snapshot to `2-outputs/`): restore it with `git checkout <sha> -- 1-wiki/sources/{stem}.md`, then reverse this run's cascade — walk the Step 7 dependents and strip only the `needs_update_reason:` clause this run appended, preserving any prior hand-off (`dependent-cascade.md` append rule). Also restore any attachment this run quarantined (from `2-outputs/quarantined/attachments/{stem}/`), and remove the pages plus the `index.md`/`hot.md`/`log.md` entries this run added — or `forget` them, since `git checkout <sha>` cannot delete files that did not exist at `<sha>`. A new-mode ingest that landed wrong is reversed by `forget` on the source page and every concept/entity page it created. Uncommitted prior edits are not recoverable, which is why Step 5 gates on a clean working tree.
- **Existing mode — reingest weakens support for a dependent.** Mark it `needs-update` or remove the source from its support list after approval. If the dependent is a synthesis and removing this source drops it below two sources, do not leave it lint-failing: surface forget's stub-or-remove decision via `AskUserQuestion` — set `single_source_stub: true` (deliberate stub), re-source it, or quarantine it (`forget`) — and write a precise `needs_update_reason:` if it stays under-supported.
- **Existing mode — saved frames no longer fit the reason** (e.g. a contradiction outside every frame). Surface it in Step 3; per the Step 1 frames rule the user can keep the frames (item stays out of scope), append a lens that admits it, or clear to unscoped.
- **Existing mode — existing attachment is fine.** Default `keep`: no extraction, no embed change.
- **Concept/entity page would exceed one idea, or evidence is too dense.** Split the page; keep dense evidence in the source page (split by section if needed).
- **Image extraction fails.** Report it in Step 4; offer a locator-only reference, a different render, or a manual screenshot the user places in `1-wiki/attachments/{stem}/`.
- **The user only needs a few missing details on an existing page.** That is existing-source mode with reason `missed evidence` — a full re-read with a targeted write, not a full deep pass.

## Limits

- One source per invocation.
- Never modify `0-raw/`. Image extraction writes only to `1-wiki/attachments/{stem}/`; removed attachments go to `2-outputs/quarantined/attachments/{stem}/` (never deleted outright).
- Concept/entity pages stay atomic, short, and capped at one defining image — even when the source page goes deep. Deep detail belongs in source pages unless it is itself the reusable idea.
- Frames and the depth purpose scope the source page only, never the concept/entity pages (which stay reusable across future framings). The remaining cross-cutting rules are stated once upstream, not re-argued here: deep requires a stated purpose (Depth), sections are not quotas (Purpose), no bold or italic and pipe-rendered `.md` wikilinks (Conventions).
