# ingest

Process one raw source into a source page, then create or update the concept/entity pages it supports. One skill, three composable modes — **new vs. existing** (auto-detected by whether `1-wiki/sources/{stem}.md` exists), **normal vs. deep** (source-page depth), and optional **frames** (angles scoping the page) — composing on one shared spine (read → write/update the source page → create/update concept-entity pages → verify). What each mode adds:

| Mode | What it adds to the spine |
|---|---|
| New (default) | Create the source page first-time; propose new concept/entity pages; add index entries. |
| Existing (page exists) | Confirm a reason; run the always-on checks (`references/existing-mode.md`); attachment keep/replace/remove/add; cascade `needs-update` to dependents; preserve `created:`. |
| Normal depth (default) | High-level orientation only. |
| Deep (explicit opt-in) | Reference-grade source page (equations, hyperparameters, tables, more figures, deeper Method/Appraisal) within a stated purpose. Concept/entity pages stay atomic. |
| Frames (optional) | Scope every source-page section to the frames' union (an upper bound on scope — sections no frame admits stay empty); save each to `frames:`. |

## Purpose

The source page preserves what this source contributes and where the exact evidence lives. Concept/entity pages capture reusable ideas in simple language — do not turn them into source summaries.

**New vs. existing source (auto-detect)** — detected in Step 1 by whether `1-wiki/sources/{stem}.md` exists. New creates the source page first-time and proposes the concept/entity pages it genuinely supports. Existing reingests in place for a stated reason, updating the evidence layer without bloating concept/entity pages into source summaries — but "bloat" is source-summary detail, not source support, so attaching a genuinely-supporting source and the claims it contributes (including to a page the re-read newly reveals it supports) is maintaining the cross-source record, not bloat. It prevents duplicate source pages and preserves the prior view via git history (Step 5 clean-tree precondition). Terminology: "update" = any in-place modification; "refresh" = the `schema refresh` reason.

**Depth.** Default normal; deep is an explicit opt-in Step 1 asks about, never inferred from the source or the request's tone. Deep — the longest-lasting page in the wiki and the easiest to bloat — **requires a stated depth purpose**: what the depth is *for* (a frame, a paper section the page will support, a specific question, a paper being written). "I want a thorough page" is not a purpose; it produces an indiscriminate source dump. Deep expands the *depth* of in-scope sections, never the breadth of out-of-scope ones, and never the concept/entity pages (which stay one-idea and simple regardless). Throughout, `ingest-deep` is the verb token used in log and hot entries.

**Frames are the ceiling, not a starting point.** A frame (lens, angle) is one positive scope on the source page — "only the attention-efficiency results", "the optimizer-convergence proof" — and `frames:` may hold several, the page scoped to their union. Every section is scoped to that union; sections no frame admits stay at the empty placeholder (`> - None noted`), never filled from outside the frames even when the source supports it. The raw is still fully read (Step 2), and Step 8 judges the page against the frames' union (or deep purpose), not the whole source — a thin page under a narrow frame is the correct outcome. In deep mode a frame, when given, *is* the depth purpose. A frame is a positive lens, never a hard exclusion, so reframing only ever *adds*; the reuse / append / clear decision and its scope-widening test are in `references/existing-mode.md`.

**A frame may be a pre-applied analysis layer.** When the frame is a categorization the user already attached to the source (tags, a typology, a labelling scheme), treat those labels as the user's analysis layer, not facts about the source: assess each against what the source actually does — never copy them verbatim as characterizations, never blanket-reject them. The page stays grounded in the source's own language, not the frame's vocabulary. (Project-specific frame conventions live in `ingest-memory.md`.)

**Sections are not quotas.** Required callout sections must appear in every page, but they do not all need content. If a section has nothing genuine and source-supported to say, leave the empty placeholder (`> - None noted` for source pages, `> - None yet` for concept/entity pages). A page with several placeholders is correct when the source genuinely earns nothing there. Do not invent, paraphrase, or pad to avoid an empty section. Deep ingest expands the depth of filled sections, not their count.

**Pattern: analyze → plan → validate → execute → verify.** Step 2 analyzes (reads the source); Step 3 plans and validates (per-decision `AskUserQuestion`, user approval the gate); Steps 4–7 execute only approved edits; Step 8 verifies (two independent packets — factual accuracy, clarity).

**A deliberate divergence.** The smart-note method assigns a human to re-voice the note at a commit point; this wiki instead keeps pages LLM-drafted at `status: draft`, with the Step 8 verification packets (factual accuracy, clarity, AI-voice scan) plus a later `audit` standing in for that. The packets are therefore load-bearing, not optional.

## Conventions (Recap)

- **Source-page filename matches the raw source stem exactly** (drop the extension, preserve case and internal punctuation): `0-raw/papers/<bibkey>.pdf` → `1-wiki/sources/<bibkey>.md`; `0-raw/articles/<slug>.md` → `1-wiki/sources/<slug>.md`. The attachments folder uses the same stem: `1-wiki/attachments/{stem}/`.
- **Every wikilink is path-qualified from the repo root, includes `.md`, and uses a pipe-rendered display name** that reads naturally: `[[1-wiki/concepts/scaled-dot-product-attention.md|Scaled Dot-Product Attention]]`, authors `[[1-wiki/entities/ashish-vaswani.md|Ashish Vaswani]]`. The pipe display is required (a bare path renders unreadably) and carries no whitespace around the pipe (`[[path|display]]`, never `[[path | display]]`); attachment embeds `![[...]]` take the path but no pipe.
- **No bold and no italic in wiki content** (source, concept/entity, synthesis, hot/index/log). The only structural exceptions are LaTeX math and the bullet markers `*\[unverified\]*` / `*\[tentative\]*` / `*\[disputed — see Contradictions\]*`. Inline backticks for paths, code, and quoted prompt addenda are fine.
- **Use `AskUserQuestion` for every user-facing decision** in Step 3. One decision per question; do not batch. Mark the option you recommend `(Recommended)` and place it first — base the lean on your read of the source (e.g. `Include (Recommended)` for a clearly-warranted concept page, `Drop (Recommended)` for a marginal candidate). Omit a recommendation only on a genuine no-lean call, and say so in the question. Phrase every question and option label for a reader who knows nothing about the wiki's internal vocabulary — gloss any callout, status, or frames term inside the option (present `add to Not This` as "record as a contrast (Not This)", `mark needs-update` as "flag for re-checking (needs-update)", `clear to unscoped` as "widen back to the whole source"), the same way the Step 1 depth question already describes both modes plainly.
- **Structural authority lives in `CLAUDE.md`.** Required-section lists, frontmatter blocks, block-ID placement, and locator rules are the shared schema — this skill cites `CLAUDE.md` rather than copying (copying drifts). The Step 8 verification spec lives in the shared `.claude/skills/multi-skill/references/verification.md` (one canonical copy, since `query` runs the same packets). The extraction/citation *procedures* the agent runs are self-contained in the skill's steps and `references/`.
- **Status and markers (recap).** New or changed durable content stays `status: draft` (a later `audit` earns `verified`); ingest never sets `verified` or re-stamps `verified_hash:`. On a `verified` page a reingest edits: a **new** non-obvious claim added but not raw-fact-checked this pass is marked `*[unverified]*`, and the page keeps `verified` with that claim as its pending delta (a masked new line never enters `verified_hash`); a **changed** verified claim moves the hash and demotes the page to `draft` (marking does not hold it), as do an embed or `Sources` change and a `frames:` scope-widening (which grows the unmarked body). Mechanical frontmatter touches (`updated:`, `source_count:`) are hash-safe. Full rules: `CLAUDE.md` → Page Status and Bullet Markers.

## When To Invoke

Use when the user names a specific raw file under `0-raw/` and asks to ingest, process, capture, digest, import, deep-ingest, reingest, or refresh it. Implicit phrasings count too — "add this paper to the wiki" (or knowledge base / vault), "make a source page for X", "write up this PDF", "summarize this PDF into the wiki", "take notes on this paper". (Absent any destination signal, "take notes on this paper" defaults to ingest — the wiki. Route elsewhere only when the user names an external structured database as the destination and the project actually defines a capture skill for it; see When Not To Invoke.) The user may optionally pass one or more frame texts to scope the summary; without one, default behaviour applies. Mode is auto-detected (Step 1), so the user need not say which.

## When Not To Invoke

- The user asks a research question. Use `query`.
- The user wants to remove existing wiki content. Use `forget`.
- The user wants to replace content while preserving the old view (a swap/merge/split decided in advance). Use `supersede`.
- The user wants to re-verify an existing page against its raw source without rewriting it. Use `audit`. (In ingest, "refresh" means re-reading the raw and rewriting the page in place — a reingest — not re-verification.)
- The user wants a source captured into an external structured database (a project-specific field set) rather than the wiki. If the project defines a capture skill for that destination, use it. `ingest` instead builds **wiki** pages — a full source page plus the atomic, reusable concept/entity pages it supports — all source-grounded and interlinked. (The difference is destination and form, not depth.)

## Procedure

```text
Ingest Progress:
- [ ] Step 1: Load orientation; detect mode (new/existing, depth); confirm reason/purpose and frame
- [ ] Step 2: Read the source (confirm full-text coverage)
- [ ] Step 3: Surface the plan via per-decision AskUserQuestion calls
- [ ] Step 4: Extract or prepare approved attachments
- [ ] Step 5: Write or update the source page
- [ ] Step 6: Draft or update concept/entity (and synthesis) page changes
- [ ] Step 7: Apply approved edits; cascade needs-update (existing mode)
- [ ] Step 8: Run ingest verification (see .claude/skills/multi-skill/references/verification.md)
- [ ] Step 9: Compile the Recommended next ingests report section
- [ ] Step 10: Update hot.md
- [ ] Step 11: Prepend log.md
```

1. **Load orientation; detect mode; confirm reason/purpose and frame.** Read `1-wiki/hot.md`, `1-wiki/index.md`, `.claude/skills/ingest/ingest-memory.md`, and `.claude/skills/multi-skill/multi-skill-memory.md` if not already loaded this session. Use hot/index to avoid duplicate concept/entity pages and to identify related notes and near-duplicates. Apply prior memory corrections about scope, framing, and source-supportedness before drafting.

   Then settle the mode:
   - **Resolve the stem first.** If the user did not give the exact raw filename, resolve `{stem}` from `0-raw/` and confirm it before the existence check (see Edge Cases) — the confirmed stem drives new/existing detection.
   - **New vs. existing:** does `1-wiki/sources/{stem}.md` exist? If yes → existing-source (reingest) mode. If no → new-source mode. Before concluding "new," also rule out a legacy page that maps to this raw under a different name: grep `1-wiki/sources/` for any page whose `file:` frontmatter points at this raw file, and scan the directory for case- or kebab-variants of the stem — anchor the variant match on the stem's extension or wikilink-closing delimiter (`\.md`/`\.pdf` followed by `]`, `|`, or `#`, mirroring `supersede`; that delimiter anchors the `file:`-frontmatter grep, whose target is wikilink-wrapped, while the bare-filename directory scan carries no `]`/`|`/`#` and is anchored instead on the extension boundary or end of name, `\.md$`) so a prefix-of-a-longer-stem neighbour does not mis-match (a search for `Devlin2019BERTPO`, say, must not also catch a different, longer stem that merely begins with it), and eyeball each hit to confirm it maps to this exact raw. A match is existing mode and a rename candidate for Step 3, not new — skipping this check silently creates a duplicate source page and abandons the prior view, the hardest ingest error to reverse. Retain the grep run and its hits (or "no match") to record in the Step 8 report; new-source mode is not entered until this check is run and its result retained — an unretained check is treated as not-run (mirroring Step 8's proof-of-re-read gate). When `1-wiki/sources/{stem}.md` does exist, before concluding reingest, confirm its `file:` frontmatter resolves to the exact raw the user named: the stem drops both extension and subdirectory (Conventions), so `0-raw/papers/{stem}.pdf` and `0-raw/articles/{stem}.md` both map to `1-wiki/sources/{stem}.md`, and if the existing page's `file:` points at a different raw this is a stem collision, not a reingest — surface it via `AskUserQuestion` (same stem, different sources) rather than silently re-reading the other raw's `file:` in Step 2.
   - **Depth:** normal unless the user opts into deep. **Do not infer deep from the source or the request's tone — ask.** If the user's request did not already specify the depth, ask once with `AskUserQuestion`, describing both modes for a reader who knows nothing about them. Recommended wording: `"Normal or deep ingest? Normal gives a high-level orientation — the key claims and where the evidence lives — and stays compact. Deep produces a reference-grade source page: equations, hyperparameters, full result tables, and careful figure/page references, so it can support writing or serve as a long-term reference. Deep takes longer and the page is much larger, and you must say what the depth is for. Concept/entity pages stay short and atomic either way."` Options: `Normal` / `Deep`. Honour an explicit request ("deep ingest", "reference-grade") without re-asking.
     - **Deep requires a stated depth purpose** — what the depth is for (a frame, a paper section the page will support, a specific question, a paper being written). A frame is the purpose only when it is a genuine scope — a section, a question, or a concrete future use; echo it back. A bare topic word (e.g. "transformers") is not a scope — treat it like "thorough" and ask for the non-frame purpose. Without a frame, ask for the non-frame purpose before reading. Do not accept "thorough" or "long-term reference." If the user can't give a purpose after one focused ask, fall back to normal and proceed. Capture it: a frame goes in the `frames:` list at write time; a non-frame purpose goes in the report's `purpose:` field (not on the page).
     - **Existing-source mode never auto-instigates deep** — settle the reingest reason and frame first; switch to deep only via the same explicit ask (`references/existing-mode.md`).
   - **Existing-source mode only — confirm the reason, then run the always-on checks** (full detail in `references/existing-mode.md`): the reingest reason (`schema refresh` / `missed evidence` / `fuller write-up` / `contradiction` / `corrected interpretation` / source-shaped-cleanup — no reason given → ask, since a reingest without one creates drift); the `needs-update` and `*[tentative]*`-resolution checks; the schema-migration check against the page and its dependents; the frames decision (reuse / append / clear, with its scope-widening test); and the deep-purpose recovery. Locate every dependent — every page listing this source in `sources:` — so Step 7's cascade can reach it.

2. **Read the source in full (confirm full-text coverage).** Read the raw file from `0-raw/` (use the existing page's `file:` frontmatter to find it in existing mode). **Every ingest — new, existing, normal, or deep — fully reads the raw, cover to cover. There is no targeted partial re-read; the reason, frames, and depth control what gets written, never how much gets read.** Deep mode reads the whole raw and additionally goes deeper on the equations/figures/tables the depth purpose needs — out-of-scope sections are still read, just written only as the scope admits. For long books or surveys, propose chapter-level source pages instead of one oversized note (the unit read is then the chapter, still read in full). If frames are in effect, read through them as well, keeping both the framed and a general read in mind. **Read critically as well as comprehendingly (Darwin's golden rule):** on sight, capture what cuts against the source's claims, the assumptions it leaves unstated, and the strongest sceptical critique — disconfirming evidence escapes attention unless caught as you read; when a frame is active, also ask what cuts against the frame's thesis (a positive lens must not become confirmation-seeking). These feed `Assumptions`, `Appraisal`, and concept-page `Disconfirming Evidence`, and `Contradictions` where they bear on scope. **Confirm full-text coverage before drafting** by a positive probe, not a vibe: the final page/section is present and intelligible (catches truncation) and body text — not only figure captions — is extractable (catches an image-only scan or abstract-only paywall). For an image- or media-based raw, confirm instead that the visual content the page draws on (every slide, frame, figure) is present and legible (absent body text is expected there; mirrors `verification.md`). If coverage is partial (truncated, image-only, OCR-garbled, paywalled, partially extracted), surface it in the planning step and draft only what is covered — never summarize partial content as the whole source. If the source's relevance is unclear, see Edge Cases before drafting.

   **Register the raw's pagination before any locator is written (PDF raws).** What a page prints (`p. M`) is a fact about the raw, not derivable by rule — an appendix may restart numbering, a proceedings build may start high, a body may print no number — and getting it wrong yields a confidently-wrong citation that passes every structural check. Record it once per raw in `.claude/skills/multi-skill/pagination-map.md`:

   ```bash
   python3 .claude/skills/multi-skill/scripts/pagination_map.py 0-raw/papers/{stem}.pdf                    # propose a map from the footers
   python3 .claude/skills/multi-skill/scripts/pagination_map.py --verify 0-raw/papers/{stem}.pdf {outdir}  # render footer crops to eyeball
   ```

   The generator proposes; **you confirm** — eyeball every proposed line against a rendered footer crop before adding the `## 0-raw/papers/{stem}.pdf` section, since a wrong `none` licenses stripping a correct printed page from a citation. An unregistered raw still ingests (the locator checks fall back to the `app.`-anchor heuristic) but raises `pagination_map_unregistered`, and `locator_page_mismatch` cannot run on it. In existing-source mode, register the raw if it is not already.

3. **Surface the plan via per-decision `AskUserQuestion` calls.** Post a single chat context message first — the full slate, so the sequence is no surprise and the user can pre-empt whole classes in chat ("just the obvious concepts, skip the images") — then ask each decision through its own call: one per call, never batched, framed include / drop / other, the recommended option marked `(Recommended)` and first. The decision types below are not a fixed question count — types 1, 2, 6, and 8 fire once per candidate, so a dense paper expands to 10–20 calls. Do not collapse per-candidate questions into one summary to save calls; the only sanctioned shortcut is the user pre-approving or pre-skipping whole classes in chat. Step 3 is complete only when every decision type with at least one candidate has fired — including the easy-to-skip contradiction (type 6) and relationship-sweep (type 8) questions; do not advance to Step 4 with mapped-but-unasked relationships or contradictions outstanding.

   The context-post catalogue (frame echo, takeaways, sibling back-references, contradiction cross-check, relationship sweep, retrieval tags, disconfirming evidence) and the per-decision question types (the emphasis question, then membership questions 1–10) are in `references/planning-questions.md` — work the plan through it, one `AskUserQuestion` call per decision.
   Only approved items become wiki content. Do not act on partial answers. If the user pauses mid-sequence, wait; on resume, keep the answers already given and continue from the next unanswered decision (do not re-ask answered ones or discard them).

4. **Extract or prepare approved attachments.** Write each approved image to `1-wiki/attachments/{stem}/{name}.png` (create the folder if needed). Tools:
   - PDFs: `pdftoppm -png -r 300 -singlefile -f {page} -l {page} "{raw}" {outprefix}` renders one page to exactly `{outprefix}.png` (300 DPI per `CLAUDE.md` → Attachments; `-singlefile` avoids the zero-padded page suffix plain `pdftoppm` adds on a 100+-page PDF, which would misname the file). `pdfimages -png -f {page} -l {page} "{raw}" {outprefix}` pulls an embedded image directly, but writes indexed names (`{outprefix}-000.png`) and may emit `.ppm`/`.pbm` for non-RGB images — rename (and convert with `magick in.ppm out.png` if needed) to the approved `{name}.png` before embedding, or the Step 5 embed dangles. To crop a region tightly to the figure plus its full caption — or to clip it directly by PDF coordinate with PyMuPDF (the most reliable path, no DPI-pixel arithmetic) — see `references/attachments.md` (the point-space clip, the render→inspect→crop loop, and a worked `magick -crop` example).
   - Non-PDF media already in `0-raw/media/`: copy or symlink into `1-wiki/attachments/{stem}/`. If the copy fails (missing or unreadable file), report it and ask rather than leaving a dangling embed.
   - **Existing mode:** these mutations run only after the Step 5 clean-working-tree gate and any approved legacy rename (a `replace`/`remove` overwrites or moves a committed attachment). `add`/`replace` write as above. When a `replace` changes the image *content* (a corrected figure, not a tighter crop or higher DPI), treat every page embedding that path as a dependent — grep the embed path `1-wiki/attachments/{stem}/{name}` across all of `1-wiki/` (a path-qualified embed can be reused by a page under a different stem, so not only this source's `sources:` dependents) and, on a `verified` embedding page whose non-obvious claim the image backs, mark that claim `*[unverified]*` (the body text is byte-identical, so `verified_hash` does not move and lint cannot catch a missing mark; mandatory, mirrors `supersede`, `dependent-cascade.md`). `remove` quarantines the file to `2-outputs/forget/quarantine/attachments/{stem}/{name}.png` per `.claude/skills/multi-skill/references/quarantine-path-convention.md` (the `cmp -s` byte-identity check, the `git check-ignore -q` guard, and the `-N` name-clash suffix rule) — never delete outright, so the prior view is preserved.
   - Never write into `0-raw/`. If extraction fails (encrypted PDF, vector-only figure), report it instead of silently skipping, and put the fallback to the user as a fresh `AskUserQuestion` (the Step 3 approval assumed a clean extraction that did not materialize): drop the embed, accept a locator-only reference, try a different render, or place a manual screenshot. A dropped embed must also be removed from the Step 5 write plan and never written to `attachments:` or embedded in the body, or Step 8 catches it as a dangling embed.

5. **Write or update the source page** at `1-wiki/sources/{stem}.md`. Use the source-page frontmatter and required callouts from `CLAUDE.md`. Every callout carries a block ID — the kebab-case of its display title — as the last line inside the `>` block (`> ^tldr`, `> ^evidence`, …; no blank `>` line before it; `CLAUDE.md` → Callout Block IDs), on every callout including empty placeholders and any inserted by the schema-migration check. For PDF raws, every page locator in `Evidence`/`Method` (and any quoted in `Key Claims`/`TL;DR`) carries a `#page=N` deep-link with the structural anchor and page **together inside** the display: `[[0-raw/papers/{stem}.pdf#page=5|sec. 3, p. 5]]` (never split the anchor outside the link or leave a page-only display — lint `source_locator_incomplete`). Cite abstract-drawn content `abstract, p. 1`, never relabelled `sec. 1`. Non-PDF raws use the per-type locator formats (`CLAUDE.md` → Source Pages → Evidence pointer formats) and link the raw without a `#page=N` anchor. `N` is the physical PDF page, which differs from the printed page whenever pagination does not start at 1 — read `p. M` off the pagination map registered in Step 2 (not by re-deriving an offset a restarting appendix would defeat), and where the map says a page prints nothing, drop `p. M` and cite the anchor alone (`CLAUDE.md` → Source Support And Verification; `lint`'s `locator_page_mismatch` checks every `p. M`). Worked examples and the figure-crop loop are in `references/attachments.md`.
   - **New:** draft the full page with `status: draft` (a later `audit` promotes it). `file:` points at the raw path-qualified into `0-raw/` (`file: "[[0-raw/papers/<bibkey>.pdf]]"`); authors use pipe-rendered display, auto-linking any author already in `1-wiki/entities/` and leaving a one-off collaborator with no page as a dangling path-qualified `authors:` wikilink (a new author page is proposed only via a Step 3 create-new question, and only for a prominent author — `CLAUDE.md` → Author page creation policy).
   - **Existing:** before the first overwrite, confirm a clean working tree across every page this run will write — the source page, every dependent the Step 7 cascade will touch (located in Step 1), and, on a legacy rename, the legacy page and its attachments folder: `git status --short 1-wiki/sources/{stem}.md` plus the legacy path and each dependent path. If any is dirty, surface via `AskUserQuestion` (proceed / commit-or-stash first / abort) — the prior view is preserved by git history of the committed page only (ingest does not snapshot to `2-outputs/`), so an uncommitted page's prior edits would be lost silently, including on a cascade-written dependent (`CLAUDE.md` Safety: never overwrite silently). If Step 3 approved a legacy-page rename, then — working tree confirmed clean — `git mv` the legacy page `1-wiki/sources/{legacy}.md` → `1-wiki/sources/{stem}.md` and its attachments folder `1-wiki/attachments/{legacy}/` → `1-wiki/attachments/{stem}/`, and rewrite every inbound reference per `.claude/skills/multi-skill/references/inbound-reference-discovery.md`, before any write — otherwise the write lands a fresh `{stem}.md` beside the untouched legacy page, re-creating the duplicate the Step 1 guard exists to prevent. Then preserve `type:`, `file:`, and `created:`; touch `updated:`. Apply the Step 1 schema-migration check — insert each missing required callout in its correct position from `CLAUDE.md`, filling from the raw where the scope makes that natural, otherwise the empty placeholder. Sync `attachments:` with the current files (drop entries for quarantined files). Keep `Concepts and Entities` synchronized. Fold any stale Figures callout's embeds into `Evidence`/`Method` and drop it.
   - **Frames frontmatter:** record each frame as an item in the `frames:` list; in existing mode apply the Step 1 frames decision (reuse / append / clear, with its scope-widening consequence — a genuine widening grows the body and resets `status: draft`). `frames:` is always present — write `frames: []` when unscoped, never omit it — and appears in frontmatter only, never in the page body.
   - **Sections are not quotas.** Fill only with genuine, source-grounded content within the frames/purpose; leave `> - None noted` otherwise. Per-callout writing guidance (what genuinely belongs in each of `TL;DR`, `Contribution`, `Key Claims`, `Evidence`, `Method`, `Assumptions`, `Limitations`, `Appraisal`, `Concepts and Entities`, `Contradictions`, `Open Questions`, `Connections`), the inline-embed format, and the citation / `attachments:` / no-bold rules are in `references/source-page-writing.md`.

6. **Draft or update concept/entity (and synthesis) page changes.**
   - For each approved concept/entity page, apply the one-idea test: one idea or named thing? explained simply? reusable for later writing? reads standalone with its source page closed? supported by at least one source page? connects to existing notes?
   - Existing page: a small edit (usually 1–3 bullets or a tighter `Idea`). New page: the full concept/entity template from `CLAUDE.md`, written `status: draft` with `aliases:` populated when the source gives the concept a genuine acronym, alternate name, or older term (e.g. `aliases: [Adam]` on an Adaptive Moment Estimation page — `CLAUDE.md` → Concepts and Entities) and on a same-as merge, otherwise `aliases: []`, each callout ending in its block ID — the kebab-case of its display title, so `[!why]` Why It Matters takes `> ^why-it-matters` (`> ^idea`, `> ^why-it-matters`, …; see `CLAUDE.md` → Callout Block IDs). Only fill sections with genuine content (sections are not quotas) — a new page often earns only `Idea`, `Why It Matters`, `Sources`; the rest stay `> - None yet`.
   - At most one defining image per page, embedded in `Idea`, illustrating the idea (not summarizing the source); never in `Examples`. Deep mode: do not push equations, metric tables, or long method detail into concept/entity pages unless the page is explicitly about that idea.
   - Add the source to `sources:` (`- "[[1-wiki/sources/{stem}.md|{stem}]]"`) and the `Sources` callout. Cite each non-obvious factual claim inline to the specific source it draws from; leave obvious/definitional bullets and own-voice judgement (`Disconfirming Evidence`, `Open Questions`) uncited — not citation wallpaper. Write the citation in one of the two canonical forms (Form 1 attributive / Form 2 parenthetical), pairing the source-page wikilink (which source) with a located raw deep-link carrying the structural anchor and printed page `p. M`, `N` the physical PDF page (a non-PDF swaps the deep-link for a backticked per-type locator) — `CLAUDE.md` → Source Support And Verification is the authority for both forms, and a bare source link with no locator is incomplete on a non-obvious claim. Never fabricate a locator: if you cannot confirm the page or section in the raw, mark the claim `*[unverified]*` / `*[tentative]*` or quote-and-locate only what you can confirm — a plausible but wrong `#page=N` is worse than an honest gap. On a bullet naming two or more sources (the cross-source callouts — `Contradictions`, `Not This`, `Disconfirming Evidence`, synthesis `Tensions`), cite each clause to the source that states it, not the source the bullet is framed about: a contrast sentence has two origins and two locators, and a true claim from source A pinned to a locator in source B passes every structural check, so only opening the cited page catches it (`CLAUDE.md` → the multi-source-bullet rule). Word each cited claim to represent its source faithfully — no overstatement (Step 8 re-checks claim by claim); a causal/comparative/evaluative claim carries its source's reason, never a fabricated mechanism (recast as own-voice or flag `*[tentative]*` if the source gives none). Wikilink every genuine reference to an existing wiki page, on every occurrence; a page never links to itself. See `CLAUDE.md` → Source Support And Verification, Plain-Language Style, and Wikilink Format.
   - Synthesis pages (existing mode): keep the cross-source answer, evidence map, tensions, and source list; update embeds the same way. ingest only updates an existing synthesis whose source list this source already belongs to; it never creates a synthesis page — that is `synthesis`'s job. A source that strengthens a new multi-source cluster is surfaced as a synthesis candidate in Step 3, not authored here.

7. **Apply approved edits; cascade (existing mode).** Mechanical fields need no separate approval: `created:` (new pages), `updated:` (touched pages), `source_count:` to match `sources:`, `attachments:` to match files, `Concepts and Entities` callout updates, and `1-wiki/index.md` entries for new/renamed pages.
   - **Existing mode — cascade `needs-update`.** Walk every dependent (located in Step 1) and apply the shared dependent-cascade mechanics in `.claude/skills/multi-skill/references/dependent-cascade.md` — what a dependent is, the `needs_update_reason:` hand-off and append-don't-overwrite rule, `source_count`/`Sources` sync, the `verified_hash` strip on demotion, and the zero/single-source guards. In ingest, **material** means a deeper read corrects a claim the dependent built on, adds a contradiction it doesn't track, or revises an interpretation it inherited — set the dependent `status: needs-update` with the hand-off reason (e.g. `needs_update_reason: inherited the pre-correction claim from {stem}; reingest revised it — re-check the Key Claims figure against {stem} Evidence and update or drop`); **additive-only** (extra evidence, fixed typos, still-correct fuller write-up, attachment adds) does not cascade — touch `updated:` only on dependents whose `Sources` callout changed mechanically. When the cascade adds or changes a claim on a `verified` dependent, apply the status/marker rules from Conventions (Recap): a non-material **added** claim is marked `*[unverified]*` (the dependent stays `verified` with that pending delta); a **changed** verified claim demotes it to `draft`.
   - **Existing mode — schema-migration to dependents.** For each dependent flagged in Step 1 as missing callouts, insert them in correct position as empty placeholders (`> - None yet`); do not fill from the source page unless the scope naturally produces the content. Remove obsolete line-by-line citation style when encountered. Update `sources:` and `source_count`. Preserve contradictions.

8. **Run ingest verification.** Run the two independent verification packets in `.claude/skills/multi-skill/references/verification.md` over the raw source and every page created or touched — independent meaning each re-derives its checks from the raw, not the drafted page (the tiered independent-refuter model, run here at the draft tier; the Tier-3 quorum that earns `verified` runs later at `audit`). Record, per packet, the one late-section raw detail re-located and the one `#page=N` spot-checked (or `n/a` for a non-PDF) — a packet recording neither has not re-read the raw and the ingest is not complete — and open every locator on a cross-source bullet (no sampling, since a mis-located citation hides behind a good sibling). Any mid-run citation fix triggers the repeated-literal sweep across `1-wiki/`; record it. Apply and re-run on any fix; the operation is complete only when both packets are clean in the same round, bounded to three fix-and-rerun rounds (stop and report on non-convergence or an oscillation, rather than looping). The reference routes the report — both modes write `2-outputs/ingest/ingest-YYYY-MM-DD-HHMM-{stem}.md` (new-source or reingest shape) and record frames/purpose; its `Recommended next ingests` section is compiled in Step 9. Do not set any touched page `status: verified` — verification is a safety pass; pages stay `draft` until a later `audit`, and ingest never re-stamps `verified_hash:`. Apply the status/marker rules from Conventions (Recap) to a `verified` page this reingest edits: a **new** unchecked claim marked `*[unverified]*` keeps the page `verified` with that pending delta; a **changed** verified claim (or an embed / `Sources` change, or a `frames:` scope-widening) demotes it to `draft`. Because lint is structural, it will not catch a `frames:` widening that fails to grow the body — this packet is that guard.

9. **Compile the `Recommended next ingests` report section.** Close the report written in Step 8 with its `Recommended next ingests` section: the papers that would fill a gap this ingest surfaced — a single-source page this ingest created or left wanting its primary, a watch item it added, a dangling concept that now warrants its own source, or uningested prior work this source cites and leans on. One entry per paper: author and year, the title, and one line on the gap it fills. This is ingest-derived, not a generic literature dump — list only what this source's gaps actually point to. List only papers you are confident genuinely exist — never fabricate a title, author, or venue (mark an uncertain one `(verify exists)`); write `none` when the ingest surfaced no specific next-source (it is not a quota — do not pad it). The report shapes and the honesty guard are in `.claude/skills/multi-skill/references/verification.md`.

10. **Update `1-wiki/hot.md`.** Reuse the single 24-hour UTC stamp from `TZ='UTC' date '+%Y-%m-%d-%H%M'` across the Step 8 report filename, this bullet, and the Step 11 log heading (one stamp, not re-fetched per step, never a bare `date` — a re-fetch drifts them across a minute boundary).
   - Prepend Recent activity: `- [YYYY-MM-DD HH:MM] {ingest | ingest-deep | reingest} | {stem}` (verb by mode).
   - Add new low-coverage, `needs-update`, and contradiction-bearing concept/entity or synthesis pages to Open threads (pipe-rendered wikilinks).
   - Never edit Active focus unless explicitly asked.

11. **Prepend `1-wiki/log.md`** (verb and report path by mode):

    New-source, normal:

    - Created: [[1-wiki/sources/{stem}.md|{stem}]], [[1-wiki/concepts/layer-normalization.md|layer normalization]]

## Edge Cases

- **Ambiguous or missing filename.** If the user names a source without its exact raw filename (e.g. "ingest the attention paper"), resolve the `{stem}` from `0-raw/` and confirm it before reading. The resolved stem also determines new vs. existing mode (Step 1).
- **No durable idea (new).** Create only the source page; do not invent concept/entity pages.
- **Near-duplicate concept/entity page.** Update the existing note instead of creating a new one.
- **Contradiction.** Preserve both sides in `Contradictions`; set affected concept/entity or synthesis pages to `needs-update`.
- **Source relevance is unclear.** Use `AskUserQuestion` to ask whether to ingest, skip, or file only a minimal source page.
- **No frame provided.** Default to unscoped: generate your own takeaways and contribution; no missed/adjacent buckets surface.
- **Frame provided but the paper doesn't fit it.** Say so in Step 3 before any questions, via one `AskUserQuestion` with three options: (a) widen the frame, (b) drop the frame and ingest unscoped, (c) skip the source.
- **Deep ingest without a purpose.** Ask before reading; fall back to normal mode if the user can't articulate one.
- **Existing mode — reingest reveals a new reusable idea.** Propose it; do not create it without approval.
- **Existing mode — a reingest landed wrong and needs reverting.** The prior committed page view is in git (ingest does not snapshot to `2-outputs/`): restore it with `git checkout <sha> -- 1-wiki/sources/{stem}.md`, then reverse this run's cascade — walk the Step 7 dependents and strip only the `needs_update_reason:` clause this run appended, preserving any prior hand-off (`dependent-cascade.md` append rule). Also restore any attachment this run quarantined (from `2-outputs/forget/quarantine/attachments/{stem}/`), and remove the pages plus the `index.md`/`hot.md`/`log.md` entries this run added — or `forget` them, since `git checkout <sha>` cannot delete files that did not exist at `<sha>`. A new-mode ingest that landed wrong is reversed by `forget` on the source page and every concept/entity page it created. Uncommitted prior edits are not recoverable, which is why Step 5 gates on a clean working tree.
- **Existing mode — reingest weakens support for a dependent.** Mark it `needs-update` or remove the source from its support list after approval. If the dependent is a synthesis and removing this source drops it below two sources, do not leave it lint-failing: surface forget's stub-or-remove decision via `AskUserQuestion` — set `single_source_stub: true` (deliberate stub), re-source it, or quarantine it (`forget`) — and write a precise `needs_update_reason:` if it stays under-supported.
- **Existing mode — saved frames no longer fit the reason** (e.g. a contradiction outside every frame). Surface it in Step 3; per the Step 1 frames rule the user can keep the frames (item stays out of scope), append a lens that admits it, or clear to unscoped.
- **Existing mode — existing attachment is fine.** Default `keep`: no extraction, no embed change.
- **Concept/entity page would exceed one idea, or evidence is too dense.** Split the page; keep dense evidence in the source page (split by section if needed).
- **Image extraction fails.** Report it in Step 4; offer a locator-only reference, a different render, or a manual screenshot the user places in `1-wiki/attachments/{stem}/`.
- **The user only needs a few missing details on an existing page.** That is existing-source mode with reason `missed evidence` — a full re-read with a targeted write, not a full deep pass.

## Limits

- One source per invocation.
- Never modify `0-raw/`. Image extraction writes only to `1-wiki/attachments/{stem}/`; removed attachments go to `2-outputs/forget/quarantine/attachments/{stem}/` (never deleted outright).
- Concept/entity pages stay atomic, short, and capped at one defining image — even when the source page goes deep. Deep detail belongs in source pages unless it is itself the reusable idea.
- Frames and the depth purpose scope the source page only, never the concept/entity pages (which stay reusable across future framings). The remaining cross-cutting rules are stated once upstream, not re-argued here: deep requires a stated purpose (Depth), sections are not quotas (Purpose), no bold or italic and pipe-rendered `.md` wikilinks (Conventions).
