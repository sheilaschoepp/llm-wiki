# lint — Check Catalogue

The full check roster for `lint`: every `check_id`, its severity, and what it means. Two provenances — **Script-emitted** (the canonical ids `check_wiki.py` prints; the script's `CHECKS` registry is the source of truth, `--list-checks` prints it as JSON) and **LLM-walk** (the Step 2 checks with no `check_id`). Cited from SKILL.md Step 2 (the walk) and Step 4 (report rows). Keep the Script-emitted half diffable against `--list-checks` — every id appears in both.

## Contents

- Script-emitted (`check_wiki.py`) — Critical / Warning / Info
- LLM-walk (Step 2; not script-emitted)
- Standalone (opt-in; not in the core battery)

## Checks

Checks come in two provenances. **Script-emitted** checks are the canonical `check_id`s `check_wiki.py` prints — cite them verbatim in report rows. The script emits `severity: error | warning | info`; in the report these render as **Critical / Warning / Info** respectively (CLAUDE.md → Severity vocabulary: wiki-facing reports use critical/warning/info — never error/suggestion). The script's `CHECKS` registry is the single source of truth for each check's `check_id` and severity; `python3 .claude/skills/lint/scripts/check_wiki.py --list-checks` prints it as JSON, so the Script-emitted list below can be diffed against the registry mechanically rather than by eye. **LLM-walk** checks have no `check_id` in the script; the agent performs them in the Step 2 walk and cites them by short name. When the script changes, this split is the doc side that must stay in sync with the registry — every id under Script-emitted must appear in `--list-checks`, and vice versa.

### Script-emitted (`check_wiki.py`)

#### Critical (script `error`)

- `frontmatter_missing`: page has no YAML frontmatter block at all (missing opening or closing `---`). Distinct from `frontmatter_missing_field` (a present block missing a required field, Warning).
- `missing_index`: `1-wiki/index.md` is absent.
- `file_field_unresolved`: Source page whose `file:` does not resolve to a file under `0-raw/`.
- `embed_unresolved`: Embedded image path does not resolve to a file under `1-wiki/attachments/`.
- `raw_without_source_page`: a raw file under `0-raw/` has no source page referencing it via `file:` — the file is uningested. **Standing**, not audit-blocking (`STANDING_NONBLOCKING`); CLAUDE.md also names this standing condition `uningested_raw_source`.
- `status_needs_update`: page marked `needs-update`. The message reports the page's `needs_update_reason:` field, or if it has none, that the `Contradictions`/`Tensions` callout carries the reason, so the reader sees why without opening the page. **Standing**, not audit-blocking. lint does not fix `needs-update` pages: resolving one is content judgement (`audit` or ingest's existing-source mode), not a mechanical edit.
- `zero_source_page` (synthesis only): synthesis page with `source_count: 0` — structurally invalid. The same check emits at **Warning** for concept/entity pages (see below).
- `verified_anchor_unaudited`: **Mechanism 1 (the diff-guard).** A `status: verified` page on which a raw-locator's structural anchor was ADDED or CHANGED relative to its git-HEAD version while the page stayed `verified`. A section/figure change is a factual claim about where the cited content sits — excluded from the verification-neutral allowlist (CLAUDE.md → Page Status) — so only a raw fact-check (`audit`) may keep the page verified; self-re-stamping a section change is not verification (it is what produced the abstract→`sec. 1` mislabels). A pure RELOCATION (same anchor + page, just repositioned relative to the link) and a minor typo/format edit are neutral and not flagged; a bullet marked `*[unverified]*` is exempt (already declared pending). Git-aware and best-effort: a no-op without a git HEAD to diff against (new repo, untracked file). Catches the change at lint-time, before commit; lint has no dedicated committed-case backstop — once committed, a section/figure mislabel is caught only by the generic body-hash demotion (if the change altered the hashed body) or by an `audit` raw fact-check, not by a second lint mechanism. Not auto-fixed (the remedy — demote, mark `*[unverified]*`, or re-verify via audit — is the editor's call): the report carries it. Logic: `check_verified_anchor_change` / `anchor_change_findings` in `.claude/skills/lint/scripts/check_wiki.py`. Rule: CLAUDE.md → Page Status.
- `locator_page_mismatch`: A `#page=N` deep-link whose display cites `p. M`, but the pagination map (`.claude/skills/lint/pagination-map.md`) says physical page N of that raw prints a DIFFERENT number — or prints none at all. The one check that catches a confidently-wrong printed page: a locator that renders plausibly and passes every completeness check yet sends a reader to a number the source does not carry (lint Limits names this the worst failure mode). Runs on every page kind that carries deep-links; keyed on the map, so an unregistered raw is silently skipped (`pagination_map_unregistered` nudges to register it). Deliberately conservative for an `error`: only a single `p. M` is matched (`pp. M–N` ranges are not), and the `p.` inside `app.` never matches (`\b`-guarded). Not auto-fixable — like a mislabelled locator it cannot tell which half is wrong (the cited page, the `#page=N`, or the source): the report carries it; `audit` or the user opens the raw and corrects it. Logic: `check_locator_page_match` in `.claude/skills/lint/scripts/check_wiki.py`. Rule: CLAUDE.md → Source Support And Verification.

#### Warning (script `warning`)

- `frontmatter_missing_field`: a required frontmatter field is missing from a present frontmatter block.
- `status_invalid`: `status:` is not one of `draft` / `verified` / `needs-update`.
- `section_order`: Missing or wrong-order callout sections.
- `callout_block_id`: A callout is missing its block ID, carries one that doesn't match the expected ID, has more than one, or has it somewhere other than the last line inside the `>` block. Every callout carries `> ^<block-id>` as its last inner line so sections are linkable via `[[path#^block-id|Display]]` (Obsidian does not anchor to callout titles). The block ID is the kebab-case of the callout title — the type slug for most callouts, the expanded form for an abbreviated type (`[!why]` → `^why-it-matters`). Auto-fixable: the expected ID follows from the callout type, so the fix is deterministic. Rule: CLAUDE.md → Callout Block IDs.
- `page_locator_unlinked`: A page locator (`p. N`, `pp. N–M`) on a wiki page is bare text rather than a `#page=N` raw-file deep-link. Every locator should be a deep-link so a click opens the source at that page — on a source page with its structural anchor and page together inside the display, `[[0-raw/papers/<stem>.pdf#page=N|sec. X, p. N]]` (the form `source_locator_incomplete` then enforces). The scan blanks inline-code spans and `[[...]]` spans first, so a locator already inside a deep-link (`[[…#page=N|p. 4186]]`) or inside `` `code` `` is not flagged. Not auto-fixable: N is the physical PDF page, which differs from the printed page for proceedings/article pagination and can't be derived without opening the PDF — the report carries it; `audit` or the user wraps it, computing N from the printed→physical offset (`N = printed − (first_printed − 1)`, where `first_printed` is the printed number on the PDF's first physical page). Rule: CLAUDE.md → Source Support And Verification.
- `citation_locator_incomplete`: On a concept/entity/synthesis page, an inline raw `#page=N` deep-link used as a citation whose display lacks a structural anchor (`sec.`/`app.`/`ch.`/`fig.`/`tab.`/`eq.`/`abstract`) and/or a printed page (`p. M`). The canonical citation form requires both inside the deep-link display (CLAUDE.md → Source Support And Verification). The front-matter `abstract` is a valid anchor — cite abstract-drawn content as `abstract, p. 1`, never relabelled `sec. 1` (on most papers sec. 1 is not on the abstract's page). One exemption: an `app.`-anchored display may omit `p. M` (`app. D.1, tab. 8`), because a published appendix often carries no printed page number and inventing one is worse than omitting it — the unpaginated-supplement exemption. It is `app.`-only, and is the *fallback* for a raw not in the pagination map; when the raw IS registered in `.claude/skills/lint/pagination-map.md`, the exemption is keyed on the map instead — a page the map says prints nothing exempts any anchor, a page that prints a number requires `p. M` (and `locator_page_mismatch` checks that `p. M`). Scan blanks inline-code spans and the `Sources` callout (its support-list links are not inline citations). Not auto-fixable: the missing anchor/page must come from the raw; `audit` or the user adds it. Source pages run the parallel `source_locator_incomplete` check (same anchor-inside-the-display form, just no paired source-page wikilink). Rule: CLAUDE.md → Source Support And Verification.
- `citation_unpaired`: On a concept/entity/synthesis page, an inline raw deep-link not preceded on its bullet by a source-page wikilink. The canonical form pairs them so the reader gets both *which* source and *where* in it — Form 1 `[[…X.md|X]] ([[…X.pdf#page=N|sec. Y, p. M]])` or Form 2 `([[…X.md|X]]; [[…X.pdf#page=N|sec. Y, p. M]])`. Not auto-fixable (the source link is content). Rule: CLAUDE.md → Source Support And Verification. (Mechanical format only; the semantic rule that *every non-obvious claim* must carry a full citation is `audit`/`ingest`'s, since lint cannot judge whether a bullet is a non-obvious claim or an allowed bare general attribution.)
- `citation_bracket_style`: On a concept/entity/synthesis page, a Form 2 citation written in the superseded square-bracket form — the source-page wikilink + locator(s) wrapped in a literal outer `[ ]`, surfacing as the triple-bracket `[[[` opener (`[[[…X.md|X]]; [[…X.pdf#page=N|sec. Y, p. M]]]`). The canonical Form 2 uses round brackets `( )`. Detection blanks inline-code spans and the `Sources` callout first, so a `[[[…]]]`-shaped example inside `` `code` `` or a quote is not flagged. Auto-fixable: swap the outer `[ ]` for `( )` only (inner wikilinks unchanged), within that same masked scope so a literal is never rewritten — see the auto-fix list. Rule: CLAUDE.md → Source Support And Verification.
- `source_locator_incomplete`: On a source page, a `#page=N` raw-file deep-link whose display does not carry a structural anchor (`sec.`/`app.`/`ch.`/`fig.`/`tab.`/`eq.`/`abstract`) AND a page (`p. M`) together inside the link. Source-page locators list the section and page together inside the display — `[[…#page=1|sec. 1, p. 1]]` — never split with the anchor outside the link (`sec. 1, [[…#page=1|p. 1]]`) or page-only. The source-page counterpart of `citation_locator_incomplete`, which enforces the same anchor-inside-the-display form on concept/entity/synthesis pages; source pages share the completeness test but drop the paired source-page wikilink (it would be a self-link), so `citation_unpaired` does not apply. This is the inverse of the retired `source_locator_anchor_inlined`, which wrongly required the anchor outside the link. Both checks share `locator_display_complete`, so the unpaginated-region exemption applies identically here, keyed on the pagination map when the raw is registered (a page the map says prints nothing may stand on its anchor alone, any anchor; a page that prints a number needs `p. M`) and falling back to the `app.`-only heuristic (`app. D.1, tab. 8`) for an unregistered raw. Scan blanks inline-code spans and the `Sources` callout. Not auto-fixable: the fix edits citation text in callout body prose (which lint never auto-edits) — the report carries it; `audit` or the user lists the anchor and page together inside the link. Logic: `check_source_locator_complete` in `.claude/skills/lint/scripts/check_wiki.py`. Rule: CLAUDE.md → Source Support And Verification.
- `source_count_mismatch`: `source_count` does not match `len(sources)`.
- `zero_source_page` (concept/entity): concept/entity page with `source_count: 0` (fragile). Synthesis pages emit this at Critical (see above).
- `index_missing_entry`: a page on disk is not listed in `index.md`.
- `index_stale_entry`: `index.md` lists a page with no matching file on disk.
- `chronology_missing_time`: an entry in `log.md` (an `## [YYYY-MM-DD HH:MM] …` header) or in `hot.md`'s Recent activity (a `- [YYYY-MM-DD HH:MM] …` bullet) carries a date but no `HH:MM` time. The time is what disambiguates ordering when work from separately-merged branches interleaves in one file (CLAUDE.md → Hot, Index, And Log). One finding per file, with the count and first offender. Auto-fixable in the determinate case: `sort_chronology.py` recovers a missing `HH:MM` from the entry's own linked report filename (`2-outputs/…-YYYY-MM-DD-HHMM-…`) when exactly one such link matches the entry's date, then sorts — it transcribes the time, never invents it, and never consults git. An entry with no recoverable link (none, or conflicting times) still needs the time added by hand (from the report, or git) before the section can be sorted. Logic: `check_chronology` in `.claude/skills/lint/scripts/check_wiki.py`; recovery and fix in `.claude/skills/lint/scripts/sort_chronology.py`. Rule: CLAUDE.md → Hot, Index, And Log.
- `chronology_out_of_order`: the timed entries in `log.md` or in `hot.md`'s Recent activity are not in descending (date, time) order (newest first). Auto-fixable: run `python3 .claude/skills/lint/scripts/sort_chronology.py`, a determinate stable sort on the `[date time]` header — entry bodies move with their header, no prose changes, equal timestamps keep their original order, and the rewrite is idempotent. The sorter first auto-recovers determinate missing times (see `chronology_missing_time`) and refuses the file only if an untimed entry has no recoverable link, so clear any such residual first. One finding per file. Logic: `check_chronology` in `.claude/skills/lint/scripts/check_wiki.py`. Rule: CLAUDE.md → Hot, Index, And Log.
- `missing_reciprocal_contradiction`: page A's `Contradictions` names `[[B]]` but B's `Contradictions` (or, for syntheses, `Tensions`) does not mention A. Navigational "see also `[[…]]`" links are excised before the check so they are not treated as contradiction parties. Synthesis namers are exempt: a synthesis's `Tensions` cites its sources as evidence, not as mutual-contradiction partners, so a synthesis naming `[[B]]` does not require B to reciprocate (a synthesis can still be a target B). Same-work pairs are exempt for the same reason: a page naming one of its own `sources:` (or a source naming a page derived from it) cites that source as evidence for a disagreement with a *third* party, not a mutual contradiction the source must reciprocate.
- `attachments_frontmatter_missing_file`: a source page's `attachments:` frontmatter lists a file that does not exist under `1-wiki/attachments/`.
- `attachments_file_unlisted`: a file in `1-wiki/attachments/{stem}/` is not listed in that source page's `attachments:` frontmatter.
- `concept_multi_image`: Concept/entity page with more than one image embed.
- `embed_not_isolated`: a standalone image embed line (`> ![[…]]`) inside a callout is not set off by a blank quoted line (`>`) both above and below it. An embed butted directly against the bullet above or the line below lazy-continues that content and mis-renders in Obsidian; the rule is blank `>`, embed, blank `>`. Auto-fixable: insert a blank `>` line on the missing side(s) (`above`, `below`, or `above and below`, per the finding). Applies to all page kinds with embeds. Rule: CLAUDE.md → Attachments / Source Pages.
- `concept_duplicate_candidate`: Near-duplicate concept/entity page titles (script-detected; ≥0.75 word-set Jaccard overlap on the H1 heading). Catches the case where a source refresh creates a near-twin instead of updating the existing page. Audit covers the harder semantic-duplicate case where titles diverge. (Reported once, filed under the first page of the pair; the partner page is named in the message — same for `alias_collision`.)
- `intra_page_redundancy`: Two bullets on ONE concept/entity/synthesis page that make the same point across (or within) callouts — the CLAUDE.md "do not paraphrase the same point across sections" rule. Distinct from `concept_duplicate_candidate`, which is cross-PAGE title duplication; this is within-PAGE bullet duplication. Mechanical/lexical only: compares content-token sets (wikilinks, citations, `*[markers]*`, inline code, and stopwords stripped) with two arms — a Jaccard arm (`BULLET_JACCARD_THRESHOLD`, similar-length near-twin bullets) and an overlap-coefficient arm (`BULLET_OVERLAP_THRESHOLD`, a short bullet's whole point folded inside a longer one), each gated by a min-token floor. The semantic half — a point fully reworded — is ingest's note-quality packet and audit's walk, not this check. Source pages are exempt: they legitimately restate a Key Claim as a verbatim Evidence anchor. Not auto-fixable (which bullet to drop or how to merge is editorial): the report carries it; `audit` or the user resolves it. Logic and thresholds: `check_intra_page_redundancy` in `.claude/skills/lint/scripts/check_wiki.py`. Rule: CLAUDE.md → Body Sections As Callouts.
- `source_context_leak`: Source-context phrase leakage on a concept or entity page. Phrases like `as the paper notes`, `the authors find`, `in this work`, `according to the source` tie the bullet's meaning back to a particular source rather than expressing the idea on its own terms. Concept and entity pages must read as standalone ideas; source support belongs in `sources:` and the `Sources` callout. Patterns scanned: see `SOURCE_CONTEXT_PHRASES` in `.claude/skills/lint/scripts/check_wiki.py`. Design rationale: D 3.11 in `a-archive/reference/llm-wiki-design-decisions.md`.
- `vague_source_referent`: A concept/entity/synthesis bullet attributes a claim or caveat to a source without naming it — a bare `the source notes`, `a source claims`, `one framework's setup`, `one study found`. These pages accrue more sources over time, so a referent that is unambiguous at one source becomes ambiguous at two. Not auto-fixable (naming requires judgement): the report carries it; `audit` or the user names the source via a pipe-rendered wikilink (or restates the idea on its own terms). Distinct from `source_context_leak`, which flags source-framing to *remove*; this flags an unnamed source to *name*. The two are non-overlapping by construction (that check keys on definite `the (paper|study|work)` + verb and `according to …`; this keys on bare `the source` and the indefinite `(a|an|one) (source|framework|study|paper|work)`). Source pages are exempt — the whole page is scoped to one source. Patterns: `VAGUE_SOURCE_REFERENT` in `.claude/skills/lint/scripts/check_wiki.py`. Rule: CLAUDE.md → Plain-Language Style.
- `hyphenated_open_compound`: An established multi-word term that field convention leaves open is hyphenated as an attributive modifier — `reinforcement-learning`, `deep-reinforcement-learning`, `machine-learning`, `deep-learning`, `imitation-learning`, `transfer-learning`, `supervised-/unsupervised-/self-supervised-learning`, `natural-language`, `language-model(s)`. CMOS 7.89 drops the hyphen when a compound is familiar and unambiguous, so the wiki writes "reinforcement learning benchmark", not "reinforcement-learning benchmark". Applies to every page kind (source pages carry these terms too). Deliberately NOT flagged: `multi-agent` (a prefixed compound, universally hyphenated), `foundation-model` (hyphenated as an attributive modifier), `in-context` (conventionally hyphenated), and a token inside a longer hyphenated construct like `natural-language-vs-code` (the `(?!-)` guard). Scans the code-, wikilink-, and double-quote-masked body, so a slug or path containing a hyphenated token is not flagged, and a hyphenated compound inside a verbatim quote (`"…"`) is never surfaced — de-hyphenating a quoted term would alter the quote, which must read exactly as the source wrote it (CLAUDE.md → Page Status, the text-content exclusion). Not auto-fixable (hyphenation lives in callout body prose, which lint never auto-edits): the report carries the suggested open form; `audit` or the user applies it (only in running prose, never inside a quote, title, or identifier). Mapping and regex: `OPEN_COMPOUND_SUGGEST` / `HYPHENATED_OPEN_COMPOUND` in `.claude/skills/lint/scripts/check_wiki.py`. Rule: CLAUDE.md → Plain-Language Style.
- `hyphenated_open_compound_noun`: A slug-derived open compound — one whose kebab-case page slug leaks into prose (`tool-use`, `belief-state`, `knowledge-exchange`, `task-execution`, `state-update`, …) — is hyphenated where it is used as a bare NOUN. Distinct from `hyphenated_open_compound` above (whose terms read open in every position): these terms are correct OPEN as a noun ("tool use is costly") but correct HYPHENATED as an attributive modifier ("tool-use training", "belief-state representation" — standard compound-modifier hyphenation, CMOS 5.91). So this check flags ONLY the noun position — the compound followed by a clause boundary or a copula/comparison word (`is/are/was/were/be/been/being/versus/vs/than`) — and never a modifier, so it does not overcorrect a grammatical hyphen. Driven by four curated lists meant to grow over time (the two below for the core noun/modifier match, plus the direction-2 HEAD-noun list `COMPOUND_MODIFIER_HEADS` and the verified-ignore escape valve, both introduced further down): DISALLOWED `OPEN_COMPOUND_NOUN_SUGGEST` (hyphenated → open noun form) and ALLOWED `HYPHENATED_COMPOUND_ALLOWED` (look-alike compounds that are correct hyphenated — proper names like `gpt-3`, established terms like `fine-tuning`/`chain-of-thought`, prefixed compounds like `multi-agent` — never flagged; a hard never-flag guard). Deliberately CONSERVATIVE: a following comma or word is ambiguous (a modifier list, or a modified noun) and is not treated as noun position, so the check errs toward silence. Scans the same code-, wikilink-, and double-quote-masked body as `hyphenated_open_compound`, so it never flags a slug, a path, a quoted term, or a wikilink-display compound; noun position is judged on the unmasked body so a masked head-noun is not read as trailing whitespace. The check is BIDIRECTIONAL — it enforces one rule (compound as a noun → open; same compound as a modifier → hyphenated) in both directions. Direction 1 (above) opens a hyphenated bare noun. Direction 2 is the inverse: an OPEN compound used as a modifier directly before a noun — overcorrected open in the past — should be RE-HYPHENATED (`belief state representation` → `belief-state representation`). Direction 2 is dangerous to infer from a "not a noun-signal" rule, because an open-compound noun is very often followed by a verb (`tool use reaches`, `belief state shows`) that such a rule would misread as a head noun and wrongly hyphenate (27 such false positives on this corpus). So direction 2 fires ONLY when the open compound is immediately followed by an explicit curated HEAD NOUN (`COMPOUND_MODIFIER_HEADS`: representation, prompt, training, method, mechanism, loop, message, content, …) — a word not on that list is never treated as a head, so a following verb can never trigger a hyphen. The verified-ignore set `HYPHENATION_VERIFIED_IGNORE` (lowercased `{compound} {following word}` phrases a human confirmed correct as written, empty by default), suppresses a flag in EITHER direction — the escape valve that lets a genuine exception be recorded once and never re-flagged. Not auto-fixable (callout body prose): the report carries the suggested form (open noun, or hyphenated modifier); `audit` corrects each flagged use by checking it in context against the raw — confirming the noun-vs-modifier call before applying the claim-neutral fix, marking a genuinely uncertain use `*[unverified]*`, or recording a false positive in `HYPHENATION_VERIFIED_IGNORE` rather than guessing. The four lists are NOT hardcoded in the script — they live in the agent-writable data file `.claude/skills/lint/hyphenation-lists.md` (four `## section`s: disallowed, allowed, heads, verified-ignore), which `check_wiki.py` parses at load (tolerant: a missing file silently no-ops the check, a malformed line is skipped). `audit` grows all four lists autonomously, each addition sub-agent-verified, as it confirms hyphenation uses against the raw (audit Step 7; CLAUDE.md → Stay In Your Lane lists the file as agent-writable data, not script logic). To extend: append to the right section of the data file (sub-agent-verified) and add a paired regression case in `tests/test_check_wiki.py` (`test_open_compound_noun_*`) for a disallowed/allowed/head addition. Loader, derived regexes, and matchers (`_load_hyphenation_lists` / `OPEN_COMPOUND_NOUN_SUGGEST` / `HYPHENATED_COMPOUND_ALLOWED` / `COMPOUND_MODIFIER_HEADS` / `HYPHENATION_VERIFIED_IGNORE` / `HYPHENATED_OPEN_COMPOUND_NOUN` / `OPEN_COMPOUND_MODIFIER` / `_NOUN_POSITION_AFTER`) in `.claude/skills/lint/scripts/check_wiki.py`. Rule: CLAUDE.md → Plain-Language Style.
- `unlinked_page_mention`: An existing wiki page's title or alias appears as plain text in another aggregator page's body (concept/entity/synthesis) where it should be a wikilink — the schema requires every genuine reference to an existing page to be linked, on every occurrence. One finding per (page, target) with the unlinked-occurrence count. Not auto-fixable (genuine-reference vs generic-wording is a judgement): `audit` or the user wikilinks each genuine reference. Source pages are scanned too — a source page's own topic is a different page (a genuine cross-reference to link), so the only exemption is a page linking to itself (its own stem/aliases). Masks code, existing `[[wikilinks]]`, and quoted spans; excludes the page's own stem/aliases (a page is never asked to link to itself). The check is heuristic — a homograph, a common noun that happens to be a page title, or the term inside a larger established phrase is not a reference to that page — so it carries a VERIFIED-IGNORE escape valve, the counterpart of `hyphenated_open_compound_noun`'s: an occurrence `audit` confirms is generic wording is recorded in the agent-writable data file `.claude/skills/lint/unlinked-mention-ignore.md` (one `{page path} :: {target stem} :: {phrase}` entry per line under `## verified-ignore`), and is not flagged again — so a correctly-judged-generic occurrence is settled once instead of re-litigated on every run. An entry is PAGE-SCOPED (genuine-vs-generic is a per-page call, so it never suppresses the same wording on another page, where a genuine reference still flags) and PHRASE-ANCHORED, hence self-invalidating: it suppresses only occurrences falling inside the recorded phrase, so rewording the sentence stops the match and the mention re-flags. A stale entry suppresses nothing and is inert — the failure mode is a re-flag, never a silently swallowed genuine reference. One entry settles one occurrence context, not a page: other unlinked mentions of the same target on that page still flag. The list is NOT hardcoded in the script — `check_wiki.py` parses the data file at load (tolerant: a missing file silently no-ops the suppression, leaving the check fully unsuppressed — the safe direction; a malformed line is skipped). `audit` grows it autonomously, each addition sub-agent-verified (audit Step 7; CLAUDE.md → Stay In Your Lane lists the file as agent-writable data, not script logic), while the check's code stays edit-only. Loader and matcher (`_load_unlinked_mention_ignore` / `UNLINKED_MENTION_IGNORE`) and logic (`check_unlinked_page_mentions`) in `.claude/skills/lint/scripts/check_wiki.py`. Rule: CLAUDE.md → Wikilink Format.
- `stale_mention_ignore`: A verified-ignore entry in `.claude/skills/lint/unlinked-mention-ignore.md` suppresses nothing — it is dead. Three ways an entry goes stale, and the message names which: the page it was recorded on no longer exists; its target page no longer exists; or no unlinked mention of the target falls inside the recorded phrase any more (the wording changed, or the mention has since been wikilinked — either way the recorded judgement no longer binds). A stale entry is INERT, never dangerous: a phrase-anchored entry can only fail to match, never wrongly suppress, so this is hygiene rather than a correctness defect — but dead entries otherwise accumulate silently as pages are reworded, renamed, or removed, and nothing would ever prompt their removal. Warning-tier all the same, because the tier says WHO ACTS: Warning is lint's authored worklist that `audit` carries out (and `audit` owns this data file), whereas Info is explicitly not audit's to action — an Info-tier stale entry would be a finding no skill is permitted to clean up. Warnings never block audit, so the gate is unaffected. Not auto-fixable by lint (lint does not edit the agent-curated data files): `audit` deletes the entry, which needs no sub-agent verification — the check has already proven the entry suppresses nothing, so removal is mechanical. If the occurrence still exists but was reworded, re-record it against the current phrase rather than editing the phrase to match, which would carry the old judgement onto text nobody re-confirmed. Logic: `_stale_mention_ignore_findings` in `.claude/skills/lint/scripts/check_wiki.py` (emitted by `check_unlinked_page_mentions`, which tracks which entries actually suppressed an occurrence). Rule: CLAUDE.md → Wikilink Format.
- `recursive_wiki_citation`: A concept/entity/synthesis page has empty `sources:` but its body wikilinks point only at other non-source wiki pages, so the page risks "the wiki cites itself as truth" without any actual source grounding. Distinct from `zero_source_page` which only checks the empty-sources condition. Design rationale: D 3.9 / D X.3.
- `stale_needs_update`: Page is `status: needs-update` and `updated:` is more than `STALE_NEEDS_UPDATE_DAYS` (default 30) days ago. Known-broken pages should not sit untouched; warning-severity (higher than `stale_draft` because the page is already flagged as broken). Design rationale: D 5.5b.
- `alias_collision`: Two or more pages claim the same `aliases:` entry. Obsidian's wikilink resolution by alias becomes ambiguous. Design rationale: D 9.13.
- `alias_shadows_filename`: An alias matches another page's filename stem. `[[alias]]` resolves to the file, not the aliased page: silent misdirection. Design rationale: D 9.13.
- `sources_callout_desync`: The `> [!sources]` callout body and `sources:` frontmatter list different source pages on a concept/entity/synthesis page. Frontmatter is machine-readable, callout is reader-facing; drift between them silently breaks the support trail. Design rationale: D X.3a.
- `source_link_unresolved`: A `sources:` frontmatter entry on a concept/entity/synthesis page does not resolve to an existing source page under `1-wiki/sources/`. Source-support links must never dangle (unlike concept/author wikilinks, which the schema permits to dangle), so a `sources:` entry pointing at a deleted or renamed source page is a broken provenance trail. Deterministic — replaces the former manual lint walk. Rule: CLAUDE.md → Source Support And Verification.
- `concept_source_asymmetry`: A source page's `concepts-entities` callout links to a page that doesn't list the source in its `sources:` (or vice versa). Forward and reverse links must agree. Synthesis pages are exempt from the reverse direction — a source's `concepts-entities` lists the concepts/entities it supports, not the syntheses that cite it. Design rationale: D X.3b.
- `needs_update_without_reason`: A page is `status: needs-update` but `needs_update_reason:` is absent/empty AND the relevant `Contradictions`/`Tensions` callout is the empty placeholder. CLAUDE.md requires one or the other. Design rationale: D 5.5c.
- `synthesis_under_supported`: A synthesis page has exactly 1 source and `single_source_stub: true` is not set in frontmatter. Schema requires synthesis pages to have ≥ 2 sources unless explicitly created as a stub. Design rationale: D 3.3 / D X.3.
- `wikilink_display_uncapitalized`: A callout bullet that opens with a wiki-page wikilink (`> - [[1-wiki/…|display]]…`) whose display starts with a lowercase letter. A sentence-initial wikilink takes a leading capital — a common-noun display stays lowercase mid-sentence but capitalizes when it opens the bullet. Scoped to `1-wiki/` page targets, so a bullet opening with a raw-file locator deep-link (`[[0-raw/…#page=5|p. 5]]`, a page token, not a page name) is not flagged; the `Sources` callout is blanked (its displays are filename-derived source stems, kept verbatim); a display whose first character is not a lowercase letter (a digit, an already-capitalized proper noun, a symbol) is left alone. Applies to every page kind. Not auto-fixable by lint: the fix edits callout body prose (the display text), which lint never auto-edits, and is not on the verification-neutral allowlist — the report carries it with the capitalized form; `audit` or the user applies it. Logic: `check_wikilink_display_caps` in `.claude/skills/lint/scripts/check_wiki.py`. Rule: CLAUDE.md → Wikilink Format.
- `wikilink_pipe_spacing`: a wikilink carries whitespace adjacent to its pipe (`[[path | display]]`), which renders as a leading space in the display name. Auto-fixable: collapse the padding to a bare `|` inside the matched wikilink span only (never a `|` outside `[[...]]` or inside inline code). Rule: CLAUDE.md → Wikilink Format.

#### Info (script `info`)

- `status_draft`: applies to every draft page; complements `stale_draft` below which catches *stale* drafts specifically.
- `unverified_claim`: a concept/entity/synthesis page carries one or more bullets marked `*[unverified]*` — non-obvious claims added or changed since the last raw fact-check, awaiting re-check (CLAUDE.md → Page Status, Bullet Markers). Reported per page with the count. A normal transient state, not drift: lint does not clear it (audit does, after a raw fact-check; partial mode re-checks just these claims on otherwise-`verified` pages). One finding per page.
- `orphan_page`: pages in `sources/`, `entities/`, `concepts/`, or `syntheses/` not reachable from `1-wiki/index.md` via wikilinks. Implemented via BFS over the wikilink graph in `check_wiki.py`. Design rationale: D 9.12.
- `placeholder_only_page`: every required callout on the page is the empty placeholder (`> - None noted` or `> - None yet`); the page is a stub with no genuine content yet. Design rationale: D X.4.
- `attachment_orphan`: Orphan attachment file: present in `1-wiki/attachments/{stem}/` but not embedded by any wiki page.
- `attachment_duplicate_basename`: the same attachment basename is reused across `{stem}/` folders. Path-qualified embeds still resolve unambiguously, so this is a readability nudge only.
- `stale_draft`: page is `status: draft` and `updated:` is more than `STALE_DRAFT_DAYS` (default 14) days ago. Earlier nudge than audit's 60-day stale-draft check. Design rationale: D 11.5/6.
- `pagination_map_unregistered`: a raw cited somewhere with a `#page=N` deep-link but absent from the pagination map (`.claude/skills/lint/pagination-map.md`). The citation still lints — the locator-completeness checks fall back to the `app.`-anchor heuristic, and `locator_page_mismatch` simply cannot run on the raw — so this is an INFO nudge, not a blocker: register the raw (`scripts/pagination_map.py` proposes a map from its footers; a human confirms each line against a rendered footer) so its `p. M` locators can be verified. One finding per unregistered raw, reported against the raw path. Logic: `check_pagination_registration` in `.claude/skills/lint/scripts/check_wiki.py`. Rule: CLAUDE.md → Source Support And Verification.

These three are real schema violations whose only safe fix is a file rename or link re-path — a user-owned safety operation that **neither lint nor audit auto-actions** (so they sit at Info, not Warning: Warning is audit's authored-tier worklist, and these are not audit's to carry out). lint surfaces them for the user:

- `filename_not_kebab`: a concept/entity/synthesis page filename, or an attachment leaf filename, is not kebab-case lowercase (ASCII letters, digits, hyphens). Source pages are exempt (their stem mirrors the raw stem, which preserves case/punctuation), as is the attachment `{stem}/` directory component. Rule: CLAUDE.md → Page Filenames.
- `bare_basename_link`: a page wikilink target is a bare basename rather than a path-qualified `[[folder/path/file.md|display]]`. Resolution still works (the script's `_link_stem` stays tolerant so graph checks keep resolving), so this is a convention fix, not a broken link. Excludes image embeds, pure-anchor self-links (`[[#^…]]`), inline code, and literal `[[…]]` in quoted prose (a target with commas/spaces is not a link). The correct folder is not always mechanically recoverable, so it is not auto-fixed. Rule: CLAUDE.md → Wikilink Format.
- `source_stem_mismatch`: a source page's stem differs from its raw-file stem (the two must share a basename; a chapter split `<raw-stem>-chNN` is allowed). Rule: CLAUDE.md → Page Filenames / Raw Sources.
- `verified_hash_mismatch` (Warning): a `status: verified` page whose body hash does not match its `verified_hash:` frontmatter field, or a `verified` page with no `verified_hash:` at all — its *checked content* changed since `audit` verified it. The script computes the hash with the shared `body_hash.py` (SHA-256 of the page body, frontmatter AND `*[unverified]*` claim lines excluded), so editing a marked claim does not trip it; only a change to unmarked content does. Detection only — the re-stamp-or-demote *decision* is SKILL.md Step 3's (the `verified_hash_mismatch` bullet), governed by the shared verification-neutral allowlist (`.claude/skills/multi-skill/references/verification-neutral-fixes.md`, CLAUDE.md → Page Status): an allowlisted format fix on a baseline-matched page re-stamps and keeps `verified`; any other change resets to `draft` and strips the stale hash so a later `audit` re-verifies. A `verified_hash:` on an already non-`verified` page is likewise stripped. The Step-2 baseline (match/mismatch before any Step 3 fix) is the run-state the decision needs and the script's single-pass finding does not carry. Rule: CLAUDE.md → Page Status.

> Terminology: `status_draft` = the page is unreviewed; `placeholder_only_page` = body bullets are all empty placeholders; `single_source_stub: true` = explicit one-source synthesis (the schema escape hatch `synthesis_under_supported` respects). The three overlap but are not the same: a page can be `draft` with rich content, `placeholder_only` after a recent `verified` promotion to a near-empty page, or a deliberate single-source synthesis stub that lint should not warn on.

### LLM-walk (Step 2; not script-emitted)

These have no `check_id` in `check_wiki.py`; the agent runs them during the Step 2 walk and cites them by short name. Severity shown is the tier they report under.

- `broken_wikilink` (Warning): a wikilink whose target file does not exist, outside intentionally dangling author links. The script does not traverse links to confirm existence; path-qualification of targets is separately script-checked via `bare_basename_link`.
- `citation_clutter` (Warning): old line-by-line citation-heavy concept/entity page style.
- `concept_multi_idea` (Warning): a concept/entity page appears to contain more than one idea (a cheap LLM atomicity heuristic; the deeper split is `audit`'s job).
- `length_cap_exceeded` (Warning): concept/entity page over 400 words, synthesis over 600, or a bullet over ~35 words (the script does not count words).
- `missing_reciprocal_connection` (Warning): a one-sided explicit `Connections` relationship — the LLM-walk analogue of the script's `missing_reciprocal_contradiction`.
- `ai_writing_tells` (Warning): mechanical AI-writing tells in a wiki page body (banned vocabulary, em-dash density, curly-quote leakage, citation-markup leakage, placeholder leakage, UTM parameters). Source list: `a-archive/style/ai-writing-tells.md`.
- `hot_log_stale` (Info): hot/log stale or approaching cap.
- `schema_drift` (Info): schema layer (`CLAUDE.md`, `README.md`, `.claude/skills/`) changed since the last consistency report; re-run `consistency`.

### Standalone (opt-in; not in the core battery)

Emitted by a dedicated script, run deliberately and separately from `check_wiki.py` — NOT part of the structural pass, the Step 3 loop, or the audit-blocking gate, and NOT among the Script-emitted ids `--list-checks` prints (so it is exempt from the Script-emitted ↔ registry diff). Listed here for a complete roster.

- `cited_figure_off_page` (Warning; `scripts/cited_figure_check.py`): a distinctive numeric figure (decimal or percentage) on a bullet that appears on NONE of that bullet's cited raw `#page=N` pages, when every cited page was readable. The backstop for the cross-source mis-location defect — a true claim cited to a page that does not carry it (CLAUDE.md → Source Support And Verification, the multi-source-bullet rule): the claim reads correctly and passes every structural check, so only opening the cited raw catches it. Detect-only, never auto-fixed — like a mislabelled locator it cannot tell which half is wrong (the figure, the page, or the source). Scope: decimals and percentages only (bare integers skipped, the false-positive source); locator machinery inside `[[…]]` spans is masked before figures are read; the presence test is whitespace-tolerant and errs toward "present" so a false finding is rare. Reads the raw PDF (PyMuPDF), so it sits OUTSIDE `check_wiki.py`'s no-raw-reading battery and runs deliberately with the `llm-wiki` conda env active; exit code 3 means PyMuPDF is absent and the check could not run (not "zero findings"). It cannot catch a mislocated qualitative claim (no figure to match) — that stays ingest's and audit's page-scoped citation check, so it is a backstop for the numeric subset, not a substitute. Logic and tests: `cited_figure_check.py` and `tests/test_cited_figure_check.py`. Rule: CLAUDE.md → Source Support And Verification.
